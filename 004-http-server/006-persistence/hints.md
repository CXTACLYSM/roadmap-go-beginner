## Уровень 2: про Save и Load с мьютексом

Шаблон:

```go
func (tl *TaskList) Save(filename string) error {
    tl.mu.Lock()
    defer tl.mu.Unlock()

    data, err := json.MarshalIndent(tl, "", "  ")
    if err != nil {
        return fmt.Errorf("сериализация: %w", err)
    }
    if err := os.WriteFile(filename, data, 0644); err != nil {
        return fmt.Errorf("запись %s: %w", filename, err)
    }
    return nil
}

func (tl *TaskList) Load(filename string) error {
    tl.mu.Lock()
    defer tl.mu.Unlock()

    data, err := os.ReadFile(filename)
    if err != nil {
        if errors.Is(err, os.ErrNotExist) {
            return nil
        }
        return fmt.Errorf("чтение %s: %w", filename, err)
    }
    return json.Unmarshal(data, tl)
}
```

Это почти точная копия из Проекта 3, только с мьютексами.

## Уровень 3: про сериализацию TaskList с мьютексом

Возможный вопрос: «А что будет с полем `mu sync.Mutex` при `Marshal`?». Ответ: **ничего**. Поле `mu` с **маленькой** буквы, поэтому `encoding/json` его игнорирует.

Если бы поле было `Mu` (с большой) — Go попытался бы сериализовать `sync.Mutex`, и результат был бы `{}` (пустой объект). Не падение, но мусор. Для надёжности можно явно поставить тег:

```go
mu sync.Mutex `json:"-"`
```

Но для приватного поля это не обязательно — `Marshal` его и так не увидит.

## Уровень 4: про загрузку в main

```go
func main() {
    tasks := &TaskList{NextID: 1}

    if err := tasks.Load(dataFile); err != nil {
        log.Fatalf("Ошибка загрузки: %v", err)
    }

    server := &Server{tasks: tasks}
    // ... обработчики, ListenAndServe
}
```

Несколько моментов:

- **`tasks` создаётся со стартовым `NextID: 1`**. Если файла нет — `Load` ничего не поменяет, и эта инициализация останется. Если файл есть — `Load` перезапишет на загруженное значение.
- **Load до создания Server** — потому что `Server` хранит указатель на `tasks`. Если бы мы загрузили после, было бы то же самое (указатель не поменяется), но порядок «инициализация → использование» более понятен.
- **`log.Fatalf`** — печатает и сразу `os.Exit(1)`. Удобно для фатальных ошибок при старте.

## Уровень 5: про saveOrWarn

Простой helper на `Server`:

```go
func (s *Server) saveOrWarn() {
    if err := s.tasks.Save(dataFile); err != nil {
        log.Printf("Ошибка сохранения: %v", err)
    }
}
```

Использование:

```go
func (s *Server) createTask(w http.ResponseWriter, r *http.Request) {
    // ... валидация ...
    task := s.tasks.Add(input.Title)
    s.saveOrWarn()
    // ... ответ ...
}
```

Заметь порядок: **сначала** `Add` (изменение в памяти), **потом** `saveOrWarn` (запись на диск), **потом** ответ клиенту. Если меняешь местами — есть риск ответить успехом до записи.

## Уровень 6: про пакет log

Замена `fmt` на `log` для серверных сообщений:

```go
import "log"

log.Println("Сервер запущен")              // пишет с временной меткой в stderr
log.Printf("Ошибка: %v", err)              // как Printf, но с меткой
log.Fatalf("Критическая ошибка: %v", err)  // печатает и os.Exit(1)
log.Panicf("...")                           // печатает и panic
```

Преимущества `log` над `fmt`:

- Автоматическая временная метка
- Стандартный вывод в `stderr` (а не stdout) — серверные логи традиционно в stderr
- Можно настроить формат, добавить файл, имя процесса и т.д.
- `Fatalf` и `Panicf` экономят строки кода

В Проекте 5+ ты познакомишься с **структурным логированием** (`log/slog`) — это уже совсем серьёзная история. Сейчас классический `log` достаточно.

## Уровень 7: про порядок после изменения

```go
// ПРАВИЛЬНО
task := s.tasks.Add(input.Title)
s.saveOrWarn()
w.WriteHeader(http.StatusCreated)
json.NewEncoder(w).Encode(task)

// НЕПРАВИЛЬНО — отвечаем до сохранения
task := s.tasks.Add(input.Title)
w.WriteHeader(http.StatusCreated)
json.NewEncoder(w).Encode(task)
s.saveOrWarn()  // если упадёт — клиент уже получил 201
```

В первом случае: если `Save` упал, мы хотя бы залогировали ошибку до того, как сказали клиенту «всё хорошо». Во втором — нет даже лога.

В обоих случаях изменение **в памяти** уже произошло, и его не откатить — это компромисс нашего дизайна. Решает проблему БД с транзакциями.

## Уровень 8: про deadlock с двойным Lock

Если ты случайно сделаешь так:

```go
func (s *Server) createTask(w http.ResponseWriter, r *http.Request) {
    s.tasks.mu.Lock()                  // ВНЕШНЯЯ блокировка
    defer s.tasks.mu.Unlock()

    task := s.tasks.Add(input.Title)   // Add ТОЖЕ Lock-ает — DEADLOCK
}
```

— программа зависнет. Mutex нереентерабельный. **Никогда не лочь снаружи методов**, которые сами лочатся.

В нашем дизайне обработчик **не трогает** `s.tasks.mu` напрямую. Только вызывает методы.

## Если совсем тупик

Скажи Claude конкретно: «модуль 6, [конкретная проблема]». Самые частые ситуации:

- **«Файл не создаётся»** → проверь права на текущую папку, посмотри сообщение об ошибке в логе
- **«Программа зависает после первого запроса»** → ищи deadlock, скорее всего двойной Lock
- **«После перезапуска NextID начинается с 1, хотя в файле другое число»** → проверь, что Load вызывается **после** создания TaskList, и что JSON парсится правильно
- **«Marshal возвращает пустой объект"** → либо забыл публичные поля с тегами, либо у тебя поле `Mu` с большой буквы (поправь на `mu`)
