## Самопроверка через финальный сценарий

Полный прогон **с нуля**:

```bash
# 1. Чистая БД
docker rm -f todo-pg
docker volume rm todo-pg-data
docker run -d --name todo-pg -e POSTGRES_PASSWORD=secret -p 5432:5432 -v todo-pg-data:/var/lib/postgresql/data postgres:16
sleep 3

# 2. Миграции
export PGPASSWORD=secret
for f in migrations/*.sql; do psql -h localhost -U postgres -d postgres -f "$f"; done

# 3. Сервер
DATABASE_URL="postgres://postgres:secret@localhost:5432/postgres?sslmode=disable" go run . &
SERVER_PID=$!
sleep 2

# 4. CRUD
curl -s -X POST http://localhost:8080/tasks -H "Content-Type: application/json" -d '{"title":"Купить молоко"}'
curl -s -X POST http://localhost:8080/tasks -H "Content-Type: application/json" -d '{"title":"Позвонить маме"}'
curl -s -X PUT http://localhost:8080/tasks/1 -H "Content-Type: application/json" -d '{"title":"Купить молоко","done":true}'
curl -s http://localhost:8080/tasks
curl -s -X DELETE http://localhost:8080/tasks/2
curl -s http://localhost:8080/tasks

# 5. Перезапуск
kill $SERVER_PID
sleep 1
DATABASE_URL=... go run . &
SERVER_PID=$!
sleep 2

# 6. Данные на месте?
curl -s http://localhost:8080/tasks

# 7. Останавливаем
kill $SERVER_PID
docker stop todo-pg
```

Если **после перезапуска ты видишь те же данные** — модуль закрыт. Если ты можешь сделать всё это **без подсказок** — модуль закрыт **на отлично**.

## 🎉 Поздравляю, Проект 5 завершён!

Это был **самый важный проект** в твоём пути. Ты построил **полноценный бэкенд-сервис**: HTTP API + PostgreSQL + транзакции + миграции + connection pool.

Что у тебя теперь есть:

- Уверенное знание SQL: CREATE, SELECT, INSERT, UPDATE, DELETE, JOIN-ы (упомянуты), constraints, индексы
- Опыт работы с **миграциями** как с частью проекта
- Понимание **`database/sql`** и драйвера `pgx`
- Идиомы работы с `Query`, `QueryRow`, `Exec`, `Scan`, `RETURNING`
- Защита от **SQL injection** через placeholders
- Понимание **транзакций**, ACID, `defer Rollback`
- Знание **уровней изоляции** (хотя бы default)
- Опыт интеграции HTTP-сервера с БД
- Привычка различать **бизнес-ошибки** и **технические ошибки**
- Понимание **connection pool** и его настройки

Самые важные мысли проекта:

1. **SQL — это инвестиция на десятилетия.** Каждый бэкендер использует SQL каждый день. Раз освоил — навык не устаревает.

2. **БД — это не «костыль для хранения данных».** Это **специализированный сервис**, который решает задачи atomicity, isolation, durability **на порядок мощнее**, чем твой Go-код мог бы.

3. **`database/sql` — низкий уровень.** Это хорошо: ты точно знаешь, что и когда происходит. ORM-ы можно подключать **потом**, когда понимание уже есть.

4. **Транзакции — это не «продвинутая фича»**, это **базовый инструмент**. `Begin → defer Rollback → Commit` — паттерн, который ты будешь использовать в каждом серьёзном проекте.

5. **Архитектура слоями** (handler → repository → БД) — это не «оверинжиниринг», это **способ делать код легко изменяемым**. С этой структурой ты можешь поменять PostgreSQL на MySQL за день, не трогая обработчики.
