# Модуль 2. Long polling и echo-бот — условие

## Задание

Превратить «hello world» из модуля 1 в **рабочего эхо-бота**: он принимает сообщения через long polling и отвечает «Ты сказал: <текст>» на любое текстовое сообщение.

## Что должно произойти

```bash
$ BOT_TOKEN="..." go run .
2026/04/13 12:00:00 Бот запущен: @lev_todo_bot
```

И в Telegram:

```
Ты:    привет
Бот:   Ты сказал: привет

Ты:    как дела
Бот:   Ты сказал: как дела

Ты:    /start
Бот:   Ты сказал: /start
```

В логе сервера:

```
2026/04/13 12:00:00 Бот запущен: @lev_todo_bot
2026/04/13 12:00:15 Сообщение от @lev_chabanov: привет
2026/04/13 12:00:18 Сообщение от @lev_chabanov: как дела
2026/04/13 12:00:30 Сообщение от @lev_chabanov: /start
```

При остановке через Ctrl+C:

```
^C
exit status 130
```

(Корректное graceful shutdown будет в модуле 6.)

## Требования

1. Создан подпроект `006-todo-bot/practice-polling/` (или продолжаешь в `practice-setup` — твой выбор)
2. Расширен `TelegramClient` методами:
   - `GetUpdates(offset int64) ([]Update, error)`
   - `SendMessage(chatID int64, text string) error`
3. Добавлены типы:
   - `Update` с полями `UpdateID`, `Message *Message`
   - `Message` с полями `MessageID`, `From *User`, `Chat Chat`, `Text`
   - `Chat` с полями `ID`, `Type`
4. `http.Client` имеет таймаут **больше** 30 секунд (например, 60 секунд) — для long polling
5. `GetUpdates` использует параметр `timeout=30` в URL
6. Главный цикл:
   - Вызывает `GetMe` при старте, выводит имя бота
   - В бесконечном цикле вызывает `GetUpdates(offset)`
   - При ошибке — логирует, ждёт 5 секунд, продолжает
   - Для каждого `update` с непустым `Message.Text`:
     - Логирует «Сообщение от @username: text»
     - Отправляет ответ через `SendMessage`
   - **После обработки** обновляет `offset = update.UpdateID + 1`
7. Ответ бота: `"Ты сказал: " + msg.Text`
8. **Бот корректно работает в Telegram** (можно написать ему сообщение, он отвечает)
9. Все ошибки логируются, бот **не падает** при ошибке `SendMessage` или `GetUpdates`

## Шаги

1. Скопируй код из модуля 1 в новую папку (или продолжай в той же)
2. Добавь типы `Update`, `Message`, `Chat`
3. Реализуй `GetUpdates`
4. Реализуй `SendMessage` (метод HTTP POST с JSON-телом)
5. Перепиши `main`:
   - Создай клиент
   - Вызови `GetMe` для проверки
   - Объяви `var offset int64 = 0`
   - Бесконечный цикл с `GetUpdates`
6. Запусти, напиши боту — должен отвечать
7. Проверь, что после перезапуска бот **не отвечает на старые сообщения** (offset работает)

## Эксперименты

1. **Воспроизведи ловушку с offset:** в цикле **не обновляй** `offset`. Запусти, напиши боту что-то, увидь, что бот отвечает **много раз** на одно сообщение в бесконечном цикле. Это и есть «забыл двигать offset». Останови, верни как было.

2. **Поставь Client.Timeout в 5 секунд** (меньше long polling timeout). Запусти. Через несколько секунд увидишь ошибку «context deadline exceeded» или похожую — клиент порвёт соединение раньше, чем Telegram ответит. Верни 60 секунд.

3. **Замерь, сколько по времени занимает каждый getUpdates:**
   ```go
   start := time.Now()
   updates, err := bot.GetUpdates(offset)
   log.Printf("getUpdates занял %v, вернул %d обновлений", time.Since(start), len(updates))
   ```
   Заметь: если у тебя нет новых сообщений — будет ровно 30 секунд (long polling timeout). Если есть — почти мгновенно.

4. **Отправь боту длинное сообщение** (несколько абзацев). Бот ответит таким же длинным. Проверь, что Telegram не обрезает.

5. **Отправь боту 5 сообщений подряд без ожидания.** В одном `getUpdates` бот получит **массив из нескольких** обновлений сразу. Цикл `for _, upd := range updates` обработает их по очереди.

6. **Останови бота посреди работы**, напиши ему ещё несколько сообщений в Telegram, **запусти заново**. Что произошло? Бот **получил все накопленные сообщения** и ответил на них. Это потому, что Telegram **хранит** обновления, пока ты не подтвердишь их через offset.

7. **Сразу при первом запуске** офсет 0 — Telegram возвращает **все** обновления, которые помнит (до 24 часов). Чтобы пропустить старые при старте — можно сделать «холодный старт»: один `GetUpdates(-1)` (специальный offset, означающий «дай только последнее») и сразу взять `last.UpdateID + 1` как стартовый offset.

8. **Попробуй отправить эмодзи или китайские иероглифы.** Telegram передаёт UTF-8, Go хранит UTF-8 — всё должно работать без проблем. Проверь.

9. **Открой ngrok или другой туннель**, попробуй настроить webhook вместо long polling (опционально, продвинутое):
   ```bash
   curl "https://api.telegram.org/bot${BOT_TOKEN}/setWebhook?url=https://YOUR_NGROK_URL/webhook"
   ```
   После этого `getUpdates` начнёт возвращать ошибку «can't use getUpdates in webhook mode». Это альтернативный путь, и для production-ботов он часто предпочтителен. Чтобы вернуться к long polling — `setWebhook` с пустым URL.

## Что показать Claude

Покажи Claude свой главный цикл и спроси: «как мне обработать ситуацию, когда бот получил `getUpdates` ошибку 5 раз подряд? Стоит ли остановиться или продолжать вечно?». Это вопрос **дизайна resilience**:

- **Простой подход** — продолжать вечно, логировать. Если ошибки временные — пройдут.
- **Лучший подход** — exponential backoff: 1с, 2с, 4с, 8с... до некоторого максимума (например, 1 минута).
- **Production-grade** — circuit breaker: после N ошибок «открыть» цепь и не делать запросы какое-то время, потом попробовать снова.

Для модуля 2 простого подхода (5-секундная пауза) достаточно. Знание про более продвинутые схемы — обязательно.

Запиши себе на будущее: **долгоживущий процесс должен переживать любые временные ошибки**. Если упал — значит, ты что-то спроектировал неправильно, и будут разбираться сначала с твоим кодом, а потом с инфраструктурой.
