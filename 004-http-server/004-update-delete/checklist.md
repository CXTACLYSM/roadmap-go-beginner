## Самопроверка: полный CRUD-сценарий

Прогон с нуля. Запусти сервер, в другом терминале:

```bash
# 1. Список (захардкоженные задачи)
curl http://localhost:8080/tasks

# 2. Создание новой
curl -X POST http://localhost:8080/tasks \
    -H "Content-Type: application/json" \
    -d '{"title": "Тестовая задача"}'
# Запомни ID, например, 3

# 3. Получение по ID
curl http://localhost:8080/tasks/3

# 4. Обновление
curl -X PUT http://localhost:8080/tasks/3 \
    -H "Content-Type: application/json" \
    -d '{"title": "Обновлённая", "done": true}'

# 5. Список снова — должна быть обновлённая
curl http://localhost:8080/tasks

# 6. Удаление
curl -i -X DELETE http://localhost:8080/tasks/3
# 204 No Content

# 7. Список снова — задачи 3 нет
curl http://localhost:8080/tasks

# 8. GET удалённой → 404
curl -i http://localhost:8080/tasks/3
```

Если **все 8 шагов** прошли как ожидается — модуль закрыт. У тебя полноценный REST API.

## Подготовка к модулю 5

К модулю 5 у тебя есть рабочий REST API, **но в нём есть скрытая проблема** — race condition.

Каждый HTTP-запрос Go обрабатывает **в отдельной горутине**. Это значит, что **два запроса могут одновременно** изменять твой `TaskList`. Например:
- Запрос A добавляет задачу
- Запрос B тоже добавляет задачу
- Они могут «столкнуться» при `append` и `nextID++`

Сейчас ты этого не видишь, потому что тестируешь руками — между твоими `curl`-ами проходят секунды. Но если запустить **много параллельных запросов**, проблемы появятся.

В модуле 5 ты:
1. **Воспроизведёшь** race condition — увидишь, как два запроса портят данные
2. **Поймёшь**, что такое **data race** и почему она опасна
3. **Защитишь** код через `sync.Mutex` — самый базовый инструмент конкурентности в Go
4. **Узнаешь** про **race detector** в Go — встроенную проверку, которая ловит data race

Это — самый концептуально сложный модуль проекта. Но без него твой сервер **формально работает**, но **на нём небезопасно держать продакшен-данные**.
