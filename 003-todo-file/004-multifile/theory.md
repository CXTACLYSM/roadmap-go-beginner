# Модуль 4. Разделение кода по файлам — теория

## Контекст

После модуля 3 твой `main.go` содержит примерно 250–300 строк кода: тип `Task`, тип `TaskList` со всеми методами, `Save`/`Load`, handle-функции, REPL, helper-ы. Это уже **много** для одного файла. Когда ты ищешь нужную функцию, ты прокручиваешь и теряешься.

Решение — **разложить код по нескольким файлам**. Это твоё первое знакомство с организацией Go-проектов. Хорошая новость: в Go это **очень просто**. Никаких импортов между своими файлами, никаких header-файлов, никаких сложных правил. Все файлы в одном пакете автоматически видят друг друга.

Этот модуль концептуально лёгкий. Ты не учишь новый синтаксис. Ты учишься **расставлять границы** — навык, который кажется тривиальным, но на самом деле определяет качество всего проекта.

## Ядро 1: что такое пакет

В Go основная единица организации кода — **пакет** (package). Пакет — это **группа файлов в одной директории**, которые вместе образуют логическую единицу.

Каждый файл в пакете начинается со строки:

```go
package main
```

Все файлы в одной директории **должны** иметь одно и то же имя пакета. Если в директории есть `main.go` с `package main` и `task.go` с `package todo` — компилятор скажет, что это ошибка.

**Внутри пакета** все функции, типы и переменные **видны друг другу** автоматически. Никаких импортов внутри пакета. Никаких объявлений «эта функция определена там-то». Просто пиши `findIndex(...)` в одном файле — и она будет найдена в любом другом файле того же пакета.

Эту лёгкость в обмене мы не получаем при работе **между** разными пакетами. Туда уже нужны импорты, видимость через большую/маленькую букву, и так далее. Но **внутри** пакета — полная свобода.

## Ядро 2: разница между пакетом и проектом

Это часто путают, поэтому проясним:

- **Пакет** = одна директория с файлами одного `package`
- **Проект** = вся твоя программа, может состоять из одного или многих пакетов
- **Модуль** (`go.mod`) = совокупность пакетов с общим управлением зависимостями. Корень — там, где `go.mod`.

В этом модуле мы остаёмся в **одном пакете** (`package main`), который содержит **несколько файлов**. Это самый простой случай.

В Проектах 4–7 ты познакомишься с многопакетной организацией: `internal/handler`, `internal/storage`, `cmd/api/main.go` и так далее. Сейчас — пока нет.

## Ядро 3: как делить код на файлы

Главный вопрос: **по какому критерию** делить?

Хорошие критерии:

1. **По логической ответственности.** Тип `Task` и его методы → один файл. Тип `TaskList` со своими методами → другой. REPL → третий. Это самый частый и самый осмысленный подход.

2. **По «слою».** Слой данных (типы) → один файл. Слой приложения (handlers) → другой. Слой ввода-вывода (REPL) → третий. Это похоже на критерий по ответственности, но более грубое.

3. **По объёму.** Если файл стал больше ~300 строк — разделить пополам. Это **аварийный** критерий, не основной.

Плохие критерии:

- ❌ **По размеру строк** просто так. Файл из 1000 строк, который делает одну вещь, — не повод его делить. Файл из 50 строк, который делает три разных вещи, — повод.
- ❌ **По времени написания.** «Это я писал в понедельник, это в среду» — это не критерий.
- ❌ **По типу языковой конструкции.** «Все типы в один файл, все функции в другой» — приводит к нелогичной структуре, в которой типы оторваны от методов.

## Структура для нашего проекта

Логичное разделение для текущего кода:

```
app/
├── go.mod
├── main.go         ← точка входа, REPL, helper'ы
├── task.go         ← тип Task и метод String
├── tasklist.go     ← тип TaskList и все его методы (Add, MarkDone, Remove, Save, Load, ...)
├── handlers.go     ← handle-функции и parseCommand/parseID
└── tasks.json      ← данные (создаётся в рантайме)
```

Принцип: **близкие по смыслу вещи лежат рядом**. Если ты завтра захочешь поменять, как сериализуется задача — ты идёшь в `task.go`. Если поменять логику команды `done` — `handlers.go`. Если добавить новую команду в REPL — `main.go`.

Заметь: я выделил `handlers.go` от `main.go` — это **разделение слоёв**, о котором мы говорили в модуле 5 Проекта 2. `main` знает про цикл и парсинг, `handlers` — про команды и их логику.

## Ядро 4: как запускать многофайловый проект

Раньше ты запускал так:

```bash
go run main.go
```

Это работало, потому что у тебя был **один** файл. Теперь файлов несколько, и эта команда **не сработает**:

```bash
go run main.go
# ./main.go:42:5: undefined: Task
```

Компилятор не знает про другие файлы. Чтобы сказать ему «компилируй всё в этой папке», используют точку:

```bash
go run .
```

Точка означает «текущая директория, как пакет». Это нужная форма для многофайловых проектов.

**Альтернативно**, можно перечислить все файлы:

```bash
go run main.go task.go tasklist.go handlers.go
```

— но это неудобно и легко забыть файл. **Используй `go run .`** всегда, когда у тебя больше одного файла.

То же самое для других команд:

- `go build .` — собрать пакет в исполняемый файл
- `go test .` — запустить тесты пакета
- `go vet .` — статический анализ

## Ядро 5: видимость внутри одного пакета

Помнишь правило про большую и маленькую букву?

- **С большой буквы** = публичное (видно из других пакетов)
- **С маленькой буквы** = приватное (видно только из своего пакета)

**Важно:** внутри одного пакета **разница не имеет значения**. Файл `handlers.go` спокойно вызовет приватную функцию `findIndex` из файла `tasklist.go`, даже если она с маленькой буквы. Внутри пакета все файлы — «свои».

Регистр имеет значение **только когда твой пакет начинают использовать снаружи**. Тогда `findIndex` действительно станет невидимой, а `MarkDone` — будет работать.

Поэтому, даже если ты сейчас всё держишь в одном пакете, **полезно соблюдать соглашения**:

- То, что **наружу** имеет смысл показывать — с большой буквы
- То, что **внутреннее** — с маленькой

В Проекте 4 ты разделишь код на пакеты, и эта дисциплина окупится.

## Что переносить в каком файле

Конкретный план переноса:

**`task.go`**:
```go
package main

import "fmt"

type Task struct {
    ID    int    `json:"id"`
    Title string `json:"title"`
    Done  bool   `json:"done"`
}

func (t Task) String() string {
    status := "[ ]"
    if t.Done {
        status = "[x]"
    }
    return fmt.Sprintf("%d. %s %s", t.ID, status, t.Title)
}
```

**`tasklist.go`**:
```go
package main

import (
    "encoding/json"
    "errors"
    "fmt"
    "os"
)

type TaskList struct {
    Tasks  []Task `json:"tasks"`
    NextID int    `json:"nextID"`
}

func (tl *TaskList) Add(title string) Task { ... }
func (tl *TaskList) MarkDone(id int) error { ... }
func (tl *TaskList) Remove(id int) error { ... }
func (tl *TaskList) findIndex(id int) int { ... }
func (tl *TaskList) All() []Task { ... }
func (tl *TaskList) Save(filename string) error { ... }
func (tl *TaskList) Load(filename string) error { ... }
```

**`handlers.go`**:
```go
package main

import (
    "fmt"
    "strconv"
    "strings"
)

func parseCommand(line string) (string, string) { ... }
func parseID(arg, command string) (int, error) { ... }
func handleAdd(list *TaskList, arg string) { ... }
func handleList(list *TaskList) { ... }
func handleDone(list *TaskList, arg string) { ... }
func handleRemove(list *TaskList, arg string) { ... }
```

**`main.go`**:
```go
package main

import (
    "bufio"
    "fmt"
    "os"
    "strings"
)

const dataFile = "tasks.json"

func main() {
    list := &TaskList{NextID: 1}
    if err := list.Load(dataFile); err != nil { ... }
    // ... REPL ...
}

func saveOrWarn(list *TaskList) { ... }
```

Заметь: **каждый файл импортирует только то, что ему нужно**. `task.go` нужен только `fmt`. `handlers.go` — `fmt`, `strconv`, `strings`. И так далее. Это чистый код.

## Ядро 6: что делать при ошибках разделения

Самые частые проблемы при разделении:

**Дублирующие импорты.** Если ты переносишь функцию в новый файл и забыл добавить туда нужный импорт — компилятор скажет «undefined: ...». Решение: посмотри, какие пакеты использует функция, и добавь в импорты файла.

**Неиспользуемые импорты.** Если ты унёс функцию из `main.go`, но забыл удалить её импорт оттуда — компилятор скажет «imported and not used». Это специально: Go не разрешает «висячие» импорты. Удали ненужный.

**Циклические зависимости.** Это когда файл A нужен для файла B, а файл B нужен для файла A. **Внутри одного пакета такого не бывает** — они все «общаются» свободно. Циклы возможны только между пакетами, и Go их жёстко запрещает. В этом модуле тебе это не грозит.

**`gofmt` и `goimports`.** В Go есть автоматический форматёр кода. После каждого изменения запускай:

```bash
gofmt -w .
```

Или, ещё лучше, `goimports` — он **сам** подбирает нужные импорты:

```bash
goimports -w .
```

(Ставится отдельно: `go install golang.org/x/tools/cmd/goimports@latest`.)

В VSCode с Go-расширением это происходит автоматически при сохранении файла. Привыкай — это норма Go-разработки.
