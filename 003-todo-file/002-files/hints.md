# Модуль 2. Работа с файлами — подсказки

> ⚠️ Открывай только после 30 минут самостоятельной борьбы.

## Уровень 1: наводящие вопросы

1. Какая функция читает весь файл за один вызов?
2. Какая функция записывает байты в файл за один вызов?
3. Что вернёт `os.ReadFile` для несуществующего файла?
4. Какая функция помогает понять, **какая именно** это ошибка?
5. Почему «файла нет» в нашей программе — это **не** ошибка?

## Уровень 2: про os.ReadFile и os.WriteFile

Обе функции — твои основные инструменты. Сигнатуры:

```go
func ReadFile(name string) ([]byte, error)
func WriteFile(name string, data []byte, perm FileMode) error
```

Использование:

```go
// Чтение
data, err := os.ReadFile("tasks.json")
if err != nil { ... }

// Запись
err := os.WriteFile("tasks.json", data, 0644)
if err != nil { ... }
```

`FileMode` — это просто число. Передавай `0644` (так Go понимает, что это восьмеричная запись прав).

## Уровень 3: про обёртывание ошибок через %w

Просто паттерн:

```go
data, err := os.ReadFile(filename)
if err != nil {
    return nil, fmt.Errorf("не могу прочитать %s: %w", filename, err)
}
```

`%w` (от **w**rap) — особый глагол `Errorf`, который **сохраняет оригинальную ошибку внутри новой**. Снаружи это выглядит как одна ошибка с расширенным сообщением:

```
не могу прочитать tasks.json: open tasks.json: no such file or directory
```

— но внутри хранится **связь** с оригинальной ошибкой. Это позволяет использовать `errors.Is` дальше по стеку:

```go
_, err := loadTasks("nonexistent.json")
if errors.Is(err, os.ErrNotExist) {
    // даже если err — это обёрнутая ошибка, errors.Is найдёт оригинал внутри
}
```

Если использовать `%s` или `%v` вместо `%w` — связь теряется, и `errors.Is` не работает. Поэтому **в Go-коде почти всегда `%w`**, не `%s`, для оборачивания ошибок.

## Уровень 4: про errors.Is

```go
import "errors"

if errors.Is(err, os.ErrNotExist) {
    // эта ошибка — это (или содержит в себе) ErrNotExist
}
```

`errors.Is` смотрит:
1. Совпадает ли `err` с `target` напрямую?
2. Если `err` — обёрнутая ошибка, **разворачивает** её и проверяет внутреннюю
3. Повторяет рекурсивно

Это правильный способ проверки типа ошибки. **Никогда не сравнивай ошибки через `err == os.ErrNotExist`** — это работает только для не-обёрнутых, и однажды сломается, когда кто-то добавит обёртку.

## Уровень 5: про обработку «файла нет»

Шаблон, который нужно запомнить:

```go
func loadTasks(filename string) ([]Task, error) {
    data, err := os.ReadFile(filename)
    if err != nil {
        if errors.Is(err, os.ErrNotExist) {
            return []Task{}, nil  // нет файла = пустой список, не ошибка
        }
        return nil, fmt.Errorf("не могу прочитать %s: %w", filename, err)
    }
    // ... парсинг JSON ...
}
```

Это **сознательное решение** — превратить одну техническую ошибку в нормальное возвращаемое значение. Дизайн ваших функций должен быть удобен **тому, кто их вызывает**, а не отражать «как оно технически устроено».

Возвращаем `[]Task{}` (пустой слайс), а не `nil`. Обе формы валидны для Go (`range` и `len` работают на обеих), но `[]Task{}` яснее показывает намерение «это пустой список, а не отсутствие данных».

## Уровень 6: про объявление пустого слайса

```go
return []Task{}, nil
```

Это создаёт **пустой, но не nil** слайс. Альтернативы:

```go
return nil, nil          // тоже работает, но семантически менее ясно
return make([]Task, 0), nil  // то же что []Task{}, длиннее
var empty []Task; return empty, nil  // nil-слайс, два шага
```

Для возвратов из функции — самое понятное `[]Task{}, nil`.

## Уровень 7: про порядок проверок

Правильный порядок в `loadTasks`:

1. **Сначала** проверка ошибки чтения (с разделением на «нет файла» и «другая ошибка»)
2. **Потом** парсинг JSON (если чтение прошло)
3. **Потом** проверка ошибки парсинга

Не наоборот. Если ты сначала попытаешься парсить, а потом проверить ошибку чтения — программа упадёт на парсинге `nil`.

## Если совсем тупик

Скажи Claude конкретно: «модуль 2, не работает [конкретная штука]». Покажи код. Самые частые ситуации:

- «`errors.Is` всегда возвращает `false`» → проверь, используешь ли ты `%w` для оборачивания ошибки
- «Файл не создаётся» → проверь права на папку (есть ли у тебя право писать туда)
- «Странное содержимое в файле» → проверь, что используешь `MarshalIndent`, и что данные передаются правильно
