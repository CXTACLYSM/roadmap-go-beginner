# Модуль 5. Race conditions и мьютексы — теория

## Контекст

Этот модуль — **самый концептуально сложный** во всём проекте 4. До него ты делал «функционально работающий» REST API. После него ты начнёшь понимать, **как сделать его безопасным** в условиях многих одновременных пользователей.

В модуле ты:

1. **Увидишь** race condition своими глазами
2. **Узнаешь**, что такое data race и почему она опасна
3. **Научишься** защищать данные через `sync.Mutex`
4. **Познакомишься** с race detector — встроенным инструментом Go для обнаружения гонок

Это базовые знания, без которых нельзя писать настоящие серверы.

## Ядро 1: что такое race condition

**Race condition** (гонка) — это ошибка, которая возникает, когда **результат программы зависит от того, в каком порядке выполнились параллельные операции**.

В Go каждый HTTP-запрос обрабатывается в **отдельной горутине**. Это значит, что если два клиента одновременно делают POST `/tasks`, твои обработчики работают **параллельно**. И если они оба обращаются к одной и той же переменной (`taskList`) — может произойти что угодно.

Конкретный пример. Метод `Add`:

```go
func (tl *TaskList) Add(title string) Task {
    task := Task{ID: tl.NextID, Title: title}
    tl.Tasks = append(tl.Tasks, task)
    tl.NextID++
    return task
}
```

Представь, что **две** горутины выполняют этот метод одновременно. Что может пойти не так?

**Сценарий А: оба читают NextID до того, как кто-то из них его инкрементировал**

```
Время:  Горутина 1                  Горутина 2
  1     прочитала NextID = 5
  2                                  прочитала NextID = 5
  3     создала Task{ID: 5, ...}
  4                                  создала Task{ID: 5, ...}
  5     append, NextID = 6
  6                                  append, NextID = 6
```

Результат: **две задачи с одинаковым ID 5**. После цикла `NextID = 6`, хотя должно было быть 7. Состояние данных нарушено.

**Сценарий Б: одновременный append**

```go
tl.Tasks = append(tl.Tasks, task)
```

Эта одна строка на самом деле — **несколько операций**:

1. Прочитать текущий слайс `tl.Tasks`
2. Если есть место (cap > len) — записать новый элемент в позицию `len`, увеличить `len`
3. Если нет — выделить новый массив, скопировать туда старые элементы, добавить новый, обновить указатель слайса

Если две горутины делают это одновременно — операции **переплетаются**. Возможно что угодно: потеряется элемент, оба элемента запишутся в одну позицию, или Go вообще паникнет с непредсказуемой ошибкой.

## Ядро 2: data race vs race condition

Эти два термина часто путают. Различие важное:

- **Data race** — это когда **две горутины обращаются к одной памяти**, и хотя бы одна из них **пишет**, и нет синхронизации. Это **формальное** определение, его можно автоматически обнаружить.
- **Race condition** — это **логическая** ошибка, когда результат зависит от порядка выполнения. Может включать data race, а может и не включать.

Пример race condition без data race:

```go
// Горутина 1
if !exists("file.txt") {
    create("file.txt")
}

// Горутина 2 (одновременно)
if !exists("file.txt") {
    create("file.txt")
}
```

Здесь нет общей памяти, но есть гонка: обе горутины могут увидеть, что файла нет, и обе попытаются его создать.

Для нашей задачи самое важное — это **data race** в `TaskList`. И к счастью, Go умеет её **автоматически обнаруживать**.

## Ядро 3: race detector в Go

Go умеет обнаруживать data race во время выполнения через специальный флаг:

```bash
go run -race .
```

С этим флагом Go вставит дополнительный код в твою программу, который следит за всеми обращениями к памяти. Если он обнаружит data race — выведет подробное сообщение с местом, где это произошло.

Минусы: программа становится в ~10 раз медленнее. Поэтому **race detector используется только для тестирования**, не для продакшена.

**Это инструмент, которым ты обязан уметь пользоваться.** В Go-проектах принято запускать тесты с `-race` в CI. Если в тестах поймали race — это блокер на мерж.

## Ядро 4: как воспроизвести race

Чтобы увидеть race своими глазами, нужно:

1. Запустить сервер с `-race`
2. Сделать **много параллельных POST-запросов**

Самый простой способ для пункта 2 — `xargs` + `curl` в bash:

```bash
seq 1 100 | xargs -n1 -P20 -I{} curl -s -X POST http://localhost:8080/tasks \
    -H "Content-Type: application/json" \
    -d '{"title": "test {}"}'
```

Что происходит: создаём числа от 1 до 100, для каждого запускаем `curl` с **20 параллельными процессами** (`-P20`). Это симулирует 20 одновременных пользователей.

После этого:

```bash
curl http://localhost:8080/tasks | jq length
```

Должно быть 100 + изначальные задачи. Если **меньше** — это и есть потерянные данные из-за race.

И в логе сервера (если запускал с `-race`) ты увидишь что-то вроде:

```
==================
WARNING: DATA RACE
Read at 0x00c0001a0040 by goroutine 12:
  main.(*TaskList).Add()
      /home/.../tasklist.go:21
...
Previous write at 0x00c0001a0040 by goroutine 8:
  main.(*TaskList).Add()
      /home/.../tasklist.go:23
...
==================
```

Это **доказательство**, что у тебя гонка. Не «может быть», а **точно**.

## Ядро 5: sync.Mutex — базовая защита

Самый простой инструмент защиты — **мьютекс** (mutual exclusion). Это «замок», который пускает внутрь критической секции только **одну** горутину одновременно.

В Go мьютекс — это `sync.Mutex` из пакета `sync`:

```go
import "sync"

type TaskList struct {
    Tasks  []Task `json:"tasks"`
    NextID int    `json:"nextID"`
    mu     sync.Mutex
}
```

Использование:

```go
func (tl *TaskList) Add(title string) Task {
    tl.mu.Lock()
    defer tl.mu.Unlock()

    task := Task{ID: tl.NextID, Title: title}
    tl.Tasks = append(tl.Tasks, task)
    tl.NextID++
    return task
}
```

Несколько важных моментов:

1. **`mu sync.Mutex`** — поле в структуре, **с маленькой буквы** (приватное). Мьютекс — это деталь реализации, наружу его показывать не надо.

2. **`tl.mu.Lock()`** — захватить замок. Если другая горутина уже захватила его — текущая будет **ждать**, пока тот её не отпустит.

3. **`defer tl.mu.Unlock()`** — отпустить замок при **выходе из функции** (даже если выход через panic). `defer` — это отложенный вызов, выполняется в конце функции.

4. **Используй `defer` для Unlock — всегда.** Если ты забудешь Unlock или не отпустишь его при ошибке — мьютекс останется захваченным навсегда, и все следующие горутины зависнут (deadlock).

## Ядро 6: где ставить мьютекс

Мьютекс должен защищать **все** обращения к данным. Если ты залочил `Add`, но забыл про `GetByID` — `GetByID` всё равно может прочитать «грязные» данные во время чужого `Add`.

Защищать нужно **каждый** метод, который читает или пишет в `TaskList`:

```go
func (tl *TaskList) All() []Task {
    tl.mu.Lock()
    defer tl.mu.Unlock()
    return tl.Tasks  // Внимание: возвращаем СЛАЙС, который потом будут читать снаружи
}

func (tl *TaskList) GetByID(id int) (Task, error) { tl.mu.Lock(); defer tl.mu.Unlock(); ... }
func (tl *TaskList) Add(title string) Task { tl.mu.Lock(); defer tl.mu.Unlock(); ... }
func (tl *TaskList) Update(id int, ...) { tl.mu.Lock(); defer tl.mu.Unlock(); ... }
func (tl *TaskList) Remove(id int) error { tl.mu.Lock(); defer tl.mu.Unlock(); ... }
```

## ⚠️ Тонкость: возврат слайса из All

Метод `All()` возвращает `tl.Tasks` — это **тот же слайс**, который защищается мьютексом. После возврата вызывающая сторона может его читать **уже без мьютекса**. И если в этот момент кто-то делает `Add`, то происходит race на слайсе **снаружи** функции.

Решение — **возвращать копию**:

```go
func (tl *TaskList) All() []Task {
    tl.mu.Lock()
    defer tl.mu.Unlock()

    copy := make([]Task, len(tl.Tasks))
    for i, t := range tl.Tasks {
        copy[i] = t
    }
    return copy
}
```

Или ещё короче через builtin `copy`:

```go
result := make([]Task, len(tl.Tasks))
copy(result, tl.Tasks)
return result
```

`Task` — структура без указателей, поэтому `copy` копирует значения, и снаружи нельзя случайно изменить оригинал.

**Это ключевая тонкость**, которую новички часто пропускают. Защитить только метод недостаточно — надо ещё убедиться, что **возвращаемые данные** тоже изолированы.

## Ядро 7: sync.RWMutex для оптимизации

Если у тебя **много чтений** и мало записей, обычный `Mutex` слишком строгий: даже два чтения, которые не мешают друг другу, не могут произойти параллельно.

**`sync.RWMutex`** — это «читательско-писательский» мьютекс:

```go
type TaskList struct {
    Tasks  []Task `json:"tasks"`
    NextID int    `json:"nextID"`
    mu     sync.RWMutex
}

// Чтение — RLock
func (tl *TaskList) GetByID(id int) (Task, error) {
    tl.mu.RLock()
    defer tl.mu.RUnlock()
    // ...
}

// Запись — Lock
func (tl *TaskList) Add(title string) Task {
    tl.mu.Lock()
    defer tl.mu.Unlock()
    // ...
}
```

- **`RLock`/`RUnlock`** — для **чтения**. Несколько читателей могут работать **одновременно**.
- **`Lock`/`Unlock`** — для **записи**. Эксклюзивный доступ, никто другой не может ни читать, ни писать.

Это даёт прирост производительности на read-heavy нагрузках. Для нашего проекта он не критичен, но **знать про RWMutex обязательно**.

**Используй обычный Mutex в этом модуле.** RWMutex — это упоминание, не задание.

## Ядро 8: альтернативы мьютексу

В Go есть и другие способы безопасного обмена данными:

- **Каналы** (`chan`) — основная идиома Go: «не общайся через память, общайся через каналы». Подходит для передачи данных между горутинами.
- **Атомарные операции** (`sync/atomic`) — для простых операций над числами без блокировок. Очень быстро, но ограничено по функционалу.
- **`sync.Map`** — потокобезопасная мапа. Подходит, если у тебя key-value хранилище и нужны очень частые чтения.

Для нашей задачи (защита структуры данных) `sync.Mutex` — самый прямой и понятный вариант. Каналы здесь были бы оверкилл.

## Полный пример

```go
package main

import (
    "fmt"
    "sync"
)

type Task struct {
    ID    int    `json:"id"`
    Title string `json:"title"`
    Done  bool   `json:"done"`
}

type TaskList struct {
    Tasks  []Task `json:"tasks"`
    NextID int    `json:"nextID"`
    mu     sync.Mutex
}

func (tl *TaskList) All() []Task {
    tl.mu.Lock()
    defer tl.mu.Unlock()

    result := make([]Task, len(tl.Tasks))
    copy(result, tl.Tasks)
    return result
}

func (tl *TaskList) GetByID(id int) (Task, error) {
    tl.mu.Lock()
    defer tl.mu.Unlock()

    for _, t := range tl.Tasks {
        if t.ID == id {
            return t, nil
        }
    }
    return Task{}, fmt.Errorf("задача не найдена: %d", id)
}

func (tl *TaskList) Add(title string) Task {
    tl.mu.Lock()
    defer tl.mu.Unlock()

    task := Task{ID: tl.NextID, Title: title}
    tl.Tasks = append(tl.Tasks, task)
    tl.NextID++
    return task
}

func (tl *TaskList) Update(id int, title string, done bool) (Task, error) {
    tl.mu.Lock()
    defer tl.mu.Unlock()

    for i := range tl.Tasks {
        if tl.Tasks[i].ID == id {
            tl.Tasks[i].Title = title
            tl.Tasks[i].Done = done
            return tl.Tasks[i], nil
        }
    }
    return Task{}, fmt.Errorf("задача не найдена: %d", id)
}

func (tl *TaskList) Remove(id int) error {
    tl.mu.Lock()
    defer tl.mu.Unlock()

    for i, t := range tl.Tasks {
        if t.ID == id {
            tl.Tasks = append(tl.Tasks[:i], tl.Tasks[i+1:]...)
            return nil
        }
    }
    return fmt.Errorf("задача не найдена: %d", id)
}
```

Заметь: я **встроил** поиск напрямую в каждый метод, не используя приватный `findIndex`. Почему? Потому что `findIndex` тоже потребовал бы локирования, и тогда у нас была бы вложенная блокировка одного и того же мьютекса — это **deadlock**.

Альтернатива — оставить `findIndex` без блокировки и **обязать вызывающий код** держать блокировку. Это работает, но опасно: легко забыть. Поэтому проще не использовать helper.

## Ядро 9: deadlock и как его избежать

**Deadlock** — это ситуация, когда две (или больше) горутин **навсегда** ждут друг друга. Самый простой пример — попытка повторно захватить тот же мьютекс:

```go
func (tl *TaskList) BadMethod() {
    tl.mu.Lock()
    defer tl.mu.Unlock()

    tl.AnotherMethod()  // Эта функция тоже делает Lock()
}

func (tl *TaskList) AnotherMethod() {
    tl.mu.Lock()         // ЗАВИСНЕТ ЗДЕСЬ — мьютекс уже захвачен
    defer tl.mu.Unlock()
    // ...
}
```

`sync.Mutex` в Go **не реентерабельный** — то есть та же горутина, которая уже захватила мьютекс, **не может захватить его снова**. Это сделано намеренно: реентерабельные мьютексы порождают сложные баги.

Правила, которые помогут избежать deadlock:

1. **Не вызывай метод, который тоже Lock-ает, изнутри метода с Lock**
2. **Если нужно вызывать — выноси в приватный метод без Lock и обозначай его комментарием**
3. **`defer Unlock` всегда** — чтобы не забыть отпустить
4. **Никаких вложенных вызовов с разными мьютексами** — это путь к классическому deadlock между двумя горутинами
