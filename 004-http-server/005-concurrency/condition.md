# Модуль 5. Race conditions и мьютексы — условие

## Задание

**Воспроизвести race condition на своём API**, увидеть её через race detector, и **защитить** код через `sync.Mutex`. Поведение API для одиночных запросов должно остаться **точно таким же**, но теперь сервер будет корректно работать под параллельной нагрузкой.

## Что должно произойти

### Часть 1: воспроизводим проблему

Запусти сервер с race detector:

```bash
$ go run -race .
Сервер запущен на http://localhost:8080
```

В другом терминале запусти 100 параллельных POST-запросов:

```bash
$ seq 1 100 | xargs -n1 -P20 -I{} curl -s -X POST http://localhost:8080/tasks \
    -H "Content-Type: application/json" \
    -d '{"title":"task {}"}' > /dev/null
```

Посмотри лог сервера — должен появиться отчёт о data race:

```
==================
WARNING: DATA RACE
Read at 0x... by goroutine N:
  main.(*TaskList).Add()
      .../tasklist.go:LINE
...
==================
```

И посмотри, сколько задач реально создалось:

```bash
$ curl -s http://localhost:8080/tasks | jq length
```

Может быть **меньше**, чем 100 + изначальные. Часть запросов потерялась.

### Часть 2: исправляем

Добавь `sync.Mutex` в `TaskList`, защити **все** методы. Перезапусти сервер с `-race`, повтори нагрузочный тест. Теперь:

- В логе **нет** WARNING DATA RACE
- Количество задач — **ровно 100 + изначальные**
- Все ID **уникальны**

## Требования

1. **Сначала — воспроизведи проблему.** Не пропускай этот шаг. Без личного опыта «вот она, race» — теория не закрепится.
2. **Запиши в комментарии** или в отдельный текстовый файл: «До защиты: получено N задач из 100». Это твоё доказательство.
3. Добавь поле `mu sync.Mutex` в `TaskList`. Поле **с маленькой буквы** (приватное).
4. Защити **все** методы `TaskList`:
   - `All`, `GetByID`, `Add`, `Update`, `Remove`
   - `Save` и `Load` (если ты их уже добавил, иначе — в модуле 6)
5. Используй паттерн `Lock + defer Unlock` в каждом методе.
6. Метод `All` возвращает **копию** слайса, а не сам `tl.Tasks`.
7. **Убери `findIndex`** или преобразуй его в неблокирующий приватный метод (см. подсказки) — иначе будет deadlock.
8. После исправления повтори нагрузочный тест и убедись, что:
   - Race detector не ругается
   - Количество задач совпадает с ожидаемым
9. **Запусти `go test -race ./...`** (даже если у тебя нет тестов — он должен вернуть `no test files`, не упасть).

## Шаги

1. Запусти сервер **без** `-race`. Прогон `xargs` команды. Посмотри количество задач — может быть равно 100, может быть меньше. Если равно — попробуй увеличить нагрузку до 200 или 500.
2. Останови, запусти **с** `-race`. Повтори. На этот раз ты должен увидеть WARNING в логе.
3. Останови сервер.
4. Открой `tasklist.go`, импортируй `sync`.
5. Добавь поле `mu sync.Mutex` в структуру.
6. В каждом методе — `tl.mu.Lock(); defer tl.mu.Unlock()` в начале.
7. **Поправь `All`**: верни копию через `make` + `copy`.
8. **Поправь `findIndex`**: либо удали (заменив на встроенный поиск в каждом методе), либо переименуй в `findIndexUnlocked` с комментарием «вызывающий должен держать mu.Lock()».
9. Проверь, что компилируется.
10. Запусти `go run -race .`, повтори нагрузочный тест.
11. Убедись, что нет WARNING.
12. Проверь количество задач — должно быть ровно как ожидалось.
13. Прогон обычных эндпоинтов через `curl` — всё должно работать как раньше.

## Эксперименты

1. **Запусти нагрузочный тест с `-P1`** (один процесс):
   ```bash
   seq 1 100 | xargs -n1 -P1 -I{} curl ...
   ```
   Race detector ругнётся? Скорее всего нет — потому что параллелизма нет, всё последовательно. Но это не значит, что race нет — просто её не воспроизвели.

2. **Забудь `defer Unlock`** в одном из методов:
   ```go
   func (tl *TaskList) Add(title string) Task {
       tl.mu.Lock()
       // нет defer Unlock!
       // ...
       return task
   }
   ```
   Запусти, попробуй сделать два запроса подряд. Что произошло? Второй зависнет. Это **deadlock** из-за забытого Unlock. Верни `defer`.

3. **Сделай вложенный Lock** (искусственный deadlock):
   ```go
   func (tl *TaskList) Add(title string) Task {
       tl.mu.Lock()
       defer tl.mu.Unlock()

       tl.mu.Lock()  // Попытка повторно захватить — зависнет
       defer tl.mu.Unlock()
       // ...
   }
   ```
   Запусти, сделай POST. Запрос зависнет. Это **deadlock из-за нереентерабельности**. Верни как было.

4. **Убери `copy` в методе `All`** — верни прямо `tl.Tasks`. Запусти race detector с командой:
   ```bash
   while true; do curl http://localhost:8080/tasks; done &
   while true; do curl -X POST http://localhost:8080/tasks -d '{"title":"x"}' -H "Content-Type: application/json"; done &
   ```
   Через несколько секунд — получишь WARNING. Это та самая «защитили метод, но утечку через возврат не закрыли». Верни `copy`.

5. **Замени Mutex на RWMutex.** Защити чтения через `RLock/RUnlock`, записи через `Lock/Unlock`. Прогон тех же тестов. Работает? Какой профит на read-heavy нагрузке (много `GET /tasks` и мало `POST`)?

6. **Запусти `go test -race ./...`** — даже без тестов команда должна выполниться чисто.

7. **Бонус: напиши простой тест**, который параллельно вызывает `Add` и проверяет финальное состояние:
   ```go
   func TestParallelAdd(t *testing.T) {
       tl := &TaskList{NextID: 1}
       var wg sync.WaitGroup
       for i := 0; i < 100; i++ {
           wg.Add(1)
           go func() {
               defer wg.Done()
               tl.Add("test")
           }()
       }
       wg.Wait()
       if len(tl.Tasks) != 100 {
           t.Errorf("ожидалось 100 задач, получено %d", len(tl.Tasks))
       }
   }
   ```
   Запусти `go test -race`. С защищённой версией — пройдёт. Без защиты — упадёт или поймает race.

## Что показать Claude

Покажи Claude свой `tasklist.go` после защиты и спроси: «всё ли защищено? Есть ли места, где я могу прочитать или записать данные без блокировки?». Особенно — про возврат слайса из `All`. Это тот самый момент, который пропускают 80% новичков.

И ещё — спроси: «как бы я переделал эту защиту, если бы у меня было **много чтений** и **мало записей**?». Ответ — `RWMutex`, и Claude поможет тебе увидеть, в каких случаях это даёт прирост.

В Проекте 6/7 — про каналы, контексты, errgroup. Сейчас твоя главная задача — **прочувствовать**, что race condition — это не теоретическая страшилка, а реальная проблема, которая существует в твоём собственном коде.
