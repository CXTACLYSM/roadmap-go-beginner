## Уровень 2: про воспроизведение race

Команда для нагрузочного теста:

```bash
seq 1 100 | xargs -n1 -P20 -I{} curl -s -X POST http://localhost:8080/tasks \
    -H "Content-Type: application/json" \
    -d '{"title":"task {}"}' > /dev/null
```

Расшифровка:
- **`seq 1 100`** — выводит числа 1, 2, ..., 100 по строке
- **`xargs -n1 -P20 -I{}`** — для каждой строки запускает команду, **20 параллельно**, заменяя `{}` на значение строки
- **`curl ... > /dev/null`** — выкидываем вывод curl, нас интересуют только побочные эффекты
- **`> /dev/null`** в конце xargs — тоже выкидываем

После этого:

```bash
curl -s http://localhost:8080/tasks | jq length
```

`jq` — утилита для работы с JSON, выводит длину массива. Если у тебя её нет:
```bash
sudo apt install jq        # Ubuntu/Debian
brew install jq            # macOS
```

Или просто посчитай задачи на глаз через `curl http://localhost:8080/tasks | grep -o '"id"' | wc -l`.

## Уровень 3: про -race

```bash
go run -race .
```

Этот флаг **обязательно** запоминай. Он включает встроенный детектор гонок. Программа становится в ~10 раз медленнее, но за это получает суперспособность видеть race condition в реальном времени.

Если в логе нет WARNING — ты не воспроизвёл гонку. Попробуй:
- Увеличить количество параллельных запросов (`-P50` или больше)
- Увеличить общий объём (`seq 1 1000`)
- Запустить тест несколько раз подряд

Race condition — **недетерминированная** ошибка. Иногда она проявляется, иногда нет.

## Уровень 4: про объявление мьютекса

```go
type TaskList struct {
    Tasks  []Task `json:"tasks"`
    NextID int    `json:"nextID"`
    mu     sync.Mutex
}
```

Несколько важных моментов:

- **`mu`** — короткое имя по соглашению Go. Можно `mutex`, но `mu` встретишь чаще.
- **С маленькой буквы** — приватное. Снаружи структуры не нужно.
- **Без JSON-тега** — мьютекс не сериализуется (`encoding/json` пропускает поля без значимых данных и поля, которые не публичные).
- **Нулевое значение `sync.Mutex` готово к использованию.** Тебе не нужен `make` или явная инициализация.

## Уровень 5: про шаблон Lock + defer Unlock

В каждом методе:

```go
func (tl *TaskList) Add(title string) Task {
    tl.mu.Lock()
    defer tl.mu.Unlock()

    // основная логика
}
```

`defer tl.mu.Unlock()` — это **отложенный вызов**, который выполнится при выходе из функции, **независимо от того, как функция завершилась** (return, panic, что угодно).

Без `defer` ты должен был бы делать `Unlock` перед каждым `return`, и легко забыть. С `defer` — забыть невозможно, пока ты не забыл сам `defer`.

## Уровень 6: про возврат копии из All

Это **самая важная тонкость модуля.**

Если ты напишешь:

```go
func (tl *TaskList) All() []Task {
    tl.mu.Lock()
    defer tl.mu.Unlock()
    return tl.Tasks  // ОПАСНО
}
```

— то после возврата вызывающий код получает **тот же слайс**, что защищён мьютексом. Но мьютекс он не держит. И если в этот момент кто-то делает `Add` (который меняет `tl.Tasks` через `append`), у нас гонка на слайсе **снаружи** функции.

Решение — копия:

```go
func (tl *TaskList) All() []Task {
    tl.mu.Lock()
    defer tl.mu.Unlock()

    result := make([]Task, len(tl.Tasks))
    copy(result, tl.Tasks)
    return result
}
```

`copy` — это builtin Go, копирует элементы из одного слайса в другой. Возвращает количество скопированных элементов.

`Task` — структура без указателей, поэтому копия слайса создаёт **полные копии** каждой задачи. Изменение копии снаружи не повлияет на оригинал.

## Уровень 7: про deadlock и findIndex

У тебя был приватный метод:

```go
func (tl *TaskList) findIndex(id int) int {
    for i, t := range tl.Tasks {
        if t.ID == id {
            return i
        }
    }
    return -1
}
```

Который вызывался из `MarkDone`, `Remove`, `GetByID`.

Если ты залочишь `Remove` и оставишь `findIndex` без блокировки — формально работать будет, но это **очень опасный паттерн**: легко забыть, что вызывающий должен держать блокировку.

Если попытаешься залочить и `Remove`, и `findIndex` — получишь **deadlock** (так как Mutex нереентерабельный).

**Лучшее решение для этого модуля: убрать `findIndex`** и встроить поиск напрямую в каждый метод:

```go
func (tl *TaskList) Update(id int, title string, done bool) (Task, error) {
    tl.mu.Lock()
    defer tl.mu.Unlock()

    for i := range tl.Tasks {
        if tl.Tasks[i].ID == id {
            tl.Tasks[i].Title = title
            tl.Tasks[i].Done = done
            return tl.Tasks[i], nil
        }
    }
    return Task{}, fmt.Errorf("задача не найдена: %d", id)
}
```

Дублирование? Да, маленькое. Безопасность? Большая.

**Альтернатива (более продвинутая)** — оставить `findIndex` без блокировки, но переименовать в `findIndexUnlocked` и добавить комментарий:

```go
// findIndexUnlocked — внутренний метод. Вызывающий должен держать tl.mu.
func (tl *TaskList) findIndexUnlocked(id int) int {
    for i, t := range tl.Tasks {
        if t.ID == id {
            return i
        }
    }
    return -1
}
```

И вызывать его из `Remove` после захвата мьютекса. Это сохраняет helper, но требует дисциплины. Для модуля — выбери первый вариант (без helper-а), он проще.

## Уровень 8: про то, почему мьютекс находится **в** TaskList, а не **снаружи**

Альтернативный дизайн — держать мьютекс в обработчике или в структуре `Server`:

```go
type Server struct {
    tasks *TaskList
    mu    sync.Mutex
}

func (s *Server) createTask(w http.ResponseWriter, r *http.Request) {
    s.mu.Lock()
    defer s.mu.Unlock()
    // ...
}
```

Это **тоже работает**, но это **хуже**, потому что:

1. **Защита привязана к одной точке использования.** Если завтра ты добавишь второй обработчик или будешь использовать `TaskList` из другого места — ты должен снова не забыть про мьютекс.
2. **Защита в неправильном слое.** `TaskList` — это **данные**. Защита данных должна быть **рядом с данными**, а не в их потребителе.

**Правило:** мьютекс защищает **состояние** структуры, поэтому он должен быть **в** этой структуре. Это называется «инкапсуляция синхронизации».

## Уровень 9: про сообщение race detector

Когда ты увидишь WARNING DATA RACE — прочитай его внимательно. Структура:

```
==================
WARNING: DATA RACE
Read at 0x... by goroutine N:    ← кто читает и где
  main.(*TaskList).Add()
      .../tasklist.go:LINE
  main.Server.createTask()
      .../server.go:LINE
...

Previous write at 0x... by goroutine M:  ← кто писал до этого
  main.(*TaskList).Add()
      .../tasklist.go:LINE
...
==================
```

Адрес `0x...` — это место в памяти, где гонка. Обычно это поле структуры. Из стека ты видишь **где** в коде это происходит. По стеку можно понять, какие именно методы проблемные.

После защиты мьютексом — этих сообщений не должно быть **вообще**.

## Если совсем тупик

Скажи Claude конкретно: «модуль 5 проекта 4, [конкретная проблема]». Самые частые ситуации:

- **«Не могу воспроизвести race»** → запусти больше параллельных запросов, попробуй разные эндпоинты одновременно
- **«После защиты — программа зависает»** → ищи deadlock: вложенные `Lock`, забытый `Unlock`, метод A вызывает метод B оба с Lock
- **«Race detector ругается даже после защиты»** → проверь, что `All` возвращает копию, не оригинал; и что **все** методы защищены, не только `Add`
- **«Программа стала очень медленной»** → это нормально с `-race`. Без `-race` она снова быстрая.
