# Go Roadmap для новичка с нуля

**Проектный подход с Claude в роли наставника**

> Путь от «никогда не программировал» до готовности к собеседованиям на джуна через 7 проектов нарастающей сложности. Каждый проект — и цель, и полигон для теории, которая подтягивается ровно в момент, когда упирается в неё практика.
>
> **Оценка времени:** ~7–8 месяцев на 7 проектов при 1.5 часа в день регулярно. Плюс 4–6 месяцев на продвинутый этап = **12–14 месяцев до готовности к собеседованиям на джуна**.

---

## Почему проектный подход

Классическая схема «сначала синтаксис → потом проекты» имеет фундаментальную проблему: мотивация умирает раньше, чем появляется первый результат. Новичок два месяца пишет функции-заглушки, не видит смысла, и бросает.

Проектный подход решает сразу несколько вещей:

1. **Контекст для теории.** Когда ты изучаешь слайсы, потому что тебе нужно хранить список задач — ты их запоминаешь. Когда «для галочки» — забываешь за неделю.
2. **Дофамин от работающей штуки.** Даже уродливый калькулятор, который реально считает — это удар мотивации.
3. **Claude как наставник в реальном времени.** Это ключевой элемент, которого не было в классических роадмапах — потому что их писали до появления LLM.
4. **Портфолио растёт с первого месяца.** К концу пути у тебя 7 публичных репозиториев на GitHub с осмысленным README — реальный материал для резюме.

---

## ⚠️ Правила контракта с Claude

Без этих правил связка «новичок + ИИ» превращается в деградацию. С ними — в ускорение. **Скопируй этот блок в user preferences Claude перед началом обучения.**

1. **Claude никогда не пишет код за меня.** Если я прошу «напиши функцию X», Claude отказывается и возвращает наводящие вопросы: «какой у функции вход? что она должна вернуть? как ты проверишь, что она работает?».
2. **Сначала я, потом проверка.** Я пишу код сам → показываю Claude → он говорит, что сильно и что улучшить. Не наоборот.
3. **Объяснение до решения.** Если я застрял, Claude сначала объясняет концепцию, которой мне не хватает, и только если я всё ещё не могу — даёт подсказку-направление, НЕ код.
4. **Формат подсказок: вопрос, а не ответ.** «Что произойдёт, если слайс пустой?» вместо «добавь проверку `if len(s) == 0`».
5. **Право на ошибку.** Если я написал кривой, но рабочий код — Claude не переписывает его «красиво», а показывает **одну** вещь, которую стоит улучшить, и объясняет почему.
6. **Разбор ошибок, а не исправление.** Когда у меня compile error — Claude объясняет, что означает сообщение компилятора, а не даёт готовый фикс.
7. **Проверка понимания.** После каждой новой темы Claude задаёт 2–3 вопроса на понимание, и я отвечаю своими словами.

## Проект 0. «Hello, я живой» — настройка окружения

**Длительность:** 3–5 дней

**Что делаю:** ставлю Go, VSCode с Go-плагином, git, создаю репозиторий на GitHub, пишу программу из двух строк `fmt.Println("Hello")`, коммичу, пушу.

**Что узнаю параллельно:**
- Терминал: `cd`, `ls`, `mkdir`, `cat`
- Git базовый: `init`, `add`, `commit`, `push`
- Что такое `go run` и `go build`

**Роль Claude:** объясняет, что такое терминал, зачем git, что происходит при пуше. Помогает дебажить ошибки установки — их будет много, и это нормально.

**Маркер готовности:** на GitHub лежит публичный репозиторий с README и `main.go`. Могу объяснить словами, что произошло, когда нажал Enter после `go run`.

## Проект 1. Калькулятор в терминале

**Длительность:** 2 недели

**Что делает:** программа принимает два числа и операцию, выводит результат.

**Итерации:**
- **v1** — жёстко захардкоженные числа в коде
- **v2** — читает числа через `fmt.Scan`
- **v3** — принимает аргументы командной строки (`os.Args`)
- **v4** — обрабатывает ошибки (деление на ноль, нечисловой ввод)
- **v5** — поддерживает больше операций через `switch`

**Теория, которая подтягивается:**
- Переменные и типы (`int`, `float64`, `string`)
- Базовый ввод-вывод
- Условия и `switch`
- Функции с множественным возвратом (`result, err`)
- **Идиоматичная обработка ошибок в Go** — первое прикосновение к тому, что отличает Go от других языков

**Роль Claude:** задаёт вопросы типа «что должно произойти, если пользователь ввёл букву вместо числа?», объясняет концепцию `error` как значения, а не исключения.

**Маркер готовности:** программа не падает ни на каком вводе, все ошибки обработаны явно.

## Проект 2. To-Do лист в памяти

**Длительность:** 3 недели

**Что делает:** консольное приложение. Команды: `add <текст>`, `list`, `done <номер>`, `remove <номер>`. Данные живут только пока программа запущена.

**Теория, которая подтягивается:**
- **Слайсы** — первая «непростая» концепция. Underlying array, длина vs ёмкость, поведение `append` при росте. Новичок сначала использует их «как массивы», потом Claude объясняет, что внутри.
- **Структуры** — `type Task struct { ID int; Title string; Done bool }`
- **Методы на структурах**
- **Бесконечный цикл с командами** — паттерн REPL

**Роль Claude:** после того как написал первую рабочую версию, Claude говорит: «Твой код работает, и это главное. Теперь давай посмотрим на одну вещь: вот здесь ты делаешь append в цикле. Как думаешь, что происходит с памятью?» — и ведёт к пониманию слайсов изнутри.

**Маркер готовности:** умею объяснить словами, чем слайс отличается от массива, и почему `append` иногда возвращает новый слайс.

## Проект 3. To-Do лист с сохранением в файл

**Длительность:** 2 недели

**Что делает:** то же самое, но задачи сохраняются в JSON-файл и загружаются при старте.

**Теория:**
- Работа с файлами (`os.ReadFile`, `os.WriteFile`)
- **JSON в Go** — `encoding/json`, тэги у полей структуры (`json:"title"`)
- Разделение кода на файлы и пакеты — первое знакомство с модульностью
- Что такое `go.mod` и зачем

**Роль Claude:** здесь важный момент — впервые столкнёшься с идеей, что код надо раскладывать по файлам. Claude не говорит «сделай так», а спрашивает: «когда `main.go` превысил 200 строк, что тебе стало труднее?» и подводит к идее разделения.

**Маркер готовности:** код разложен минимум на 2 файла (например, `main.go` и `storage.go`), и могу объяснить, почему именно так.

## Проект 4. Первый HTTP-сервер — Todo API

**Длительность:** 3–4 недели

> **Это переломный проект.** Здесь впервые пишешь то, что называется «бэкендом».

**Что делает:** REST API с эндпоинтами:

| Метод | Путь | Действие |
|-------|------|----------|
| `GET` | `/tasks` | список |
| `POST` | `/tasks` | создать |
| `PUT` | `/tasks/{id}` | обновить |
| `DELETE` | `/tasks/{id}` | удалить |

Данные пока в памяти + сохранение в JSON при завершении. Только `net/http` из стандартной библиотеки, **никаких фреймворков**.

**Теория:**
- HTTP на уровне «что такое запрос, ответ, метод, статус, заголовок, тело»
- JSON как формат обмена
- `net/http`: handler, ServeMux, ResponseWriter, Request
- Тестирование API через curl и Postman
- **Первое неявное знакомство с горутинами** — узнаёшь, что каждый запрос HTTP-сервер обрабатывает в отдельной горутине, и это вызовет вопрос «а что если двое одновременно добавят задачу?»

**Роль Claude:** критически важно не дать Claude написать boilerplate. Он должен спрашивать: «что такое endpoint? что должно быть в теле POST-запроса? какой статус вернуть, если задача не найдена?».

> **Race condition как подарок.** Когда обнаружишь, что при одновременных запросах данные портятся — это **золотой момент для объяснения мьютексов**. Не до, а именно когда сам увидел проблему.

**Маркер готовности:** API работает, можно дёрнуть из Postman, умею объяснить, что такое race condition и как его решил.

## Проект 5. Тот же Todo, но с PostgreSQL

**Длительность:** 3–4 недели

**Что делает:** переезд с in-memory хранилища на настоящую БД.

**Теория:**
- SQL базовый (`SELECT`, `INSERT`, `UPDATE`, `DELETE`, `WHERE`)
- Установка PostgreSQL через Docker — первое знакомство с Docker как «команда, которая поднимает БД»
- DBeaver как GUI для просмотра данных
- `jackc/pgx` — драйвер для Go
- Параметризованные запросы и **почему нельзя конкатенировать SQL** (SQL-инъекции)
- Что такое индекс на уровне «оглавление книги»
- Разделение кода: `handler` работает с HTTP, `storage` — с БД, они не знают друг о друге

**Роль Claude:** подводит к пониманию, почему переезд на БД — это не просто «заменить одно на другое». Задаёт вопросы: «что происходит с твоими данными, когда перезапускаешь программу сейчас? а в будущем?», «почему плохо склеивать SQL через `fmt.Sprintf`?».

> **Важный момент:** здесь впервые возникает понятие **архитектуры** — разделение на слои. Не через теорию «чистой архитектуры», а через боль «у меня всё свалено в один файл, и я не могу разобраться».

**Маркер готовности:** API работает с Postgres, данные переживают рестарт, код разложен на слои, умею написать базовый SELECT с WHERE и JOIN.

## Проект 6. Телеграм-бот «Погода»

**Длительность:** 3 недели

**Что делает:** бот принимает город или геолокацию, возвращает погоду из публичного API.

**Теория:**
- HTTP-клиент (обратная сторона HTTP — не только принимать, но и отправлять)
- Работа с внешними API, API-ключи, переменные окружения
- Библиотека для Telegram Bot API
- Долгоживущий процесс, который слушает события
- Обработка паник и graceful shutdown (Ctrl+C должен корректно завершать работу)

**Зачем этот проект после REST API?** Он даёт **разную перспективу**: если Todo-API учит принимать запросы, то бот учит инициировать взаимодействия. Это две ортогональные модели, и обе нужны в реальной работе.

**Роль Claude:** объясняет, что такое long-polling, чем отличается от webhook (на уровне концепции), помогает разобраться с переменными окружения и почему API-ключ нельзя хардкодить.

**Маркер готовности:** бот задеплоен хотя бы локально, работает 24 часа без падения, умею объяснить, что такое переменная окружения и почему секреты не хранят в коде.

## Проект 7. Блог-платформа с авторизацией

**Длительность:** 6–8 недель

> **Финальный проект джуна.** Здесь всё собирается вместе. Урезанная версия хабра-клона — без Kafka и микросервисов, но со всем остальным.

**Что делает:**
- Регистрация и логин (хэширование паролей через `bcrypt`)
- JWT-токены и middleware аутентификации
- CRUD для статей (с авторизацией — редактировать только свои)
- Комментарии к статьям
- Простейший поиск по заголовкам (через `LIKE` в SQL)
- Redis для кэша списка последних статей
- Docker Compose с тремя сервисами: приложение, Postgres, Redis
- Базовые тесты на хэндлеры
- Линтер `golangci-lint` настроен и проходит без ошибок
- README с инструкцией `docker-compose up`

**Теория, которая здесь подтягивается:**
- Middleware как паттерн
- JWT изнутри (три части, подпись, проверка)
- Что такое Redis и зачем кэш — не «потому что круто», а «потому что вот этот запрос медленный»
- Docker и Docker Compose на уровне пользователя
- Как писать тесты в Go (`testing` пакет, table-driven tests)
- Структура проекта по `golang-standards/project-layout`

**Роль Claude:** здесь Claude уже ведёт себя скорее как ревьюер, чем как учитель. Проверяет архитектурные решения, задаёт вопросы типа «что произойдёт, если Redis упадёт?», «как ты проверишь, что пароль действительно хэшируется?». Помогает увидеть дыры, но не закрывает их за тебя.

**Маркер готовности:** проект развёртывается одной командой на чужом компьютере, код проходит линтер, есть 5+ тестов, README понятен незнакомому человеку.

## Мета-правила, которые спасут от главных ловушек

1. **Темп и регулярность.** 1–2 часа в день стабильно сильно лучше, чем 10 часов в субботу. Навык проседает за пропуски как мышцы.
2. **Правило 70/30.** 70% времени — руки на клавиатуре, 30% — теория. У новичков часто перекос в обратную сторону — это **туториал-ад**, главная ловушка.
3. **Синдром «ещё одной книги».** Копишь ресурсы вместо кода. Лечится прямым вопросом: «что я вчера написал руками?».
4. **Боязнь compile error.** Ошибка — это диалог с компилятором, он помогает, а не ругает.
5. **Не гнать стек.** Хочется Kafka, Kubernetes и чистую архитектуру на втором месяце — жёстко тормози себя.
6. **Гуглить по-английски.** Русскоязычных ответов на StackOverflow в разы меньше — это критичный навык.
7. **Читать чужой код.** Раз в неделю заходи в репозиторий популярной Go-библиотеки (например, `chi`) и читай любой один файл. Просто читай.
