# Модуль 5. Транзакции — теория

## Контекст

В Проекте 4 ты столкнулся с **race condition** и решил её через `sync.Mutex`. Это работало, но было неудобно: тебе пришлось защищать **каждый** метод, помнить про deadlock, не забывать про копирование возвращаемых данных. И главное — это защита **внутри одного процесса**. Если запустить два экземпляра сервера с одним файлом, мьютексы разных процессов ничего не знают друг о друге.

В этом модуле ты познакомишься с **транзакциями** — фундаментальной идеей реляционных БД, которая решает все эти проблемы **на порядок мощнее**.

Транзакции — это, наверное, **самая важная концепция**, которую ты увезёшь из всего Проекта 5. Без понимания транзакций невозможно писать надёжные бэкенд-сервисы.

## Ядро 1: что такое транзакция

**Транзакция** — это **группа SQL-операций**, которые БД выполняет как **одну неделимую единицу**. Либо все операции прошли успешно, либо ни одна не применилась. Никаких «полуприменённых» состояний.

Канонический пример — перевод денег между счетами:

```sql
-- Без транзакции
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
-- ... в этот момент сервер упал ...
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
```

Что произошло? **С первого счёта 100 списали, на второй — нет.** Деньги исчезли. Это катастрофа.

С транзакцией:

```sql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;
```

`BEGIN` начинает транзакцию. `COMMIT` фиксирует все изменения. **Между `BEGIN` и `COMMIT` все изменения «видны» только внутри транзакции** — другие соединения не видят их, пока не пройдёт `COMMIT`.

Если что-то пошло не так до `COMMIT`:

```sql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
-- упс, ошибка
ROLLBACK;
```

`ROLLBACK` отменяет **все** изменения внутри транзакции. БД возвращается в то состояние, которое было до `BEGIN`. Деньги на месте.

Это — **атомарность**: «либо всё, либо ничего».

## Ядро 2: ACID

Транзакции в реляционных БД дают четыре свойства, которые называют **ACID**:

| Буква | Свойство | Что значит |
|-------|----------|------------|
| **A** | Atomicity (атомарность) | Либо все операции применились, либо ни одна |
| **C** | Consistency (согласованность) | После транзакции БД остаётся в валидном состоянии (constraints, foreign keys) |
| **I** | Isolation (изолированность) | Параллельные транзакции не «видят» друг друга до commit |
| **D** | Durability (долговечность) | После commit данные точно на диске и переживут падение сервера |

**Atomicity** — это про «всё или ничего» из примера выше.

**Consistency** — про то, что если у тебя есть constraint `CHECK (balance >= 0)`, то ни одна транзакция не сможет его нарушить. БД проверяет.

**Isolation** — про то, что если две транзакции работают одновременно, они **изолированы** друг от друга. Каждая видит данные так, как будто она единственная. Подробнее ниже.

**Durability** — про то, что после `COMMIT` данные **точно** на диске. Даже если через секунду сервер упадёт — после перезагрузки данные на месте.

В Проекте 4 у тебя была примитивная имитация атомарности через мьютекс. Но `D` ты не получал: между `Add` и `Save` могло пройти время, и если в этот момент сервер упал — данные в файле были устаревшими. БД решает это правильно.

## Ядро 3: транзакции в database/sql

В Go транзакция — это объект `*sql.Tx`. Получаешь его через `db.Begin()`:

```go
tx, err := db.Begin()
if err != nil {
    return fmt.Errorf("begin: %w", err)
}
defer tx.Rollback()  // ← КРАЙНЕ ВАЖНО

// Делаем несколько операций через tx, не db:
_, err = tx.Exec("UPDATE accounts SET balance = balance - 100 WHERE id = 1")
if err != nil {
    return fmt.Errorf("debit: %w", err)
}

_, err = tx.Exec("UPDATE accounts SET balance = balance + 100 WHERE id = 2")
if err != nil {
    return fmt.Errorf("credit: %w", err)
}

// Если дошли сюда — всё хорошо, фиксируем
if err := tx.Commit(); err != nil {
    return fmt.Errorf("commit: %w", err)
}
return nil
```

Несколько важных моментов:

1. **`db.Begin()`** возвращает `*sql.Tx` — объект транзакции. Он реализует те же методы, что `*sql.DB` (`Query`, `QueryRow`, `Exec`), но всё внутри одной транзакции.

2. **`defer tx.Rollback()`** — **обязательная** строка сразу после `Begin`. Это страховка: если функция выйдет с ошибкой до `Commit` (через `return`, `panic`, что угодно) — `Rollback` отменит транзакцию.

3. **Вызовы через `tx`, а не `db`.** Если ты случайно сделаешь `db.Exec(...)` внутри функции, которая работает с транзакцией — этот запрос пройдёт **вне** транзакции, в новом соединении. Это серьёзный баг.

4. **`tx.Commit()`** в самом конце, **только** если всё прошло хорошо.

## Ядро 4: паттерн defer tx.Rollback()

Это — самый важный паттерн в работе с транзакциями. Разберём, **почему** `defer tx.Rollback()` сразу после `Begin` работает.

```go
tx, err := db.Begin()
if err != nil {
    return err
}
defer tx.Rollback()

// ... много операций ...

if err := tx.Commit(); err != nil {
    return err
}
return nil
```

Что происходит:

- **Случай 1**: всё хорошо. `Commit()` срабатывает. Когда функция возвращается, `defer tx.Rollback()` вызывает `Rollback`, **но это no-op** — после `Commit` транзакция уже закрыта, и `Rollback` молча возвращает ошибку «уже закрыта», которую мы игнорируем.

- **Случай 2**: что-то пошло не так до `Commit`. Функция возвращает ошибку через `return err`. `defer tx.Rollback()` срабатывает и отменяет транзакцию. Данные не изменились.

- **Случай 3**: функция упала с panic. `defer tx.Rollback()` всё равно вызывается (это особенность `defer`) и отменяет транзакцию.

**Без `defer tx.Rollback()`:** если ты забудешь явно вызвать `Rollback` в одной из веток ошибок, транзакция останется **открытой** и будет держать соединение из пула. Со временем это превращается в утечку и зависание.

**Правило:** **сразу** после `Begin` — `defer tx.Rollback()`. Никаких исключений, никаких «я вызову вручную». Только так.

## Ядро 5: пример с TaskRepository

Допустим, ты хочешь добавить **сразу две** задачи атомарно: либо обе создаются, либо ни одна. Это полезно, например, для импорта из CSV.

```go
func (r *TaskRepository) AddBatch(titles []string) ([]Task, error) {
    tx, err := r.db.Begin()
    if err != nil {
        return nil, fmt.Errorf("begin: %w", err)
    }
    defer tx.Rollback()

    var created []Task
    for _, title := range titles {
        var t Task
        err := tx.QueryRow(
            "INSERT INTO tasks (title) VALUES ($1) RETURNING id, title, done",
            title,
        ).Scan(&t.ID, &t.Title, &t.Done)
        if err != nil {
            return nil, fmt.Errorf("insert %q: %w", title, err)
        }
        created = append(created, t)
    }

    if err := tx.Commit(); err != nil {
        return nil, fmt.Errorf("commit: %w", err)
    }

    return created, nil
}
```

Если один из INSERT упал (например, из-за CHECK constraint) — `defer tx.Rollback()` отменит **все** уже сделанные INSERTs. Никаких частично созданных данных.

Заметь: внутри цикла **`tx.QueryRow`**, не `r.db.QueryRow`. Это важно: все запросы должны идти через **один и тот же** `tx`.

## Ядро 6: уровни изоляции

Когда несколько транзакций работают **параллельно**, возникает вопрос: что одна транзакция «видит» от другой?

PostgreSQL поддерживает четыре уровня изоляции (от слабого к строгому):

| Уровень | Что предотвращается |
|---------|----------------------|
| Read Uncommitted | (не реализован в PostgreSQL, на самом деле = Read Committed) |
| **Read Committed** | Грязные чтения. Это **default** в PostgreSQL. |
| Repeatable Read | + Неповторяющиеся чтения |
| Serializable | + Фантомы. Полная сериализация. |

Каждый уровень более строгий и более медленный.

**Для большинства задач PostgreSQL-default `Read Committed` достаточен.** Это означает: транзакция видит только данные, которые были **закоммичены** до её начала или коммита другой транзакции. «Грязные» (незакоммиченные) данные — никогда.

Когда нужны более строгие уровни:

- **Repeatable Read** — если внутри транзакции ты делаешь несколько SELECT-ов и хочешь, чтобы они **видели одинаковые данные**.
- **Serializable** — если у тебя сложная логика, которая нарушится, если две параллельные транзакции одновременно изменят одни данные. Например, проверка «есть ли свободное место + бронирование».

**Для нашего проекта — `Read Committed` (default), и ничего настраивать не нужно.** Знание про другие уровни — для общего развития.

Если когда-нибудь понадобится:

```go
tx, err := db.BeginTx(ctx, &sql.TxOptions{
    Isolation: sql.LevelSerializable,
})
```

## Ядро 7: транзакция как способ решения race condition

Помнишь race condition в Проекте 4 при `Add`? Когда две горутины читали `nextID`, обе получали 5, и обе вставляли с ID=5?

С PostgreSQL **этой проблемы нет**, даже **без явных транзакций**. Почему?

- **`BIGSERIAL` под капотом** — это **sequence**, которое атомарно увеличивается при каждом `nextval()`. Невозможно, чтобы две INSERT-операции получили один и тот же id.
- **`PRIMARY KEY`** гарантирует уникальность. Даже если ты вручную попытаешься вставить дубликат — БД откажется.
- **Каждый INSERT — это автокоммит** (если не в транзакции). PostgreSQL обрабатывает их последовательно, без гонок.

Это и есть прелесть БД: **проблемы, которые ты решал ручным мьютексом, она решает встроенно и правильно**.

## Ядро 8: когда транзакция нужна, а когда нет

**Не нужна (один запрос — одна операция):**

- `INSERT INTO tasks (title) VALUES ($1)` — atomic сам по себе
- `UPDATE tasks SET done = true WHERE id = $1` — atomic
- `DELETE FROM tasks WHERE id = $1` — atomic
- `SELECT ... FROM tasks` — чтение, атомарность не релевантна

**Нужна (несколько запросов, которые должны быть атомарны):**

- Перевод денег между счетами (UPDATE одного, UPDATE другого)
- Создание заказа + резерв товара на складе (INSERT в orders, UPDATE inventory)
- Импорт батча данных, который должен пройти целиком
- Сложные UPDATE на основе предварительного SELECT (например, «увеличить балансы всех пользователей с balance < 100 на 50»)

**Для нашего проекта 5 транзакции практически не нужны** — все операции элементарные. Но **знать и уметь** их обязательно. Поэтому в condition.md задание — реализовать `AddBatch` как упражнение, и `MoveTaskToStart` (UPDATE + перенумерация) как более сложный случай.

## Ядро 9: типичные ошибки

**Забыл `defer tx.Rollback()`.**
Если в одной из веток ошибки забыл `Rollback` явно, транзакция повисает и держит соединение.

**Использовал `db` вместо `tx` внутри транзакции.**
Запрос пройдёт **вне** транзакции, в другом соединении. Атомарности не будет.

**Долгая транзакция.**
Транзакция держит **блокировки** на затронутых строках. Чем дольше открыта — тем больше блокировок, тем хуже параллелизм. **Транзакция должна быть короткой**: открыл → сделал несколько операций → закоммитил. Не делай в транзакции HTTP-запросы, чтение файлов или ожидание.

**Транзакция через несколько HTTP-запросов.**
Иногда хочется «открыть транзакцию в одном запросе и закоммитить в другом». **Так делать нельзя** — транзакция привязана к соединению, а соединение возвращается в пул после каждой операции. Транзакция должна жить **внутри одного обработчика**.

## Полный пример с обработкой ошибок

```go
func (r *TaskRepository) AddBatch(titles []string) ([]Task, error) {
    if len(titles) == 0 {
        return nil, nil
    }

    tx, err := r.db.Begin()
    if err != nil {
        return nil, fmt.Errorf("begin transaction: %w", err)
    }
    defer tx.Rollback()  // ← страховка

    created := make([]Task, 0, len(titles))
    for _, title := range titles {
        var t Task
        err := tx.QueryRow(
            "INSERT INTO tasks (title) VALUES ($1) RETURNING id, title, done",
            title,
        ).Scan(&t.ID, &t.Title, &t.Done)
        if err != nil {
            return nil, fmt.Errorf("insert task %q: %w", title, err)
            // defer Rollback сработает автоматически
        }
        created = append(created, t)
    }

    if err := tx.Commit(); err != nil {
        return nil, fmt.Errorf("commit: %w", err)
        // даже если Commit упал — defer Rollback сработает
        // Rollback после Commit — no-op
    }

    return created, nil
}
```
