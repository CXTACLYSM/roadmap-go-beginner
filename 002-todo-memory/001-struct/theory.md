# Модуль 1. Структура Task — теория

## Контекст

В Проекте 1 ты работал с **простыми типами**: `int`, `float64`, `string`. Каждая переменная содержала **одно значение**.

Но в реальной жизни многие сущности — это **наборы связанных значений**. Задача в to-do листе — это не одна штука, а несколько: уникальный номер, текст, статус «выполнена или нет». Эти три поля логически связаны: они описывают **одну задачу**.

Можно было бы держать три разных переменных:

```go
var taskID int
var taskTitle string
var taskDone bool
```

Но как только задач станет больше одной — этот подход развалится. Тебе нужна способ **сгруппировать связанные данные** в один объект. В Go для этого есть **структуры**.

## Ядро: что такое структура

**Структура** (`struct`) — это собственный тип данных, который ты создаёшь сам. Структура объединяет несколько **полей** разных типов под одним именем.

```go
type Task struct {
    ID    int
    Title string
    Done  bool
}
```

Разберём по частям:

- `type` — ключевое слово «я объявляю новый тип»
- `Task` — имя нового типа. По соглашению Go — с большой буквы, если тип нужно использовать снаружи пакета (об этом — ниже)
- `struct { ... }` — «этот тип — структура с такими полями»
- Внутри — поля: имя, потом тип. Каждое поле на своей строке (можно через запятую, но так не пишут)

После этого `Task` становится полноценным типом — таким же, как `int` или `string`.

## Создание экземпляра структуры

После того как тип объявлен, ты можешь создавать **экземпляры** — конкретные значения этого типа.

Способ 1: с явным указанием полей (предпочтительный)

```go
t := Task{
    ID:    1,
    Title: "Купить молоко",
    Done:  false,
}
```

Способ 2: позиционный (короткий, но хрупкий)

```go
t := Task{1, "Купить молоко", false}
```

Второй способ требует, чтобы значения шли **в том же порядке**, что и поля в объявлении типа. Если ты потом поменяешь порядок полей в `Task` — все позиционные конструкторы сломаются. Поэтому в реальном коде используют **только первый способ**. Второй — для очень маленьких структур из 1–2 полей.

Способ 3: пустая структура

```go
var t Task
```

Все поля получат **нулевые значения**: `0` для `int`, `""` для `string`, `false` для `bool`. Это полезно, когда ты собираешься заполнять структуру по частям.

## Доступ к полям

К полю обращаются через точку:

```go
t := Task{ID: 1, Title: "Купить молоко", Done: false}

fmt.Println(t.ID)     // 1
fmt.Println(t.Title)  // Купить молоко
fmt.Println(t.Done)   // false

t.Done = true         // изменили поле
fmt.Println(t.Done)   // true
```

Точка — универсальный оператор «дай мне поле». Работает и для чтения, и для записи.

## Почему так — а не классы как в других языках

Если ты слышал про объектно-ориентированное программирование (ООП), у тебя может возникнуть вопрос: «а где классы? где наследование? где конструкторы?».

В Go этого **нет**, и это сознательное решение. Авторы языка посчитали, что классическое ООП с наследованием создаёт больше проблем, чем решает. Вместо этого Go предлагает:

- **Структуры** для группировки данных
- **Методы** для поведения (см. ниже)
- **Интерфейсы** для абстракции (увидишь позже)
- **Композицию** (встраивание) вместо наследования

Это другая философия, и поначалу она кажется «недоделанной». Через пару проектов ты поймёшь, что она проще, явнее и предсказуемее. Главное — не пытайся писать на Go так, как ты бы писал на Java или Python. Это другой язык с другим устройством.

## Методы — функции, привязанные к типу

Метод — это функция, которая **принадлежит** какому-то типу. Синтаксис почти как у обычной функции, но с одной добавкой:

```go
func (t Task) String() string {
    status := "[ ]"
    if t.Done {
        status = "[x]"
    }
    return fmt.Sprintf("%d. %s %s", t.ID, status, t.Title)
}
```

Новая часть — `(t Task)` между `func` и именем. Это **получатель** (receiver) — он говорит: «эта функция — метод типа `Task`, и внутри тип будет доступен под именем `t`».

Использовать так:

```go
task := Task{ID: 1, Title: "Купить молоко", Done: false}
fmt.Println(task.String())  // "1. [ ] Купить молоко"
```

Метод вызывается через точку, как доступ к полю — но со скобками.

## Receiver: значение или указатель?

У receiver-а есть две формы:

```go
// 1. Значение (копия)
func (t Task) String() string { ... }

// 2. Указатель
func (t *Task) MarkDone() { t.Done = true }
```

**Главное правило:**

- Если метод **только читает** данные структуры — используй **значение** (`t Task`)
- Если метод **изменяет** данные структуры — используй **указатель** (`t *Task`)

Почему? Потому что receiver-значение — это **копия**. Если ты в методе с receiver-значением напишешь `t.Done = true`, ты изменишь **копию**, а оригинал останется как был. Это самый частый «тихий баг» новичка в Go.

```go
// НЕ работает — изменения теряются
func (t Task) MarkDone() {
    t.Done = true  // меняет копию
}

// Работает — изменения видны снаружи
func (t *Task) MarkDone() {
    t.Done = true  // меняет оригинал по указателю
}
```

Как использовать оба варианта — одинаково:

```go
task := Task{ID: 1, Title: "Купить молоко"}
task.MarkDone()  // Go сам сообразит, нужен ли указатель
fmt.Println(task.Done)  // true
```

Go автоматически берёт адрес `task`, если метод требует указатель. Это работает только для **переменных** — для значений, лежащих внутри других значений (например, элементов слайса), правила сложнее. Но об этом — в модулях 2 и 4.

## Полный пример

```go
package main

import "fmt"

type Task struct {
    ID    int
    Title string
    Done  bool
}

// Метод-чтение: receiver-значение
func (t Task) String() string {
    status := "[ ]"
    if t.Done {
        status = "[x]"
    }
    return fmt.Sprintf("%d. %s %s", t.ID, status, t.Title)
}

// Метод-изменение: receiver-указатель
func (t *Task) MarkDone() {
    t.Done = true
}

func main() {
    task := Task{
        ID:    1,
        Title: "Купить молоко",
        Done:  false,
    }

    fmt.Println(task.String())  // 1. [ ] Купить молоко

    task.MarkDone()
    fmt.Println(task.String())  // 1. [x] Купить молоко
}
```

## Большая буква в имени поля

Ты заметил, что поля называются `ID`, `Title`, `Done` — с большой буквы? Это **не стиль**, это **семантика**.

В Go видимость определяется регистром первой буквы:

- **С большой буквы** — публичное (видно снаружи пакета)
- **С маленькой буквы** — приватное (видно только внутри пакета)

Пока ты пишешь всё в одном пакете `main`, разницы не видно. Но когда начнёшь раскладывать код по пакетам (Проект 3), это станет важно. Привыкай сразу: **поля и методы, которые могут использоваться извне — с большой буквы**.
