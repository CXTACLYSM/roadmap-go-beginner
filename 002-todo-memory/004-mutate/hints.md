## Уровень 2: про strconv.Atoi

```go
import "strconv"

id, err := strconv.Atoi("42")
if err != nil {
    fmt.Println("ID должен быть числом:", arg)
    continue
}
// здесь id уже корректный int
```

`Atoi` (ASCII to integer) возвращает два значения: число и ошибку. Это твой старый друг — паттерн `result, err := ...` из проекта 1.

## Уровень 3: про функцию поиска

Самая простая форма:

```go
func findIndex(tasks []Task, id int) int {
    for i, t := range tasks {
        if t.ID == id {
            return i
        }
    }
    return -1
}
```

Несколько моментов:

- `range` здесь безопасен — мы только **читаем** `t.ID`, не меняем `t`. Range можно использовать для чтения сколько угодно.
- Возвращаем **индекс** (`i`), а не саму задачу (`t`). Через индекс можно потом и менять, и удалять.
- `-1` — сигнальное значение «не нашёл».

## Уровень 4: про изменение через индекс

Когда у тебя есть индекс `i`, изменение поля делается так:

```go
tasks[i].Done = true
```

Это **не копия**, это прямой доступ к элементу. После этой строки `Done` действительно изменился в слайсе.

Если бы ты написал:

```go
t := tasks[i]      // вот тут произошла копия
t.Done = true      // меняем копию
```

— это **не сработало бы**. Та же ловушка, что и с range.

## Уровень 5: про удаление через append

Идиома удаления из слайса:

```go
tasks = append(tasks[:i], tasks[i+1:]...)
```

Разберём:

- `tasks[:i]` — всё **до** удаляемого элемента (без него)
- `tasks[i+1:]` — всё **после** удаляемого (без него)
- `append(..., ...)` — склеиваем эти два куска
- `...` после второго аргумента — «распакуй слайс в отдельные элементы»
- `tasks =` — присваиваем результат обратно (как всегда с append)

Без `...` ты получишь ошибку компилятора:

```
cannot use tasks[i+1:] (variable of type []Task) as type Task in argument to append
```

Расшифровка: «append ждёт элементы типа `Task`, а ты передал слайс `[]Task` целиком». Многоточие говорит компилятору: «распакуй».

## Уровень 6: про границы при удалении

Что если удаляешь **первый** элемент (`i = 0`)?

- `tasks[:0]` → пустой слайс
- `tasks[1:]` → всё, кроме первого

Результат: всё кроме первого. Корректно.

Что если удаляешь **последний** (`i = len(tasks)-1`)?

- `tasks[:len-1]` → всё, кроме последнего
- `tasks[len:]` → пустой слайс (это валидно — «срез от конца до конца»)

Результат: всё, кроме последнего. Корректно.

Слайсинг в Go устроен так, что границы `0` и `len(tasks)` — **валидные**, не паника. Это удобно как раз для таких случаев.

## Уровень 7: про дублирование

Когда напишешь обе команды (`done` и `remove`), ты увидишь, что они **очень похожи**:

```go
case "done":
    if arg == "" { /* подсказка */; continue }
    id, err := strconv.Atoi(arg)
    if err != nil { /* ошибка */; continue }
    i := findIndex(tasks, id)
    if i == -1 { /* не нашли */; continue }
    tasks[i].Done = true
    fmt.Printf("...\n")

case "remove":
    if arg == "" { /* подсказка */; continue }
    id, err := strconv.Atoi(arg)
    if err != nil { /* ошибка */; continue }
    i := findIndex(tasks, id)
    if i == -1 { /* не нашли */; continue }
    tasks = append(tasks[:i], tasks[i+1:]...)
    fmt.Printf("...\n")
```

Семь строк из десяти **одинаковые**. Это «копипаста» в чистом виде, и она просится в функцию. Но **не торопись** рефакторить прямо сейчас — этим займёмся в модуле 5. Сейчас твоя задача — сделать рабочий код, **прочувствовать** боль от дублирования, и только потом устранить её осознанно.

## Если совсем тупик

Скажи Claude: «модуль 4, не работает удаление» (или другая конкретика). Покажи код. Самые частые ситуации:
- Забыл `...` в `append` → ошибка компилятора про несоответствие типов
- Забыл `tasks =` → удаление «не происходит»
- `findIndex` ищет по позиции в слайсе, а не по `t.ID` → удаляются не те задачи
