## Уровень 2: про docker run

Полная команда:

```bash
docker run -d \
    --name todo-pg \
    -e POSTGRES_PASSWORD=secret \
    -p 5432:5432 \
    -v todo-pg-data:/var/lib/postgresql/data \
    postgres:16
```

Если ты разбиваешь на несколько строк через `\` — будь осторожен с пробелами после `\` (их быть не должно). Иначе bash подумает, что строка закончилась, и будет ошибка.

В одну строку:

```bash
docker run -d --name todo-pg -e POSTGRES_PASSWORD=secret -p 5432:5432 -v todo-pg-data:/var/lib/postgresql/data postgres:16
```

## Уровень 3: про распространённые ошибки docker run

**«Bind for 0.0.0.0:5432 failed: port is already allocated»** — у тебя уже запущен PostgreSQL на порту 5432. Варианты:
- Останови старый: `docker stop <name>` или `sudo systemctl stop postgresql`
- Используй другой порт: `-p 5433:5432` (тогда подключайся через `-p 5433`)

**«Container name "/todo-pg" is already in use»** — контейнер с таким именем уже существует. Удали:
```bash
docker rm -f todo-pg
```

**«docker: command not found»** — Docker не установлен или не в PATH. Проверь установку.

## Уровень 4: про подключение через psql

```bash
psql -h localhost -U postgres -d postgres
```

Если psql спрашивает пароль — введи `secret` (или то, что ты указал в `POSTGRES_PASSWORD`).

Если psql **не** спрашивает пароль и сразу подключается — у тебя локальный PostgreSQL или настроена авторизация без пароля. Проверь, что подключение реально к контейнеру через `\conninfo` после подключения.

Если получаешь `psql: error: connection to server at "localhost" (::1), port 5432 failed: Connection refused`:
- Контейнер не запущен — проверь `docker ps`
- Контейнер запустился, но ещё не готов — подожди 5 секунд и попробуй снова
- Порт занят другим процессом — проверь `sudo lsof -i :5432`

## Уровень 5: про SQL-синтаксис

Каждая команда заканчивается **точкой с запятой**:

```sql
CREATE TABLE tasks (id SERIAL PRIMARY KEY, title TEXT NOT NULL);
```

Если ты забудешь `;`, psql выведет `postgres-#` (вместо `postgres=#`) и будет ждать продолжения. Просто допиши `;` и нажми Enter.

Можно писать на нескольких строках:

```sql
CREATE TABLE tasks (
    id SERIAL PRIMARY KEY,
    title TEXT NOT NULL,
    done BOOLEAN NOT NULL DEFAULT false
);
```

Для psql главное — `;` в конце.

## Уровень 6: про кавычки

В SQL **строки** в **одинарных** кавычках:

```sql
INSERT INTO tasks (title) VALUES ('Купить молоко');
```

Если используешь двойные:

```sql
INSERT INTO tasks (title) VALUES ("Купить молоко");
```

— получишь ошибку `column "Купить молоко" does not exist`. Двойные кавычки в SQL — это **имя колонки или таблицы**, а не строка. Это распространённая ошибка для тех, кто пришёл из языков, где двойные кавычки — это строки.

Если в твоей строке есть одинарная кавычка (например, `O'Brien`), её нужно **удвоить**:

```sql
INSERT INTO tasks (title) VALUES ('O''Brien забыл');
```

Это PostgreSQL-стандарт.

## Уровень 7: про SERIAL и nextval

Тебе может показаться странным: ты вставил `INSERT INTO tasks (id, title) VALUES (100, 'тест')`, а следующий `INSERT` создал строку с `id = 4` (или похожим маленьким числом), а не 101. Почему?

Потому что `SERIAL` под капотом — это **sequence** (последовательность). Когда ты сделал `CREATE TABLE`, PostgreSQL создал sequence с начальным значением 1. Каждый `INSERT` без указания id вызывает `nextval(sequence)` и получает следующее значение.

Когда ты вручную вставил `id = 100`, **sequence не изменился**. Он по-прежнему выдаёт 4, 5, 6 — пока не дойдёт до 100, и тогда будет конфликт `duplicate key`.

Решение — синхронизировать sequence:

```sql
SELECT setval('tasks_id_seq', (SELECT MAX(id) FROM tasks));
```

Это редко нужно в обычной работе. Просто **не вставляй id вручную**, и `SERIAL` будет работать как ожидаешь.

## Уровень 8: про мета-команды psql

Самые полезные:

```
\l              — список баз данных
\c имя_бд       — переключиться на другую БД
\dt             — список таблиц
\d              — список всех объектов (таблицы, sequences, и т.д.)
\d имя_таблицы  — структура таблицы (колонки, типы, constraints)
\q              — выйти из psql
\?              — справка по мета-командам
\h              — справка по SQL (например, \h SELECT)
\timing         — включить/выключить замер времени запросов
```

`\timing on` особенно полезно для понимания, какие запросы быстрые, а какие медленные.

## Уровень 9: про boolean и f/t

Когда ты делаешь `SELECT * FROM tasks`, в колонке `done` ты увидишь `f` или `t`, а не `false`/`true`:

```
 id | title          | done
----+----------------+------
  1 | Купить молоко  | f
  2 | Позвонить маме | t
```

`f` = false, `t` = true. Это **сокращённая запись** в выводе psql, ничего необычного. В Go это будет `bool`, никаких `f`/`t`.

Чтобы проверить булево в WHERE:

```sql
SELECT * FROM tasks WHERE done = true;   -- работает
SELECT * FROM tasks WHERE done = false;  -- работает
SELECT * FROM tasks WHERE done;           -- ТОЖЕ работает (= true)
SELECT * FROM tasks WHERE NOT done;       -- = false
```

Последние две формы — короче и считаются более идиоматичными.

## Если совсем тупик

Скажи Claude конкретно: «модуль 1 проекта 5, [конкретная проблема]». Самые частые ситуации:

- **«Контейнер не запускается»** → покажи вывод `docker logs todo-pg` целиком
- **«psql не подключается»** → покажи вывод `docker ps` (контейнер должен быть в статусе `Up`) и точную команду psql
- **«синтаксическая ошибка SQL»** → покажи запрос, обычно дело в кавычках или забытом `;`
- **«данные пропали после restart»** → проверь, что ты использовал `-v` при `docker run`. Без volume данные действительно теряются.
