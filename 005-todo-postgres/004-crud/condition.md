# Модуль 4. CRUD из Go — условие

## Задание

Реализовать тип `TaskRepository` с пятью методами (`GetAll`, `GetByID`, `Add`, `Update`, `Remove`), которые работают с PostgreSQL через `database/sql`. Это **полная замена** in-memory `TaskList` из проектов 2-4.

Это **полигон**, не интеграция. Программа в этом модуле — простой `main`, который вызывает методы репозитория и выводит результаты. Интеграция в HTTP-сервер — модуль 6.

## Требования

1. Создан тип `Task` с полями `ID int64`, `Title string`, `Done bool` (с JSON-тегами для будущего).
2. Создан тип `TaskRepository` с полем `db *sql.DB`.
3. Создана функция `NewTaskRepository(db *sql.DB) *TaskRepository`.
4. Реализованы **все пять методов**:
   - `GetAll() ([]Task, error)`
   - `GetByID(id int64) (Task, error)`
   - `Add(title string) (Task, error)`
   - `Update(id int64, title string, done bool) (Task, error)`
   - `Remove(id int64) error`
5. **Все запросы используют `$1`, `$2` placeholders**, никаких `fmt.Sprintf` в SQL.
6. **`SELECT *` нигде нет** — везде явно перечисленные колонки.
7. **`defer rows.Close()` сразу после успешного `Query`** в `GetAll`.
8. **`rows.Err()` проверяется после цикла** в `GetAll`.
9. `GetByID` использует `errors.Is(err, sql.ErrNoRows)` для отличия «не найдено» от других ошибок.
10. `Update` использует `RETURNING` для возврата обновлённой задачи.
11. `Remove` использует `RowsAffected()` для проверки «было ли что-то удалено».
12. В `main` сделан полный сценарий из примера выше: добавление 3 задач, GetAll, GetByID, Update, Remove, ошибочные случаи.
13. Перед началом сценария **очищается таблица** через `DELETE FROM tasks`, чтобы запуск был воспроизводимым.
14. Все ошибки оборачиваются через `fmt.Errorf("...: %w", err)`.
15. Использован пакет `log`.

## Эксперименты

1. **Запусти `go run .` дважды.** Что произошло? Если ты не очищаешь таблицу — задачи будут накапливаться. Это разумное поведение для боевого репозитория, но для теста добавь `DELETE FROM tasks` в начало `main`.

2. **Попробуй `SELECT *` вместо явных колонок:**
   ```go
   rows, err := r.db.Query("SELECT * FROM tasks ORDER BY id")
   ```
   Сейчас работает (3 колонки). Теперь добавь миграцию `004_add_priority.sql` с новым полем `priority`. Применили? Запусти программу — упадёт на `Scan` с ошибкой про количество колонок. Это и есть причина **никогда не использовать `SELECT *`**.

3. **Попробуй `fmt.Sprintf` вместо placeholder:**
   ```go
   sql := fmt.Sprintf("SELECT * FROM tasks WHERE id = %d", id)
   r.db.QueryRow(sql)
   ```
   Это работает! Так часто пишут новички. Теперь сделай:
   ```go
   id := 1
   maliciousID := "1; DELETE FROM tasks; --"
   sql := fmt.Sprintf("SELECT * FROM tasks WHERE id = %s", maliciousID)
   _, _ = r.db.Exec(sql)
   ```
   Получишь **полное удаление** всех задач. Это **SQL injection** в чистом виде. Verify через `psql`. **После этого закодируй обратно безопасно через `$1`** — это твоя главная защита.

4. **Забудь `defer rows.Close()`** в `GetAll`. Теперь сделай в цикле:
   ```go
   for i := 0; i < 100; i++ {
       _, _ = repo.GetAll()
   }
   ```
   Программа продолжает работать, но соединения утекают. Со временем (если бы цикл был больше или у тебя был бы веб-сервер) ты бы упёрся в лимит пула. **Верни `defer`**.

5. **Сделай неуспешный `Add`:** попробуй вставить через метод задачу с пустым `title`:
   ```go
   _, err := repo.Add("")
   ```
   Что произошло? Если у тебя нет валидации в Go-коде — задача создалась с пустым title. PostgreSQL не запрещает пустые строки (NOT NULL разрешает `""`). Чтобы запретить — добавь CHECK constraint:
   ```sql
   ALTER TABLE tasks ADD CONSTRAINT title_not_empty CHECK (length(title) > 0);
   ```
   Теперь повтори — упадёт с ошибкой constraint. **БД сама защищает данные**.

6. **Получи задачу с несуществующим id через `GetByID(999)`:**
   ```go
   _, err := repo.GetByID(999)
   if err != nil {
       log.Printf("Ошибка: %v", err)
   }
   ```
   Должно вывести «задача не найдена». Проверь, что `errors.Is(err, sql.ErrNoRows)` сработал.

7. **Удали задачу дважды:**
   ```go
   _ = repo.Remove(1)        // OK
   err := repo.Remove(1)     // должна быть ошибка «не найдена»
   ```
   Это та же идемпотентность, что в Проекте 4: первый раз удалили, второй — нечего удалять.

8. **Используй `int` вместо `int64` для id:**
   ```go
   func (r *TaskRepository) GetByID(id int) (Task, error) {
       // ...
   }
   ```
   На большинстве платформ `int` = `int64`, поэтому работать будет. Но **семантически правильно** использовать `int64` для `BIGSERIAL`/`BIGINT`. Это документация: «эта переменная — большое целое». Привыкай.

9. **(Опционально) Добавь контексты:**
   ```go
   func (r *TaskRepository) GetAllContext(ctx context.Context) ([]Task, error) {
       rows, err := r.db.QueryContext(ctx, "SELECT id, title, done FROM tasks ORDER BY id")
       // ...
   }
   ```
   Сделай несколько методов с контекстом. В модуле 6 ты их используешь при интеграции с HTTP, чтобы запросы отменялись при отвале клиента.

## Что показать Claude

Покажи Claude свой `Update` метод и спроси: «правильно ли я обновляю **через RETURNING**, а не через Exec + отдельный SELECT? Какие плюсы и минусы у каждого варианта?». Это важный вопрос дизайна:

- **`UPDATE ... RETURNING` + Scan** — один запрос, атомарно. Если строки нет → `sql.ErrNoRows`.
- **`Exec` + проверка `RowsAffected` + отдельный `SELECT`** — два запроса, две точки сбоя, и между ними кто-то может изменить данные.

`RETURNING` побеждает почти всегда, когда тебе нужен результат после изменения. Это **PostgreSQL feature**, которой нет в стандартном SQL — учись её использовать, она экономит миллионы багов.

Запиши себе: для **PostgreSQL** проектов всегда предпочитай `RETURNING`, когда нужно получить изменённую строку. Для проектов на MySQL придётся писать в два запроса — это одна из причин, почему PostgreSQL популярнее в современном Go.
