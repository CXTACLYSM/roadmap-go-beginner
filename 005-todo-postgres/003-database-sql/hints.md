## Уровень 2: про установку драйвера

```bash
go get github.com/jackc/pgx/v5/stdlib
```

`go get` скачивает пакет и добавляет его в `go.mod`. После этого пакет можно импортировать.

Если ошибка вроде `cannot find module providing package ...`:
- Проверь, что ты в папке с `go.mod`
- Проверь интернет-соединение
- Попробуй `go mod tidy` — он подчистит зависимости

## Уровень 3: про import _

```go
import (
    "database/sql"

    _ "github.com/jackc/pgx/v5/stdlib"
)
```

Подчёркивание перед именем пакета — это **импорт ради побочного эффекта**. Это значит:

- Пакет **загружается** (его `init()` функции выполняются)
- Но **сам пакет не используется** в твоём коде (нет `pgx.Foo()`)

В случае драйверов БД пакет в своём `init()` **регистрирует себя** в `database/sql`:

```go
// внутри pgx/v5/stdlib
func init() {
    sql.Register("pgx", &Driver{})
}
```

После этого `sql.Open("pgx", ...)` находит зарегистрированный драйвер и использует его.

Если ты импортируешь без `_`:

```go
import "github.com/jackc/pgx/v5/stdlib"
```

— компилятор скажет «imported and not used», потому что ты не используешь имя `stdlib` в коде. Подчёркивание решает эту проблему.

## Уровень 4: про DSN

DSN — Data Source Name. Формат для PostgreSQL:

```
postgres://USER:PASSWORD@HOST:PORT/DATABASE?param1=value1&param2=value2
```

Расшифровка нашего:

```
postgres://postgres:secret@localhost:5432/postgres?sslmode=disable
         |       |     |        |    |        |
         user    pass  host     port db        param
```

- **`user`** — имя пользователя PostgreSQL (`postgres` — стандартный admin)
- **`pass`** — пароль (`secret` — то, что мы указали в `POSTGRES_PASSWORD`)
- **`host`** — адрес сервера (`localhost`, потому что контейнер пробрасывает порт)
- **`port`** — порт (`5432` — стандартный)
- **`db`** — база данных (`postgres` — создаётся автоматически в контейнере)
- **`sslmode=disable`** — выключаем SSL (контейнер по умолчанию его не настраивает)

## Уровень 5: про чтение переменной окружения

```go
dsn := os.Getenv("DATABASE_URL")
if dsn == "" {
    log.Fatal("переменная DATABASE_URL не задана")
}
```

`os.Getenv(name)` возвращает значение переменной или **пустую строку**, если её нет. **Ошибки нет** — пакет `os` различает «нет переменной» и «есть, но пустая» только через `os.LookupEnv`.

Для нашей задачи проверки на пустоту достаточно.

## Уровень 6: про sql.Open

```go
db, err := sql.Open("pgx", dsn)
if err != nil {
    log.Fatalf("sql.Open: %v", err)
}
defer db.Close()
```

**Главный сюрприз**: `sql.Open` обычно **не падает**, даже если БД недоступна. Он только парсит DSN и создаёт объект пула. Реальная попытка подключиться происходит позже.

Поэтому `if err != nil` после `Open` ловит только **синтаксические** ошибки в DSN, не сетевые.

## Уровень 7: про db.Ping

```go
if err := db.Ping(); err != nil {
    log.Fatalf("db.Ping: %v", err)
}
```

`Ping` форсирует **реальное** открытие соединения и шлёт минимальный запрос «ты живой?». Если что-то не так — узнаешь **здесь**.

**Это критичная привычка для серверов:** всегда вызывай `Ping` сразу после `Open`, чтобы упасть на старте, а не через час, когда придёт первый запрос.

## Уровень 8: про QueryRow + Scan

```go
var n int
err := db.QueryRow("SELECT 1").Scan(&n)
if err != nil {
    log.Fatalf("QueryRow: %v", err)
}
log.Printf("результат: %d", n)
```

Несколько моментов:

1. **`QueryRow`** возвращает объект `*sql.Row` — он представляет одну строку (или ноль)
2. **`.Scan(&n)`** копирует первую (и единственную) колонку результата в переменную `n`. Нужен **указатель**, потому что Scan записывает значение **в** переменную
3. **Цепочка вызовов** — `QueryRow().Scan()` — это идиома Go для одной строки

Можно разделить на два вызова:

```go
row := db.QueryRow("SELECT 1")
err := row.Scan(&n)
```

— но цепочка короче и чаще встречается.

## Уровень 9: про Scan и типы

`Scan` принимает указатели разных типов. PostgreSQL и Go типы должны быть **совместимы**:

| PostgreSQL | Go (для Scan) |
|------------|---------------|
| `INTEGER`, `BIGINT`, `SERIAL`, `BIGSERIAL` | `int`, `int32`, `int64` |
| `TEXT`, `VARCHAR` | `string` |
| `BOOLEAN` | `bool` |
| `TIMESTAMPTZ`, `TIMESTAMP` | `time.Time` |
| `NUMERIC` | строка `string` или `*big.Float` (нет идеального) |
| любая колонка, может быть NULL | `*sql.NullString`, `*sql.NullInt64` и т.д. |

Если несоответствие типов — `Scan` вернёт ошибку.

Для `SELECT 1` подходит `int`. Для `SELECT version()` подходит `string`.

## Уровень 10: про несколько колонок в QueryRow

Если запрос возвращает несколько колонок, передавай несколько указателей в `Scan`:

```go
var id int
var title string
var done bool
err := db.QueryRow("SELECT id, title, done FROM tasks WHERE id = 1").Scan(&id, &title, &done)
```

Порядок аргументов **должен совпадать** с порядком колонок в SELECT. Это — частая ошибка.

В модуле 4 ты будешь активно работать с этим.

## Если совсем тупик

Скажи Claude конкретно: «модуль 3, [проблема]». Самые частые ситуации:

- **«unknown driver "pgx"»** → забыл `_ "github.com/jackc/pgx/v5/stdlib"` в импорте
- **«connection refused»** → контейнер не запущен (`docker ps`)
- **«password authentication failed»** → неправильный пароль в DSN
- **«database "X" does not exist»** → нет такой БД, используй `postgres` или создай через `CREATE DATABASE`
- **«SSL is not enabled»** → добавь `?sslmode=disable` к DSN
- **«sql.Open не возвращает ошибку, но Ping падает»** → это нормально, см. уровень 6
