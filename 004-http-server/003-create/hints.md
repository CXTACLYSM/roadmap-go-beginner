## Уровень 2: про чтение тела

```go
var input struct {
    Title string `json:"title"`
}

err := json.NewDecoder(r.Body).Decode(&input)
if err != nil {
    http.Error(w, "невалидный JSON", http.StatusBadRequest)
    return
}
```

Несколько моментов:

- **Локальная структура** объявляется прямо в обработчике. Это анонимный тип, видимый только в этой функции.
- **`Decode(&input)`** — указатель на переменную, как у `Unmarshal`
- **JSON-теги** — обязательны, иначе клиент должен будет писать `Title` с большой буквы
- **`r.Body`** не нужно явно закрывать — Go это сделает автоматически

## Уровень 3: про порядок проверок

В `createTask` есть **три уровня** валидации, и они идут в **этом порядке**:

```go
// Уровень 1: можно ли вообще распарсить JSON?
if err := json.NewDecoder(r.Body).Decode(&input); err != nil {
    http.Error(w, "невалидный JSON", http.StatusBadRequest)
    return
}

// Уровень 2: есть ли обязательные поля?
if strings.TrimSpace(input.Title) == "" {
    http.Error(w, "title обязателен", http.StatusBadRequest)
    return
}

// Уровень 3: можно ли вообще создать задачу?
// (для модуля 3 здесь больше нечего проверять)

// Только теперь — действие
task := s.tasks.Add(input.Title)
```

Сначала самые низкоуровневые проверки (формат), потом семантические (бизнес-правила).

## Уровень 4: про TrimSpace

```go
strings.TrimSpace(input.Title) == ""
```

`TrimSpace` убирает пробелы (включая табы и переводы строк) с обоих концов. Это нужно, чтобы:

- `""` (пустая строка) → пустая → ошибка
- `"   "` (только пробелы) → после TrimSpace `""` → ошибка
- `"  test  "` → после TrimSpace `"test"` → не пустая → ОК

Без `TrimSpace` пользователь может «обмануть» проверку, прислав строку из пробелов.

## Уровень 5: про статус 201

Для успешного создания используй `http.StatusCreated` (это 201):

```go
w.Header().Set("Content-Type", "application/json")
w.WriteHeader(http.StatusCreated)
json.NewEncoder(w).Encode(task)
```

Порядок: сначала заголовки → потом статус → потом тело. Если поменять — Go выведет предупреждение в логи.

Если ты случайно использовал 200 — клиент всё равно поймёт, что всё хорошо. Но 201 — это **точное** соглашение, и хорошие API его соблюдают. В Проектах 5-7 это будет важно.

## Уровень 6: про возврат созданной задачи

После вызова `s.tasks.Add(input.Title)`, метод **должен вернуть** созданную задачу (с присвоенным ID):

```go
func (tl *TaskList) Add(title string) Task {
    task := Task{
        ID:    tl.NextID,
        Title: title,
        Done:  false,
    }
    tl.Tasks = append(tl.Tasks, task)
    tl.NextID++
    return task
}
```

Если у тебя `Add` сейчас ничего не возвращает — поправь. Иначе нечего будет вернуть клиенту.

## Уровень 7: про регистрацию обработчика

```go
http.HandleFunc("POST /tasks", server.createTask)
```

Тот же путь `/tasks`, что у GET — но другой метод. Go различает их и направит запрос в нужный обработчик.

## Уровень 8: про DTO

Ты можешь спросить: «зачем мне отдельная структура для входа, если она такая же, как `Task`?». Ответ — **она НЕ такая же**:

```go
// Внутренний тип
type Task struct {
    ID    int    `json:"id"`
    Title string `json:"title"`
    Done  bool   `json:"done"`
}

// DTO для создания
type createTaskInput struct {
    Title string `json:"title"`
}
```

В DTO **нет ID** (его присваивает сервер) и **нет Done** (новые задачи всегда не выполнены). Если использовать `Task` напрямую — клиент сможет прислать любой ID и Done=true в обход твоей логики.

В нашем простом проекте это «всего лишь» дырка в логике. В реальном API с пользователями и правами — это **уязвимость**, через которую можно создать чужую задачу или украсть ресурс.

## Уровень 9: про curl с -d и -H

Команды для тестирования:

```bash
# POST с JSON-телом
curl -X POST http://localhost:8080/tasks \
    -H "Content-Type: application/json" \
    -d '{"title": "Купить молоко"}'

# С -i — увидеть заголовки ответа
curl -i -X POST http://localhost:8080/tasks \
    -H "Content-Type: application/json" \
    -d '{"title": "test"}'
```

- **`-X POST`** — указывает метод
- **`-H "..."`** — добавляет заголовок
- **`-d '...'`** — тело запроса (одинарные кавычки чтобы внутри были двойные для JSON)

Если у тебя сложный JSON, удобнее положить его в файл:
```bash
echo '{"title": "test"}' > body.json
curl -X POST http://localhost:8080/tasks -H "Content-Type: application/json" -d @body.json
```

## Если совсем тупик

Скажи Claude конкретно: «модуль 3 проекта 4, [конкретная проблема]». Покажи код. Самые частые ситуации:

- «`input.Title` всегда пустой** → проверь JSON-тег на поле, его имя должно совпадать с тем, что в JSON
- «`Decode` всегда возвращает ошибку** → проверь, посылаешь ли ты валидный JSON через curl (одинарные кавычки снаружи)
- «Задача создаётся с ID 0** → метод `Add` не инкрементирует `NextID` или начинается с 0
- «Все запросы возвращают 405** → ты зарегистрировал `"POST /tasks"`, но клиент шлёт `GET /tasks` (или наоборот) — это нормально, надо два обработчика
