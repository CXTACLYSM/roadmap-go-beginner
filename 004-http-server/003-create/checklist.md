## Самопроверка через сценарий

Прогон с нуля:

```bash
go run .

# В другом терминале:
curl http://localhost:8080/tasks
# Должен быть массив с захардкоженными задачами

curl -i -X POST http://localhost:8080/tasks \
    -H "Content-Type: application/json" \
    -d '{"title": "Новая задача"}'
# Должен быть 201 Created с JSON

curl http://localhost:8080/tasks
# Должна появиться новая задача с правильным ID

curl -i -X POST http://localhost:8080/tasks \
    -H "Content-Type: application/json" \
    -d '{"title": ""}'
# Должен быть 400

curl -i -X POST http://localhost:8080/tasks \
    -H "Content-Type: application/json" \
    -d 'не json'
# Должен быть 400

curl -i -X POST http://localhost:8080/tasks
# Пустое тело → 400
```

Если все запросы дают ожидаемые ответы — модуль закрыт.

## Подготовка к модулю 4

В модуле 4 ты добавишь два последних эндпоинта CRUD: **PUT /tasks/{id}** (обновление) и **DELETE /tasks/{id}** (удаление). После них твой API будет **полным**: можно создавать, читать, обновлять, удалять.

Главные новые мысли модуля 4:

- **PUT vs PATCH** — два разных подхода к обновлению. PUT заменяет ресурс целиком, PATCH — частично. Мы делаем PUT, упоминаем PATCH.
- **Идемпотентность** — почему PUT и DELETE считаются идемпотентными, и что это значит на практике
- **Статус 204 No Content** — для DELETE, когда возвращать нечего
- **Ошибка «не найдено» уже встречалась в модуле 2** — теперь ты увидишь её в трёх разных эндпоинтах и подумаешь о повторении кода

В модуле 5 ты впервые увидишь **race condition** на живом примере и научишься защищаться от неё через `sync.Mutex`. Это самая концептуально сложная часть проекта — но ты уже близко.
