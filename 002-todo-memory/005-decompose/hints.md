## Уровень 2: про создание TaskList

Структура объявляется так же, как `Task` — через `type ... struct`:

```go
type TaskList struct {
    tasks  []Task
    nextID int
}
```

Поля **с маленькой буквы** — потому что это внутреннее состояние, и внешний код не должен в него лазить напрямую. К нему обращаемся **только через методы**.

Создание экземпляра:

```go
list := &TaskList{nextID: 1}
```

Заметь `&` — мы создаём **указатель** на структуру. Это потому, что все методы будут с receiver-указателем. Если создать без `&`, ты получишь значение, и при вызовах методов Go будет автоматически брать адрес — это работает, но через указатель явнее.

`tasks` мы не указываем — он по умолчанию `nil`-слайс, что эквивалентно пустому слайсу для `append` и `range`.

## Уровень 3: про методы TaskList

Базовый шаблон метода:

```go
func (tl *TaskList) Add(title string) Task {
    task := Task{ID: tl.nextID, Title: title}
    tl.tasks = append(tl.tasks, task)
    tl.nextID++
    return task
}
```

Несколько моментов:

- `*TaskList` в receiver — это указатель, потому что мы **меняем** состояние (`tl.tasks = ...`, `tl.nextID++`)
- `tl` — короткое имя receiver. Идиома Go: receiver обычно называют 1–2 буквами, не `this` и не `self`
- Возвращаем созданную задачу — чтобы вызывающий мог её показать

## Уровень 4: про возврат error

Шаблон для методов, которые могут провалиться:

```go
func (tl *TaskList) MarkDone(id int) error {
    i := tl.findIndex(id)
    if i == -1 {
        return fmt.Errorf("задача не найдена: %d", id)
    }
    tl.tasks[i].Done = true
    return nil
}
```

И вызов:

```go
if err := list.MarkDone(id); err != nil {
    fmt.Println(err)
    return
}
```

Конструкция `if err := ...; err != nil` — это **очень частая идиома Go**. Она означает: «выполни выражение, объяви `err`, и сразу проверь. Переменная `err` живёт только внутри `if`».

Это полезно, чтобы не загромождать функцию переменными `err`, которые нужны только на одну строку.

## Уровень 5: про то, как делать рефакторинг безопасно

**Главное правило рефакторинга**: маленькие шаги + проверка после каждого.

Не пытайся переписать всё сразу. Делай так:

1. Сделай **одно** изменение
2. Запусти программу
3. Проверь, что она работает
4. Сделай коммит (или хотя бы запомни, что всё ок)
5. Следующее изменение

Если что-то сломалось — ты знаешь, **что именно** ты только что сделал. Откатить одно маленькое изменение в 100 раз проще, чем разбираться, что из 50 изменений сломало программу.

Это называется **incremental refactoring** и это рабочий метод любого профессионала.

## Уровень 6: про parseID

Эта функция — твой первый «настоящий» helper, который устраняет реальную дубликацию:

```go
func parseID(arg, command string) (int, error) {
    if arg == "" {
        return 0, fmt.Errorf("использование: %s <ID>", command)
    }
    id, err := strconv.Atoi(arg)
    if err != nil {
        return 0, fmt.Errorf("ID должен быть числом: %s", arg)
    }
    return id, nil
}
```

Параметр `command` нужен только для **сообщения об ошибке** — чтобы оно было информативным («использование: done <ID>», а не безликое «нужен ID»).

Использование:

```go
func handleDone(list *TaskList, arg string) {
    id, err := parseID(arg, "done")
    if err != nil {
        fmt.Println(err)
        return
    }
    // ...
}
```

## Уровень 7: про порядок действий

Если запутался, что переносить в каком порядке — вот безопасная последовательность:

1. **Сначала тип TaskList.** Просто объяви, без использования.
2. **Перенеси самую простую функцию** — `findIndex`. Сначала как метод, не удаляя оригинал.
3. **Переключи использование** на метод. Запусти. Если работает — удали оригинал.
4. **Повтори для Add, MarkDone, Remove.** По одной за раз.
5. Когда все методы работают — **выноси handle-функции**. Тоже по одной.
6. В самом конце — `parseCommand` и `parseID`.
7. Финальный прогон всех сценариев.

## Если совсем тупик

Скажи Claude: «модуль 5, рефакторинг, застрял на [конкретный шаг]». Покажи код **до** и **после** твоей попытки. Claude поможет понять, что не так — но **не перепишет за тебя**. Это твой код, и навык переделывать его без потери функциональности — это то, что ты тренируешь.

Самые частые ситуации:

- «Не понимаю, почему методу нужен указатель» → перечитай уровень 3, плюс модуль 1 теории про receiver
- «Сломалось после переноса Remove» → проверь, что ты пишешь `tl.tasks = append(...)`, а не `append(tl.tasks, ...)` без присваивания
- «`main` всё равно длинный» → значит, ты не до конца вынес handle-функции. Каждый case в switch должен быть **одной строкой** вызова handler-а
