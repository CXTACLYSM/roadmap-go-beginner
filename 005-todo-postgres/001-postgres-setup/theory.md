# Модуль 1. Установка и базовый SQL — теория

## Контекст

До этого момента твои данные жили в **файлах** (Проекты 3-4). Это работало для одного пользователя и одного экземпляра программы, но как только мы хотим:

- Несколько одновременных пользователей
- Поиск по сложным критериям («все задачи, созданные за неделю, с done=false, принадлежащие пользователю X»)
- Сохранение только **изменённой** части, а не всего файла
- Гарантию, что либо все изменения применились, либо ни одно
- Возможность запустить два экземпляра сервера на одних данных

— файл перестаёт справляться. Нужна **база данных**.

## Ядро 1: что такое реляционная БД

**Реляционная БД** — это специализированная программа, которая хранит данные в виде **таблиц** и предоставляет к ним доступ через язык **SQL**.

Таблица — это как Excel-лист:

```
Таблица: tasks

 id |     title      | done 
----+----------------+------
  1 | Купить молоко  | false
  2 | Позвонить маме | true
  3 | Сделать домашку| false
```

У таблицы есть **колонки** (столбцы) и **строки**. Каждая колонка имеет **тип** (`integer`, `text`, `boolean`, `timestamp` и так далее). Каждая строка — это **одна запись**.

**Реляционная** часть названия означает, что таблицы могут быть **связаны** друг с другом через специальные колонки (foreign keys). Например, у тебя может быть таблица `users` и таблица `tasks`, и каждая задача знает, к какому пользователю она принадлежит. В Проекте 5 у нас одна таблица — связи будут в Проекте 7.

PostgreSQL, MySQL, SQLite, Oracle, MS SQL Server — это всё реляционные БД. Они отличаются деталями, но **базовые концепции и SQL — одинаковые**.

## Ядро 2: почему PostgreSQL

PostgreSQL (часто говорят «постгрес» или «PG») — самая популярная **бесплатная** реляционная БД для серьёзных проектов. Её используют:

- Большинство Go/Python/Ruby/Node.js проектов «по умолчанию»
- Многие крупные компании, включая Apple, Spotify, Reddit, Stripe
- Много стартапов и бэкенд-сервисов

Преимущества PostgreSQL:

- **Надёжность** — данные не теряются
- **ACID-транзакции** — мощные, корректные, проверенные временем
- **Богатый SQL** — поддерживает почти весь стандарт + много расширений
- **Расширения** — JSON, geo, full-text search, и много другого
- **Активное сообщество** — много документации, ответов на StackOverflow

Альтернативы:

- **MySQL** — тоже популярная, чуть проще, чуть менее мощная
- **SQLite** — встроенная БД для одного процесса. Подходит для мобильных приложений и тестов, но не для серверов
- **MongoDB** — не реляционная, документная. Другая философия
- **ClickHouse** — для аналитики, не для OLTP

Для бэкенда — **PostgreSQL по умолчанию**, и ты не пожалеешь.

## Ядро 3: установка через Docker

Самый простой способ запустить PostgreSQL — через **Docker**. Никакой нативной установки, никаких конфликтов с системными пакетами. Один контейнер — одна БД.

```bash
docker run -d \
    --name todo-pg \
    -e POSTGRES_PASSWORD=secret \
    -p 5432:5432 \
    postgres:16
```

Разберём по частям:

- **`docker run`** — запустить контейнер
- **`-d`** — в фоновом режиме (detached)
- **`--name todo-pg`** — даём контейнеру понятное имя, чтобы потом можно было его остановить через `docker stop todo-pg`
- **`-e POSTGRES_PASSWORD=secret`** — пароль для пользователя `postgres` (создаётся автоматически)
- **`-p 5432:5432`** — пробрасываем порт 5432 контейнера на порт 5432 хоста (5432 — стандартный порт PostgreSQL)
- **`postgres:16`** — образ PostgreSQL версии 16 (последняя стабильная на момент создания материала; используй актуальную)

После запуска у тебя есть PostgreSQL, доступный на `localhost:5432`. Логин: `postgres`, пароль: `secret`.

**Полезные команды:**

```bash
docker ps                  # увидеть запущенные контейнеры
docker stop todo-pg        # остановить контейнер
docker start todo-pg       # запустить уже созданный
docker rm -f todo-pg       # удалить (с потерей всех данных!)
docker logs todo-pg        # посмотреть логи
```

⚠️ **Важно про данные.** По умолчанию данные контейнера живут только пока он существует. Если ты сделаешь `docker rm`, всё пропадёт. Чтобы данные пережили пересоздание контейнера, нужен **volume**:

```bash
docker run -d \
    --name todo-pg \
    -e POSTGRES_PASSWORD=secret \
    -p 5432:5432 \
    -v todo-pg-data:/var/lib/postgresql/data \
    postgres:16
```

`-v todo-pg-data:/var/lib/postgresql/data` создаёт **именованный volume** `todo-pg-data` и монтирует его в контейнер. Теперь даже если ты пересоздашь контейнер — данные останутся.

Для учебного проекта это не критично, но привыкай добавлять `-v`. В продакшене без volume'ов данные теряются при первом же restart.

## Ядро 4: psql — командный клиент

`psql` — это интерактивный клиент для PostgreSQL. С ним ты делаешь всё то же самое, что Postman для HTTP: подключаешься, шлёшь запросы, смотришь результаты.

Подключение:

```bash
psql -h localhost -U postgres -d postgres
```

Расшифровка:

- `-h localhost` — хост
- `-U postgres` — пользователь
- `-d postgres` — база данных (по умолчанию у тебя есть БД с именем `postgres`)

Тебя попросит пароль — ввести `secret` (или то, что ты указал в `POSTGRES_PASSWORD`).

После подключения увидишь приглашение:

```
postgres=#
```

Здесь можно вводить SQL-команды. Каждая должна заканчиваться `;`, без него psql ждёт продолжения.

Полезные **мета-команды** psql (начинаются на `\`):

```
\l           — список баз данных
\dt          — список таблиц в текущей БД
\d tasks     — структура таблицы tasks
\q           — выйти из psql
\?           — справка по мета-командам
```

Эти команды — **специфика psql**, не SQL. Они работают только в интерактивном клиенте.

## Ядро 5: базовый SQL — четыре операции CRUD

SQL делит операции на четыре основные:

| Операция | SQL | Что делает |
|----------|-----|------------|
| Create | `INSERT` | Добавить строку(и) |
| Read | `SELECT` | Прочитать строки |
| Update | `UPDATE` | Изменить строки |
| Delete | `DELETE` | Удалить строки |

И ещё одна — **DDL** (Data Definition Language) — для управления **структурой**:

- `CREATE TABLE` — создать таблицу
- `ALTER TABLE` — изменить таблицу
- `DROP TABLE` — удалить таблицу

В этом модуле мы вручную сделаем всё через `psql`, чтобы ты увидел SQL «голым», без обёртки Go. В модулях 3-4 ты будешь делать те же операции из Go-кода.

## CREATE TABLE

```sql
CREATE TABLE tasks (
    id      SERIAL PRIMARY KEY,
    title   TEXT NOT NULL,
    done    BOOLEAN NOT NULL DEFAULT false
);
```

Разберём:

- **`CREATE TABLE tasks (...)`** — создать таблицу с именем `tasks`
- **`id SERIAL PRIMARY KEY`** — колонка `id`, тип `SERIAL` (автоинкрементное целое), является первичным ключом
- **`title TEXT NOT NULL`** — колонка `title`, тип `TEXT` (строка), не может быть `NULL` (обязательная)
- **`done BOOLEAN NOT NULL DEFAULT false`** — `done` — булево, обязательное, по умолчанию `false`

**SERIAL** — это специальный тип PostgreSQL: автоинкрементное целое. Когда ты вставляешь строку без указания `id`, PostgreSQL присвоит следующее свободное число (1, 2, 3, ...).

**PRIMARY KEY** — означает «эта колонка уникально идентифицирует строку». На неё автоматически создаётся индекс, и поиск по `id` будет очень быстрым.

**NOT NULL** — запрещает значение `NULL` (отсутствие значения). Без этого ограничения колонка может быть «пустой», и любое сравнение с ней даёт неожиданные результаты. Хорошее правило: **NOT NULL по умолчанию, NULL только когда осознанно нужен**.

## INSERT

```sql
INSERT INTO tasks (title) VALUES ('Купить молоко');
```

Заметь:

- Указываем **только** `title` — `id` и `done` получат значения по умолчанию (autoincrement и `false`)
- Строка обернута в **одинарные** кавычки (двойные в SQL означают другое — имя колонки)

Чтобы вставить с возвратом созданной строки:

```sql
INSERT INTO tasks (title) VALUES ('Купить молоко') RETURNING *;
```

`RETURNING *` — это PostgreSQL-расширение SQL-стандарта. Возвращает вставленную строку. Очень полезно, чтобы узнать `id`, который только что присвоился.

## SELECT

```sql
SELECT * FROM tasks;             -- все колонки, все строки
SELECT id, title FROM tasks;      -- только id и title

SELECT * FROM tasks WHERE done = false;        -- только невыполненные
SELECT * FROM tasks WHERE id = 1;              -- одна по id
SELECT * FROM tasks WHERE title LIKE '%молоко%'; -- по подстроке

SELECT * FROM tasks ORDER BY id DESC LIMIT 5;  -- последние 5
SELECT COUNT(*) FROM tasks;                     -- сколько всего
```

Мощь SQL — в том, как **просто** делается фильтрация, сортировка, агрегация. Это то, что в файлах было бы десятками строк Go-кода.

## UPDATE

```sql
UPDATE tasks SET done = true WHERE id = 1;
UPDATE tasks SET title = 'Новый заголовок' WHERE id = 2;
UPDATE tasks SET done = true;  -- ⚠️ обновит ВСЕ строки!
```

⚠️ **Без `WHERE` UPDATE применится ко всем строкам.** Это — самая частая катастрофическая ошибка новичков. Всегда проверяй, что у тебя есть `WHERE`, прежде чем нажать Enter.

## DELETE

```sql
DELETE FROM tasks WHERE id = 3;
DELETE FROM tasks WHERE done = true;
DELETE FROM tasks;  -- ⚠️ удалит ВСЕ строки!
```

То же предупреждение: **без `WHERE` удалит всё**. Это абсолютно валидный SQL, но обычно не то, что ты хотел.

## Зачем тебе всё это знать «с psql»

В модулях 3-6 ты будешь делать **те же операции** из Go-кода. И когда что-то пойдёт не так — например, твой Go-код возвращает странную ошибку — первым делом ты будешь идти **в psql** и проверять руками: «что в таблице? что произойдёт, если я выполню этот же запрос вручную?».

Это — **базовый навык бэкендера**. Не «уметь нажимать кнопки в DBeaver», а **руками** через `psql` посмотреть, что в БД. Поэтому первый модуль — про вручную, без Go.
