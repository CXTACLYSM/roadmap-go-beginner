# Проект 4. HTTP-сервер — Todo API

**Длительность:** 3–4 недели
**Цель:** написать первый настоящий бэкенд. Понять, как работает HTTP, как принимать и отвечать на запросы, как обрабатывать ошибки сети, как защищать данные от одновременных изменений.

> 🔥 **Это переломный проект.** До этого ты писал программы «для себя» — запустил, поработал, закрыл. Теперь твоя программа становится **сервером**: она слушает порт и отвечает на запросы, которые приходят извне. С этого момента ты — бэкенд-разработчик, а не «учусь Go».

## Что в итоге будет уметь программа

REST API, к которому можно стучаться из браузера, через `curl`, через Postman, или из любой другой программы:

```
GET    /tasks         → список всех задач
GET    /tasks/{id}    → одна задача
POST   /tasks         → создать новую задачу
PUT    /tasks/{id}    → обновить задачу
DELETE /tasks/{id}    → удалить задачу
```

Пример сессии в терминале:

```bash
$ go run .
Сервер запущен на http://localhost:8080

# В другом терминале:
$ curl -X POST http://localhost:8080/tasks \
    -H "Content-Type: application/json" \
    -d '{"title": "Купить молоко"}'
{"id":1,"title":"Купить молоко","done":false}

$ curl http://localhost:8080/tasks
[{"id":1,"title":"Купить молоко","done":false}]

$ curl -X PUT http://localhost:8080/tasks/1 \
    -H "Content-Type: application/json" \
    -d '{"title": "Купить молоко", "done": true}'
{"id":1,"title":"Купить молоко","done":true}

$ curl -X DELETE http://localhost:8080/tasks/1
# (пустой ответ, статус 204)

$ curl http://localhost:8080/tasks
[]
```

И главное: данные **переживают перезапуск сервера** — потому что в модуле 6 ты подключишь сохранение в файл из Проекта 3.

## Что нового по сравнению с предыдущими проектами

- **HTTP** — протокол, на котором держится весь современный веб. Заявки, ответы, статусы, заголовки, методы. Знание HTTP — это не «бонус» для бэкендера, это базовый язык.
- **Сервер** — программа, которая работает **долго** и обрабатывает много **разных** клиентов. Раньше у тебя был один процесс с одним пользователем. Теперь — много параллельных запросов от незнакомых клиентов.
- **Конкурентность** — твой сервер обрабатывает каждый запрос в **отдельной горутине**. Это значит, что несколько обработчиков могут одновременно лезть в твой `TaskList`. Без защиты — это race condition. С защитой (мьютексами) — корректно. **Это самая большая концептуальная новинка проекта.**
- **JSON в обе стороны** — раньше JSON был только для сохранения. Теперь — это формат **обмена** с клиентами. Принять JSON в теле запроса, ответить JSON в теле ответа.
- **Path patterns в Go 1.22+** — современный стандартный способ роутинга без внешних библиотек. `GET /tasks/{id}` распарсится автоматически.

## Главные мысли проекта

**Первое.** HTTP — это **текстовый протокол поверх TCP**. Запрос — это просто текст определённого формата, который твоя программа читает из сокета. Ответ — тоже текст. Никакой магии. Когда ты это поймёшь, веб перестанет быть страшным.

**Второе.** Каждый HTTP-обработчик в Go запускается в **отдельной горутине**. Это бесплатная конкурентность, но за неё надо платить **дисциплиной защиты данных**. Любая структура, к которой могут обращаться несколько горутин одновременно, должна быть защищена.

**Третье.** Дизайн API — это содержательная инженерная задача. Какой статус возвращать в каком случае? Какие заголовки? Что в теле ответа? Эти вопросы не имеют единственно правильных ответов, но есть **соглашения**, которые ты должен знать. Я их буду показывать по ходу.

**Четвёртое.** Ошибки в HTTP — это **отдельный класс**. Сетевая ошибка не такая, как ошибка диска. Клиент может прислать невалидный JSON, может вообще ничего не прислать, может оборвать соединение посередине. Обработка всех этих случаев — нормальная работа сервера, а не «крайний случай».

## Перед началом

В этом проекте ты будешь часто использовать:

- **`curl`** — командная утилита для HTTP-запросов. На macOS/Linux уже есть. На Windows — ставится отдельно или используй WSL.
- **Браузер** — для GET-запросов. Просто открываешь URL.
- **Postman** (опционально) — графический клиент для HTTP. Удобен, но `curl` достаточно.

Установи `curl` сейчас, если ещё нет. Без него работать с API будет неудобно.
