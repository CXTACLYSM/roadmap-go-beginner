## Уровень 2: про bufio.Scanner

Шаблон, который ты будешь использовать всегда:

```go
import "bufio"
import "os"

scanner := bufio.NewScanner(os.Stdin)

for scanner.Scan() {
    line := scanner.Text()
    // обработка line
}
```

Здесь два момента:

1. `bufio.NewScanner(os.Stdin)` — создание сканера для стандартного ввода
2. `scanner.Scan()` возвращает `true`, пока есть что читать. Это **удобный способ** объединить «прочитать» и «проверить, не кончилось ли»

Для REPL шаблон чуть другой — **бесконечный цикл с явной проверкой**:

```go
for {
    fmt.Print("> ")
    if !scanner.Scan() {
        break
    }
    line := scanner.Text()
    // ...
}
```

Это нужно, чтобы выводить приглашение `> ` **на каждой итерации**, до чтения.

## Уровень 3: про SplitN vs Split

`strings.Split` делит на **все** части:

```go
strings.Split("add foo bar baz", " ")
// ["add", "foo", "bar", "baz"]
```

`strings.SplitN(s, sep, n)` делит **максимум на n частей**:

```go
strings.SplitN("add foo bar baz", " ", 2)
// ["add", "foo bar baz"]
```

Тебе нужен второй вариант — чтобы команда отделилась, а **весь остаток** остался одной строкой.

## Уровень 4: про обращение к parts[1]

Если пользователь ввёл `list` (без аргумента), `SplitN(line, " ", 2)` вернёт **слайс из одного элемента**: `["list"]`. Если ты попытаешься обратиться к `parts[1]` — упадёшь с index out of range.

Поэтому проверка:

```go
parts := strings.SplitN(line, " ", 2)
command := parts[0]
arg := ""
if len(parts) > 1 {
    arg = parts[1]
}
```

## Уровень 5: про состояние между итерациями

**Самая частая ошибка модуля.** Если ты пишешь так:

```go
for {
    var tasks []Task   // <-- ОШИБКА: новый слайс на каждой итерации
    // ...
}
```

— на каждой итерации Go создаёт **новый пустой слайс**, а старый уходит в сборщик мусора. Поэтому ты добавляешь задачу через `add`, а на следующей итерации (когда вызываешь `list`) — слайс снова пустой.

**Правильно — объявить ДО цикла:**

```go
var tasks []Task    // <-- здесь
nextID := 1

for {
    // здесь меняем tasks и nextID, они сохраняют значения
}
```

Это — глобальная идея для всего, что касается серверов и интерактивных программ. **Состояние живёт снаружи цикла обработки**, цикл только меняет его.

## Уровень 6: про exit через return

Чтобы корректно выйти из программы из середины switch внутри цикла:

```go
case "exit":
    fmt.Println("До встречи!")
    return
```

`return` выйдет из функции (`main`), а вместе с ней — из цикла и из программы.

Альтернатива — флаг `running`:

```go
running := true
for running {
    // ...
    case "exit":
        running = false
}
```

— но это лишний код. Просто `return` чище.

## Уровень 7: про `continue`

`continue` — это «пропустить остаток тела цикла и перейти к следующей итерации». Полезно для:

- Пустой строки: `if line == "" { continue }`
- Команды без аргумента: `if arg == "" { fmt.Println("..."); continue }`

После `continue` цикл не завершается — он начинает следующую итерацию с приглашения.

## Если совсем тупик

Скажи Claude: «модуль 3, REPL, проблема: [конкретное описание]». Покажи код. Самые частые ситуации:

- «Задачи не сохраняются» → ищем `var tasks` внутри цикла
- «Add работает только с одним словом» → ищем `Split` вместо `SplitN`
- «Exit с пробелом не работает» → ищем отсутствие `TrimSpace`
- «Программа не реагирует на Enter» → ищем отсутствие `fmt.Print("> ")` или забытый `scanner.Scan()`
