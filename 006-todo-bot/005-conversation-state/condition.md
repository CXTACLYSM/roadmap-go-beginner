# Модуль 5. Состояние диалога — условие

## Задание

Добавить в бота простую FSM, которая позволяет вводить `/add` без аргумента и присылать текст задачи следующим сообщением. Реализовать `/cancel` для выхода из любого состояния.

## Что должно происходить

**Сценарий 1: добавление через состояние**

```
Ты:    /add
Бот:   Введи текст задачи следующим сообщением (или /cancel для отмены):
Ты:    Купить молоко
Бот:   Задача #1 добавлена: Купить молоко

Ты:    /list
Бот:   1. [ ] Купить молоко
```

**Сценарий 2: отмена**

```
Ты:    /add
Бот:   Введи текст задачи следующим сообщением (или /cancel для отмены):
Ты:    /cancel
Бот:   Отменено.

Ты:    /list
Бот:   1. [ ] Купить молоко
```

**Сценарий 3: команда вместо текста**

```
Ты:    /add
Бот:   Введи текст задачи следующим сообщением (или /cancel для отмены):
Ты:    /list
Бот:   Состояние сброшено, обрабатываю команду.
       1. [ ] Купить молоко
```

**Сценарий 4: старое поведение (с аргументом) — не сломалось**

```
Ты:    /add Сделать домашку
Бот:   Задача #2 добавлена: Сделать домашку
```

**Сценарий 5: изоляция между пользователями**

Лев пишет `/add` (входит в состояние ожидания). Аня пишет `привет` — Аня **в Idle**, бот отвечает «Не понимаю». Состояние Льва не пересекается с Аней.

## Требования

1. Создан тип `State int` с двумя константами через `iota`:
   - `StateIdle = 0`
   - `StateWaitingForTaskTitle`
2. Создан тип `StateStore` с полями `states map[int64]State` и `mu sync.Mutex`
3. Реализован конструктор `NewStateStore()`
4. Реализованы методы:
   - `Get(chatID int64) State`
   - `Set(chatID int64, state State)`
5. **`Set` для `StateIdle`** — удаляет запись из map (экономия памяти)
6. **Все методы** `StateStore` используют мьютекс
7. Тип `Bot` обновлён — добавлено поле `states *StateStore`
8. Конструктор `NewBot` принимает `states`
9. Метод `handleMessage` обновлён:
   - Сначала проверяет состояние через `b.states.Get(chatID)`
   - Если состояние не `Idle` — вызывает `handleStateful`
   - Иначе — вызывает `handleCommand` (как раньше)
10. Реализован `handleStateful(chatID, state, text)`:
    - Команда `/cancel` сбрасывает состояние и отвечает «Отменено»
    - Любая команда (начинается с `/`) сбрасывает состояние и обрабатывается через `handleCommand`
    - В `StateWaitingForTaskTitle` — добавляет задачу через `repo.Add` и сбрасывает состояние
    - В `default` — лог + сброс
11. Метод `handleAdd` обновлён:
    - При **пустом** аргументе — устанавливает состояние `WaitingForTaskTitle` и просит ввести текст
    - При **непустом** — работает как раньше (добавляет сразу)
12. В `handleCommand` команда `/cancel` отвечает «Нечего отменять» (когда нет состояния)
13. В `main` создаётся `StateStore` и передаётся в `NewBot`
14. Все сценарии работают
15. **Состояние Льва не влияет на Аню** (per-user изоляция)

## Эксперименты

1. **Перезапусти бота посреди диалога:**
   - Напиши `/add`
   - Бот скажет «Введи текст задачи»
   - Останови бота (Ctrl+C)
   - Запусти снова
   - Напиши `Купить молоко`
   - Что произошло? Бот ответит «Не понимаю команду» — потому что состояние **в памяти**, оно потерялось при перезапуске. Это нормально для модуля 5.

2. **Измерь, что в `StateStore.states` после успешного диалога:** добавь в `handleStateful` после успешного добавления:
   ```go
   log.Printf("После добавления: states map size = %d", len(b.states.states))
   ```
   Должно быть 0 — потому что мы сделали `Set(chatID, StateIdle)`, что удаляет запись.

3. **Подумай, что произойдёт, если пользователь напишет `/add` дважды подряд:**
   ```
   Ты:    /add
   Бот:   Введи текст задачи:
   Ты:    /add
   ```
   В нашей реализации второй `/add` обрабатывается через `handleStateful` → ветка «команда» → сбросит состояние и снова поставит `WaitingForTaskTitle`. То есть бот ответит ещё раз «Введи текст задачи» — это разумное поведение.

4. **Добавь третье состояние:** например, `StateWaitingForDeleteConfirmation`. Когда пользователь делает `/delete N`, бот спрашивает «Точно удалить #N?», и удаляет только при ответе `да`. Это требует:
   - Запоминать **не только** state, но и **аргумент** (id задачи)
   - Тип `StateEntry` со полями `State` и `Data interface{}` (или `int64` для id)
   - Это упражнение, не обязательное для зачёта модуля

5. **Замени map на sync.Map:** попробуй использовать `sync.Map` вместо `map + mutex`. Стало проще или сложнее? Для нашего случая `sync.Map` — оверкилл, она оптимизирована для специфических нагрузок (много чтений, мало записей одних и тех же ключей). Для нашего сценария обычная map с мьютексом — стандартный выбор.

6. **Добавь TTL на состояние:**
   ```go
   type StateEntry struct {
       State    State
       Deadline time.Time
   }
   ```
   Если в `Get` `Deadline` прошёл — возвращай `StateIdle` и удаляй запись. Запусти, проверь, что после 5 минут «зависания» состояние само сбросится.

7. **Проверь изоляцию состояний:** напиши `/add` от Льва, потом сразу с другого аккаунта (Ани) напиши обычную команду `/list`. Аня не должна попасть в состояние Льва.

## Что показать Claude

Покажи Claude свой `handleStateful` и спроси: «правильно ли я обрабатываю случай, когда пользователь в состоянии `WaitingForTaskTitle` пишет другую команду?». Это вопрос **юзабилити**:

- **Жёсткое поведение** — игнорировать команды в состоянии, просить либо текст, либо `/cancel`
- **Дружественное** (наш выбор) — любая команда сбрасывает состояние и выполняется
- **Гибридное** — некоторые команды сбрасывают (`/list`, `/help`), некоторые — нет

Для нашего проекта дружественное поведение лучше, потому что:
- Не «застреваешь» в состоянии при ошибочном `/add`
- Команды интуитивно «приоритетнее» обычного текста

Запиши себе на будущее: **дизайн UX state machines** — это отдельная дисциплина. Главное правило: **всегда давай пользователю способ выйти**. Без `/cancel` (или эквивалента) ты строишь ловушку.
