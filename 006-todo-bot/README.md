# Проект 6. To-Do бот для Telegram

**Длительность:** 3 недели
**Цель:** написать долгоживущего клиента к внешнему HTTP API. Понять, как устроены боты «изнутри», без волшебных библиотек. Освоить event-loop парадигму, состояние диалога, и graceful shutdown.

> 🔄 **Это смена парадигмы.** В Проектах 4-5 ты писал **сервер** — программу, которая слушает порт и отвечает на запросы. В Проекте 6 — **клиент**: программа, которая сама опрашивает чужой API и реагирует на полученные события. Это два самых распространённых типа бэкенд-программ, и оба ты должен уметь писать.

## Что в итоге будет уметь программа

Telegram-бот, к которому можно написать в личку и управлять своим списком задач:

```
Ты:    /start
Бот:   Привет! Я твой to-do бот. Команды:
       /list — показать задачи
       /add <текст> — добавить задачу
       /done <id> — отметить выполненной
       /delete <id> — удалить
       /help — справка

Ты:    /add Купить молоко
Бот:   Задача #1 добавлена: Купить молоко

Ты:    /add
Бот:   Введи текст задачи следующим сообщением:

Ты:    Позвонить маме
Бот:   Задача #2 добавлена: Позвонить маме

Ты:    /list
Бот:   1. [ ] Купить молоко
       2. [ ] Позвонить маме

Ты:    /done 1
Бот:   Задача #1 отмечена выполненной

Ты:    /list
Бот:   1. [x] Купить молоко
       2. [ ] Позвонить маме
```

Каждый пользователь Telegram имеет **свой собственный** список задач. Данные хранятся в PostgreSQL.

## Что нового по сравнению с предыдущими проектами

- **HTTP как клиент, не как сервер** — теперь ты **сам** отправляешь HTTP-запросы (через `net/http` Client), а не отвечаешь на них. Это важная зеркальная картинка: понимать обе стороны протокола.
- **Long polling** — паттерн получения событий через постоянные «висячие» запросы. Альтернатива webhook'ам, требующим публичного URL.
- **External API client** — тонкая обёртка над чужим HTTP API. У тебя будет свой `tgbotapi`-подобный объект, написанный с нуля.
- **Multi-user state** — каждый Telegram-юзер — отдельный «клиент» со своими данными. Данные бота — это не один глобальный список, а **много** независимых.
- **Stateful conversations** — диалог, в котором следующее сообщение пользователя интерпретируется на основе **прошлых**. Простая форма FSM (Finite State Machine).
- **`context.Context`** — впервые серьёзно. Контекст переносится через все слои, передаётся в БД и HTTP-запросы, отменяется при graceful shutdown.
- **Signal handling** — `signal.Notify` для перехвата Ctrl+C / SIGTERM. Без этого долгоживущий процесс не сможет завершиться корректно.

## Что нового **не** будет

- **Webhooks** — упомянем как альтернативу long polling, но не реализуем. Нужен публичный HTTPS-endpoint, что требует деплоя или туннеля типа ngrok.
- **Inline keyboards и кнопки** — упомянем в extras, в основной поток не входят. Концептуально это просто другой формат ответа.
- **Голосовые / фото / видео** — Telegram Bot API поддерживает много типов сообщений, но для изучения концепций хватает текста.
- **`go-telegram-bot-api/telegram-bot-api`** или другие библиотеки — **сознательно**. Сначала ты должен понимать, что внутри, прежде чем доверять обёртке. Поэтому пишем своего минимального клиента.
- **Деплой на сервер** — это уже Проект 7 или отдельная тема.

## Главные мысли проекта

**Первое.** Бот — это не «магическая программа». Это **HTTP-клиент**, который в цикле спрашивает у Telegram «есть новые сообщения?», получает JSON, обрабатывает, и шлёт другие HTTP-запросы для отправки ответов. Когда ты это поймёшь, **любой** REST API мира станет для тебя «просто данными по HTTP».

**Второе.** Долгоживущий процесс — это не «сервер с другим интерфейсом». Это **другой тип ответственности**: ты должен выдержать часы, дни и недели работы без перезапуска. Это значит — не утекать памятью, корректно обрабатывать ошибки сети, переподключаться при сбоях, и **корректно завершаться** при сигнале остановки.

**Третье.** State per user — самая частая структура данных в любом сервисе. У каждого пользователя свой контекст, свои данные, свои настройки. Это требует **изоляции**: данные одного пользователя не должны попадать к другому. В Telegram это решается через `chat_id` (id чата = id пользователя в личке), и каждый запрос к БД фильтруется по нему.

**Четвёртое.** Stateful conversations — это первое прикосновение к **state machines** в реальной задаче. Когда пользователь пишет `/add` без аргумента, бот переходит в **состояние «жду текст»**, и следующее сообщение от **этого** пользователя интерпретируется не как команда, а как заголовок задачи. Это маленькая FSM, и её принципы масштабируются на куда более сложные задачи.

**Пятое.** Graceful shutdown — это **обязательная** часть production-ready сервиса. Когда придёт сигнал «остановись» (Ctrl+C, SIGTERM от systemd, kubernetes preStop hook), твой код должен:
1. Прекратить принимать новые задачи
2. Доделать те, что уже взял в работу
3. Закрыть БД, HTTP-клиенты, файлы
4. Завершиться с кодом 0

В Проектах 4-5 ты завершал сервер через Ctrl+C «грубо». Здесь ты научишься делать это правильно.

## Перед началом

1. **Установи Telegram** на телефон или используй [Telegram Web](https://web.telegram.org). Без этого бота не с кем тестировать.
2. **Найди @BotFather** в Telegram — это «отец всех ботов», через которого регистрируются новые боты. Ничего сейчас не делай, просто убедись, что нашёл — в модуле 1 ты создашь там бота.
3. **Убедись, что у тебя работает PostgreSQL** из Проекта 5. В модуле 4 ты переиспользуешь контейнер `todo-pg` (можно тот же, можно отдельный — это твой выбор).
4. **Подумай об интернете.** Long polling работает поверх обычного HTTPS, без VPN и прокси (если у тебя нет блокировок Telegram). Если есть — настрой VPN или используй прокси на уровне `net/http.Client`.
