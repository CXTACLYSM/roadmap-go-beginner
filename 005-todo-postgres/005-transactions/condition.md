# Модуль 5. Транзакции — условие

## Задание

Расширить `TaskRepository` методом `AddBatch`, который вставляет **много задач атомарно** через транзакцию. Прочувствовать принцип «всё или ничего» на собственном опыте через эксперименты с искусственным сбоем посередине.

## Что должна делать программа

```bash
$ DATABASE_URL="postgres://postgres:secret@localhost:5432/postgres?sslmode=disable" go run .
2026/04/13 12:00:00 До: 0 задач
2026/04/13 12:00:00 AddBatch: создано 3 задач
2026/04/13 12:00:00   {ID:1 Title:Купить молоко Done:false}
2026/04/13 12:00:00   {ID:2 Title:Позвонить маме Done:false}
2026/04/13 12:00:00   {ID:3 Title:Сделать домашку Done:false}
2026/04/13 12:00:00 После: 3 задач

2026/04/13 12:00:00 --- Эксперимент: сбой посередине ---
2026/04/13 12:00:00 AddBatch с одной плохой задачей упал: insert task "": ...
2026/04/13 12:00:00 После сбоя: 3 задач (ничего не добавилось)
```

Главный момент: после второго `AddBatch`, который должен был вставить 3 задачи, но упал на пустом title — **в БД должно быть всё ещё 3 задачи**, а не 4 или 5. Это и есть атомарность.

## Требования

1. К существующему `TaskRepository` (из модуля 4) добавлен метод `AddBatch(titles []string) ([]Task, error)`
2. `AddBatch` использует транзакцию через `db.Begin()`
3. **Сразу после `Begin`** стоит `defer tx.Rollback()`
4. Все INSERT внутри `AddBatch` идут через `tx`, не через `r.db`
5. В конце вызывается `tx.Commit()`
6. Если **любой** из INSERTs упал — `Commit` не вызывается, и `defer Rollback` отменяет всю транзакцию
7. В таблицу добавлен `CHECK constraint`:
   ```sql
   ALTER TABLE tasks ADD CONSTRAINT title_not_empty CHECK (length(title) > 0);
   ```
   Это нужно, чтобы пустой title вызывал ошибку **на уровне БД**, и мы могли испытать `Rollback`.
8. В `main` сделаны два сценария:
   - **Успешный**: `AddBatch(["Купить молоко", "Позвонить маме", "Сделать домашку"])` → 3 задачи в БД
   - **С ошибкой посередине**: `AddBatch(["задача 1", "", "задача 3"])` → ошибка, и в БД **ничего не добавилось**
9. После каждого сценария вызывается `GetAll` для проверки состояния
10. Используется уже существующий код модуля 4 (типы, базовые методы, подключение) — это **продолжение**, а не новый проект

## Эксперименты

1. **Уберите `defer tx.Rollback()`:**
   ```go
   tx, _ := r.db.Begin()
   // нет defer
   ```
   Запустите сбойный сценарий. Что происходит? В большинстве случаев `database/sql` сам в конце выполнит `Rollback` (потому что `tx` уходит в garbage collection и его финализатор откатывает транзакцию). **Но это не гарантировано** и может занять время. Без явного `Rollback` ты надеешься на сборщик мусора. **Это плохо.** Верните `defer`.

2. **Уберите `Commit` (вообще):**
   ```go
   // if err := tx.Commit(); err != nil { ... }
   return created, nil
   ```
   Запусти успешный сценарий. Что в БД после `GetAll`? **Ничего!** Без `Commit` `defer Rollback` отменит даже успешные операции. Это иллюстрирует: **`Commit` обязателен для применения изменений**.

3. **Используй `r.db.Exec` вместо `tx.Exec` внутри транзакции:**
   ```go
   for _, title := range titles {
       _, err := r.db.Exec("INSERT INTO tasks (title) VALUES ($1)", title)  // ← через db!
       // ...
   }
   ```
   Запусти сбойный сценарий. Что в БД? **Две задачи добавились** — те, что были до сбоя. Это потому, что `r.db.Exec` выполнялся **вне** транзакции. Атомарности нет.

4. **Замерь время батчевой вставки vs одиночных:**
   ```go
   import "time"

   // одиночные
   start := time.Now()
   for i := 0; i < 1000; i++ {
       _, _ = repo.Add(fmt.Sprintf("task %d", i))
   }
   log.Printf("Одиночные: %v", time.Since(start))

   // батч
   titles := make([]string, 1000)
   for i := range titles {
       titles[i] = fmt.Sprintf("task %d", i)
   }
   start = time.Now()
   _, _ = repo.AddBatch(titles)
   log.Printf("Батч: %v", time.Since(start))
   ```
   Сравни. Батч в транзакции обычно **в разы** быстрее. Это потому, что транзакция уменьшает накладные расходы на коммит и сетевые туда-обратно.

5. **Сделай две параллельные транзакции:**
   ```go
   var wg sync.WaitGroup
   for i := 0; i < 2; i++ {
       wg.Add(1)
       go func(i int) {
           defer wg.Done()
           _, _ = repo.AddBatch([]string{
               fmt.Sprintf("горутина %d - 1", i),
               fmt.Sprintf("горутина %d - 2", i),
           })
       }(i)
   }
   wg.Wait()
   ```
   Сколько задач в итоге? 4 — все 4. **Никаких race conditions** — БД сама обрабатывает параллельные транзакции. Это и есть **изоляция**.

6. **Попробуй внутри транзакции сделать `time.Sleep(10 * time.Second)`:**
   ```go
   tx, _ := r.db.Begin()
   defer tx.Rollback()

   _, _ = tx.Exec("UPDATE tasks SET done = true WHERE id = 1")
   time.Sleep(10 * time.Second)  // ← плохо!
   _, _ = tx.Exec("UPDATE tasks SET done = false WHERE id = 1")
   tx.Commit()
   ```
   В одном терминале запусти это. В другом, через 1-2 секунды, попробуй обновить ту же строку через `psql`:
   ```sql
   UPDATE tasks SET title = 'другое' WHERE id = 1;
   ```
   Что произошло? Второй запрос **зависнет** до коммита первой транзакции. Это и есть **блокировка строки**. Урок: **никогда не делай долгие операции внутри транзакции**.

7. **Воспроизведи сценарий «частичный сбой» с явной паникой:**
   ```go
   for i, title := range titles {
       if i == 1 {
           panic("симуляция падения")
       }
       // ... INSERT через tx ...
   }
   ```
   Запусти. Программа упадёт с panic, но `defer tx.Rollback()` всё равно сработает, и в БД ничего не появится. Это **defer работает даже при panic** — одна из суперспособностей Go.

## Что показать Claude

Покажи Claude свой `AddBatch` и спроси: «правильно ли я обрабатываю ошибку `Commit`? Что произойдёт, если `Commit` упадёт?». Это тонкий момент:

- **Если `Commit` упал** — транзакция **уже** в неопределённом состоянии. Может быть, изменения применились, может быть, нет. Обычно ты возвращаешь ошибку, и `defer Rollback` пытается откатить (но если уже закоммичено — это no-op).
- **Что делать клиенту** — повторить операцию? Зависит от того, идемпотентна ли она. Для INSERTs обычно нельзя повторять (получишь дубликаты).

Это серьёзная тема, и в Проектах 6-7 ты увидишь, как с ней работают через **идемпотентные ключи** и **outbox pattern**. Сейчас просто понимай: транзакции **не панацея**, и edge case-ы существуют.

Запиши себе: **транзакции дают atomicity, isolation, durability — но не освобождают тебя от продумывания edge cases**. В реальном продакшен-коде есть retry-логика, идемпотентные ключи, и компенсирующие действия. Это уже Saga, Outbox, и другие паттерны distributed systems.
