# Модуль 3. database/sql из Go — теория

## Контекст

В модулях 1-2 ты работал с PostgreSQL **снаружи Go**: через `psql`, через файлы миграций, через ручной SQL. Это нужно было, чтобы понимать, **что именно** происходит в БД, без обёртки.

Теперь — Go. В этом модуле ты подключишь свою программу к PostgreSQL и сделаешь первый запрос. Никаких CRUD-операций — только подключение, ping, простой `SELECT 1`. Это **минимальный «hello world»** для работы с БД из Go.

## Ядро 1: что такое database/sql

`database/sql` — это **стандартный пакет** Go для работы с SQL-базами данных. Он находится в стандартной библиотеке (не нужно ставить отдельно).

Главная идея: `database/sql` — **абстракция**. Он не знает про конкретную БД (PostgreSQL, MySQL, SQLite). Он знает только про **SQL** в общем. Конкретный диалект — это **драйвер**, который ты подключаешь отдельно.

Это даёт большое преимущество: если ты сегодня пишешь под PostgreSQL, а завтра решишь переехать на MySQL — твой Go-код почти не изменится. Изменится только драйвер и, возможно, мелочи в самих SQL-запросах.

```go
import (
    "database/sql"
    _ "github.com/jackc/pgx/v5/stdlib"   // драйвер
)

db, err := sql.Open("pgx", "postgres://user:pass@host:port/dbname")
```

`database/sql` знает про **интерфейс** `Driver` — а конкретные драйверы реализуют его. Импорт `_ "..."` — это **импорт ради побочного эффекта**: пакет регистрирует себя в `database/sql`, чтобы быть доступным под именем `"pgx"`.

## Ядро 2: какой драйвер выбрать

Для PostgreSQL в современном Go есть два главных драйвера:

| | `lib/pq` | `pgx` |
|---|----------|-------|
| Возраст | Старый (с 2012) | Новее (с 2017) |
| Активная разработка | Заморожен | Активная |
| Производительность | Средняя | Высокая |
| Нативный API | — | Есть (помимо database/sql) |
| Поддержка фич PostgreSQL | Базовая | Расширенная |
| Используется в новых проектах | Редко | **Обычно** |

**Используем `pgx`.** Это современный стандарт. Конкретно — `pgx/v5/stdlib`, который реализует интерфейс `database/sql/driver`, чтобы ты мог использовать его через стандартный `database/sql`.

Если бы тебе нужен был максимум возможностей pgx (типы PostgreSQL без потерь, более удобный API, batch-операции), ты бы использовал **нативный** `pgx` (без `stdlib`). Но это уже другой курс. Для нас `database/sql + pgx как драйвер` — оптимально.

Установка:

```bash
go get github.com/jackc/pgx/v5/stdlib
```

## Ядро 3: подключение через sql.Open

```go
import (
    "database/sql"
    _ "github.com/jackc/pgx/v5/stdlib"
)

func main() {
    dsn := "postgres://postgres:secret@localhost:5432/postgres?sslmode=disable"
    db, err := sql.Open("pgx", dsn)
    if err != nil {
        log.Fatalf("sql.Open: %v", err)
    }
    defer db.Close()

    // ...
}
```

Несколько важных моментов:

1. **`sql.Open("pgx", dsn)`** — первый аргумент это **имя драйвера** (то самое, под которым `pgx/v5/stdlib` себя зарегистрировал). Второй — **DSN** (Data Source Name).

2. **DSN-формат**: `postgres://user:password@host:port/database?param=value`. Стандарт URL — параметры через `?` и `&`. `sslmode=disable` нужен, потому что наш контейнер не настроен на SSL (в продакшене всегда `sslmode=require` или строже).

3. **`sql.Open` НЕ ОТКРЫВАЕТ соединение.** Это контринтуитивно: имя обманывает. На самом деле `Open` только парсит DSN и создаёт **объект пула**. Реальное соединение откроется при первом запросе.

4. **`defer db.Close()`** — закрыть пул при выходе из `main`. На практике для серверов это не критично (программа всё равно завершится), но привычка хорошая.

## Ядро 4: db.Ping — реальная проверка соединения

Чтобы убедиться, что подключение **реально работает**, нужно выполнить `Ping`:

```go
if err := db.Ping(); err != nil {
    log.Fatalf("db.Ping: %v", err)
}
```

`Ping` форсирует открытие соединения и шлёт минимальный запрос («ты живой?»). Если БД недоступна — ты узнаешь **сейчас**, а не при первом полезном запросе через 30 секунд.

**Делай `Ping` сразу после `sql.Open`** — это лучшая практика для серверных приложений. Если БД нет — программа должна **упасть на старте**, а не молча работать с надеждой, что данные где-то есть.

## Ядро 5: connection pool

`sql.DB` — это **не одно соединение**, это **пул соединений**. Когда ты делаешь `db.Query(...)`, происходит:

1. `database/sql` берёт **свободное** соединение из пула
2. Шлёт по нему запрос
3. После получения результата возвращает соединение в пул (для следующих запросов)

Это даёт несколько важных свойств:

- **Параллельность**: разные горутины могут делать разные запросы **одновременно**, потому что у каждой будет своё соединение из пула
- **Производительность**: установка нового TCP-соединения с БД дорогая (~100ms). Пул переиспользует существующие.
- **Контроль**: можно ограничить максимальное число соединений, чтобы не положить БД

### Настройка пула

```go
db.SetMaxOpenConns(25)         // максимум открытых соединений
db.SetMaxIdleConns(5)          // максимум idle (готовых, но не используемых)
db.SetConnMaxLifetime(5 * time.Minute)  // макс. время жизни одного соединения
```

Эти настройки **важны для продакшена**:

- **`SetMaxOpenConns(25)`** — типичное значение для веб-сервера. Без ограничения сервер под нагрузкой может создать тысячи соединений и положить БД.
- **`SetMaxIdleConns(5)`** — сколько соединений держать «горячими» для быстрого использования.
- **`SetConnMaxLifetime(5 * time.Minute)`** — переоткрывать соединения раз в 5 минут. Помогает при балансировке нагрузки и сетевых сбоях.

Для нашего учебного проекта эти настройки не критичны — но **знать их обязательно**.

## Ядро 6: конфигурация через переменные окружения

В коде **никогда не хардкодь** учётные данные БД:

```go
// ПЛОХО
dsn := "postgres://postgres:secret@localhost:5432/postgres?sslmode=disable"

// ХОРОШО
dsn := os.Getenv("DATABASE_URL")
if dsn == "" {
    log.Fatal("DATABASE_URL не задан")
}
```

Запуск:

```bash
DATABASE_URL="postgres://postgres:secret@localhost:5432/postgres?sslmode=disable" go run .
```

Или, для удобства, через `.env` файл (но это требует библиотеки типа `godotenv` или ручного парсинга).

**Зачем переменные окружения?**

1. **Безопасность** — пароль не лежит в коде, который может быть в публичном репозитории
2. **Гибкость** — можно запускать с разными БД (локальная, тестовая, продакшен) без перекомпиляции
3. **12-Factor App** — это часть стандарта современного бэкенда (см. https://12factor.net/)

В Проекте 7 ты узнаешь про **viper** и более структурированную конфигурацию. Сейчас — простые `os.Getenv`.

## Ядро 7: первый запрос — db.QueryRow

Самый простой запрос, который возвращает одну строку:

```go
var result int
err := db.QueryRow("SELECT 1").Scan(&result)
if err != nil {
    log.Fatalf("QueryRow: %v", err)
}
fmt.Println("Результат:", result)
```

Разберём:

1. **`db.QueryRow(sql)`** — выполняет запрос и возвращает объект `*sql.Row`, представляющий **одну** строку результата
2. **`.Scan(&result)`** — копирует значения колонок результата в указанные переменные. Здесь `&result` — указатель на `int`, потому что `Scan` записывает данные **в** переменную
3. **`SELECT 1`** — самый минимальный запрос: «дай мне число 1». Используется для тестирования соединения.

`QueryRow` подходит, когда ты **точно ожидаешь** одну строку (или ноль). Если запрос возвращает несколько — `QueryRow` берёт первую и игнорирует остальные.

`Scan` — самый частый источник ошибок: количество и **типы** аргументов должны соответствовать колонкам результата. Если запрос возвращает 3 колонки, а ты передал 2 указателя — ошибка.

## Ядро 8: типичные ошибки подключения

Если ты увидишь одну из этих ошибок, вот что они значат:

**`pq: SSL is not enabled on the server`** или **`server does not support SSL`**

Добавь в DSN `?sslmode=disable`. Контейнер по умолчанию не настроен на SSL. В продакшене — другое дело.

**`dial tcp [::1]:5432: connect: connection refused`**

PostgreSQL не запущен или не на этом порту. Проверь `docker ps`.

**`pq: password authentication failed for user "postgres"`**

Неправильный пароль. Проверь, что ты передал правильное значение в DSN.

**`pq: database "todo" does not exist`**

Базы данных с таким именем нет. Сначала создай её через `CREATE DATABASE` или используй существующую (`postgres` создаётся автоматически).

**`unknown driver "pgx"`** или **`sql: unknown driver "pgx" (forgotten import?)`**

Ты забыл импортировать драйвер. Должно быть:

```go
import _ "github.com/jackc/pgx/v5/stdlib"
```

Подчёркивание перед именем пакета — это **import for side effect**. Пакет регистрирует себя в `database/sql`, и ты используешь его через `sql.Open("pgx", ...)`.

## Полный пример — программа модуля 3

```go
package main

import (
    "database/sql"
    "log"
    "os"

    _ "github.com/jackc/pgx/v5/stdlib"
)

func main() {
    dsn := os.Getenv("DATABASE_URL")
    if dsn == "" {
        log.Fatal("DATABASE_URL не задан")
    }

    db, err := sql.Open("pgx", dsn)
    if err != nil {
        log.Fatalf("sql.Open: %v", err)
    }
    defer db.Close()

    if err := db.Ping(); err != nil {
        log.Fatalf("db.Ping: %v", err)
    }

    log.Println("Подключение к PostgreSQL установлено")

    var n int
    if err := db.QueryRow("SELECT 1").Scan(&n); err != nil {
        log.Fatalf("QueryRow: %v", err)
    }
    log.Printf("SELECT 1 вернул: %d", n)

    var version string
    if err := db.QueryRow("SELECT version()").Scan(&version); err != nil {
        log.Fatalf("version: %v", err)
    }
    log.Printf("Версия PostgreSQL: %s", version)
}
```

Запуск:

```bash
DATABASE_URL="postgres://postgres:secret@localhost:5432/postgres?sslmode=disable" go run .
```

Вывод:

```
2026/04/13 12:00:00 Подключение к PostgreSQL установлено
2026/04/13 12:00:00 SELECT 1 вернул: 1
2026/04/13 12:00:00 Версия PostgreSQL: PostgreSQL 16.1 (...)
```

Это твой первый Go-код, общающийся с PostgreSQL. Возьми момент порадоваться — это вход в большой и важный мир.
