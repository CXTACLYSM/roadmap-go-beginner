## Самопроверка: финальный сценарий

Полный прогон с нуля:

```bash
# 1. Удаляем старый файл
rm -f tasks.json

# 2. Запускаем сервер
go run -race . &
SERVER_PID=$!
sleep 1

# 3. Создаём задачи
curl -s -X POST http://localhost:8080/tasks -H "Content-Type: application/json" -d '{"title":"Купить молоко"}' > /dev/null
curl -s -X POST http://localhost:8080/tasks -H "Content-Type: application/json" -d '{"title":"Позвонить маме"}' > /dev/null
curl -s -X PUT http://localhost:8080/tasks/1 -H "Content-Type: application/json" -d '{"title":"Купить молоко","done":true}' > /dev/null

# 4. Проверяем файл
cat tasks.json

# 5. Останавливаем сервер
kill $SERVER_PID
sleep 1

# 6. Запускаем заново
go run -race . &
SERVER_PID=$!
sleep 1

# 7. Проверяем, что данные сохранились
curl -s http://localhost:8080/tasks

# 8. Останавливаем
kill $SERVER_PID
```

Если **после перезапуска** ты видишь те же задачи (включая `done: true` для первой) — модуль закрыт.

- Опыт работы с **пакетом `log`** для серверных приложений
- Применение **DTO-паттерна** для безопасной работы с входными данными

Самые важные мысли, которые ты усвоил:

1. **HTTP — это просто текст.** Никакой магии. Каждый запрос — это структурированный текст, который ты читаешь из сокета.

2. **Дизайн API — содержательная инженерная задача.** Какой статус? Какие заголовки? Что в теле? Эти вопросы определяют, насколько твой API удобен.

3. **Конкурентность — это серьёзно.** Один и тот же код может работать без проблем под одиночной нагрузкой и портить данные под параллельной. Защищать состояние — это **правило**, а не «хорошо бы».

4. **DTO защищает от уязвимостей.** Принимать `Task` напрямую от клиента — это дыра, через которую можно установить чужие поля. DTO — простая и эффективная защита.

5. **Слои в архитектуре.** Данные знают, как себя защищать (мьютекс). Обработчики знают про HTTP. Main знает про инициализацию. Никто не лезет в чужие зоны.
