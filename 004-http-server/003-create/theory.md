# Модуль 3. POST — создание задачи — теория

## Контекст

В модуле 2 твой сервер умеет **отдавать** данные. Это половина REST API. Вторая половина — **принимать** данные от клиента и что-то с ними делать. Самый базовый случай — создание ресурса через POST.

Этот модуль вводит несколько важных концепций, которые ты будешь использовать во всех следующих эндпоинтах:

- **Чтение тела запроса** — клиент присылает JSON, ты его парсишь
- **Валидация входных данных** — данные снаружи нельзя доверять, их надо проверять
- **Правильный статус для создания** — 201 Created, не 200 OK
- **Возврат созданного ресурса** — клиенту нужен ID, который ты ему присвоил

## Ядро 1: чтение тела запроса

Тело HTTP-запроса — это поток байтов, которые клиент прислал в дополнение к стартовой строке и заголовкам. Для POST это обычно JSON.

В Go тело — это `io.ReadCloser`, доступный через `r.Body`. Симметрично с записью JSON через `Encoder`, для чтения JSON есть **`Decoder`**:

```go
var task Task
err := json.NewDecoder(r.Body).Decode(&task)
if err != nil {
    http.Error(w, "невалидный JSON", http.StatusBadRequest)
    return
}
```

Несколько важных моментов:

1. **`json.NewDecoder(r.Body)`** — создаём декодер, привязанный к телу запроса
2. **`.Decode(&task)`** — парсим JSON и записываем в структуру по указателю (как `Unmarshal`)
3. **Если JSON битый** — `Decode` вернёт ошибку. Это ошибка **клиента** → возвращаем **400**

`r.Body` сам по себе требует **закрытия** после использования (это `Closer`), но `Decoder` делает это за нас, плюс Go автоматически закрывает тело при завершении обработчика. Так что явно закрывать **не обязательно**.

## Encoder/Decoder vs Marshal/Unmarshal

В Проекте 3 ты использовал `Marshal`/`Unmarshal` для работы с файлами. Здесь — `Encoder`/`Decoder`. Зачем два набора, делающих одно и то же?

Разница в **источнике/получателе**:

- **`Marshal`/`Unmarshal`** работают с `[]byte`. Удобно, когда у тебя данные «в руках» (например, из `os.ReadFile`).
- **`Encoder`/`Decoder`** работают с потоками (`io.Reader`/`io.Writer`). Удобно для HTTP, файлов, сетевых соединений — когда данные **текут**, а не лежат целиком в памяти.

Для HTTP-обработчиков **используй `Decoder` и `Encoder`** — это и удобнее, и чуть эффективнее (не создаются промежуточные буферы).

## Ядро 2: валидация входных данных

**Это самая важная мысль модуля.**

До этого момента ты работал с **своими** данными — теми, что сам же и создал. К ним было доверие. Теперь данные приходят **извне**, и доверять им **нельзя**.

Что может прислать клиент:

- Пустую строку: `{"title": ""}`
- `null`: `{"title": null}`
- Поле, которого ты не ждёшь: `{"title": "test", "evil": true}`
- Лишний мусор: `{"title": "test"<binary garbage>}`
- **Вообще не JSON**: `<html>привет</html>`
- **Пустое тело**

Твой обработчик должен **корректно обработать** каждый из этих случаев. Не упасть. Не вернуть 500. Вернуть 400 с понятным сообщением — «ты прислал то-то и то-то, что не так».

Это и называется **input validation**. Это огромная тема, и многие баги в продакшен-системах — из-за того, что разработчики не валидируют входные данные.

## Шаги валидации для создания задачи

Минимальная валидация для POST `/tasks`:

```go
func (s *Server) createTask(w http.ResponseWriter, r *http.Request) {
    var input struct {
        Title string `json:"title"`
    }

    err := json.NewDecoder(r.Body).Decode(&input)
    if err != nil {
        http.Error(w, "невалидный JSON: "+err.Error(), http.StatusBadRequest)
        return
    }

    // Валидация: title не должен быть пустым
    if strings.TrimSpace(input.Title) == "" {
        http.Error(w, "title обязателен и не может быть пустым", http.StatusBadRequest)
        return
    }

    // Создаём задачу
    task := s.tasks.Add(input.Title)

    // Возвращаем созданную задачу со статусом 201
    w.Header().Set("Content-Type", "application/json")
    w.WriteHeader(http.StatusCreated)
    json.NewEncoder(w).Encode(task)
}
```

Несколько важных моментов:

1. **Локальная анонимная структура** для приёма входа. Не используем `Task` напрямую, потому что:
   - `Task` имеет поле `ID`, которое **сервер должен присвоить сам**, а не клиент
   - `Task` имеет поле `Done`, которое при создании всегда `false`
   - У клиента может быть **другое** представление, чем у внутреннего хранилища

   Это называется **DTO** (Data Transfer Object) — отдельная структура для обмена с внешним миром. Очень распространённый паттерн.

2. **`strings.TrimSpace`** перед проверкой пустоты. Иначе клиент пришлёт `"   "` (пробелы) и пройдёт проверку.

3. **`w.WriteHeader(http.StatusCreated)`** — статус **201**, не 200. Это правильный статус для «успешно создал ресурс».

## Ядро 3: статус 201 Created

REST-соглашения по статусам для разных операций:

| Операция | Статус | Когда |
|----------|--------|-------|
| GET (получение) | 200 OK | Ресурс найден и возвращён |
| GET (нет ресурса) | 404 Not Found | Ресурса нет |
| POST (создание) | **201 Created** | Ресурс успешно создан |
| PUT (обновление) | 200 OK | Ресурс обновлён, в ответе обновлённое состояние |
| PUT (без тела ответа) | 204 No Content | Ресурс обновлён, тела нет |
| DELETE | 204 No Content | Ресурс удалён, тела нет |
| Любая ошибка валидации | 400 Bad Request | Клиент прислал плохие данные |
| Аутентификация не пройдена | 401 Unauthorized | Кто ты? |
| Прав не хватает | 403 Forbidden | Я знаю, кто ты, но не разрешу |
| Конфликт состояния | 409 Conflict | Например, попытка создать ресурс с уже существующим уникальным полем |
| Внутренняя ошибка | 500 Internal Server Error | Сломался сервер |

Эти соглашения — **не догма**, но если ты их соблюдаешь, твой API легко понять любому, кто знаком с REST. Если нарушаешь — клиенты будут путаться.

**Используй константы**: `http.StatusCreated`, `http.StatusNoContent`, и так далее. Не голые числа.

## Ядро 4: возврат созданного ресурса

Когда POST успешен, **возвращай созданный ресурс целиком** в теле ответа:

```json
HTTP/1.1 201 Created
Content-Type: application/json

{"id": 5, "title": "Купить молоко", "done": false}
```

Зачем? Потому что клиент **не знает** ID, который сервер присвоил. Если ты не вернёшь созданный объект, клиенту придётся делать второй запрос (`GET /tasks`) и догадываться, какая задача его. Это и неудобно, и чревато гонками.

Альтернативный подход — возвращать **только** заголовок `Location: /tasks/5` (где 5 — ID), без тела. Тоже валидно по REST, но менее удобно для клиента. Большинство современных API возвращают и заголовок Location, и тело.

## Метод Add возвращает Task

Чтобы это работало, метод `Add` должен возвращать созданную задачу:

```go
func (tl *TaskList) Add(title string) Task {
    task := Task{ID: tl.NextID, Title: title}
    tl.Tasks = append(tl.Tasks, task)
    tl.NextID++
    return task
}
```

## Регистрация обработчика

В `main`:

```go
http.HandleFunc("POST /tasks", server.createTask)
```

Заметь: путь `/tasks` тот же, что для GET. Это нормально — Go различает их по методу. Один и тот же URL, разные операции.

## Полный пример обработчика

```go
func (s *Server) createTask(w http.ResponseWriter, r *http.Request) {
    var input struct {
        Title string `json:"title"`
    }

    if err := json.NewDecoder(r.Body).Decode(&input); err != nil {
        http.Error(w, "невалидный JSON: "+err.Error(), http.StatusBadRequest)
        return
    }

    if strings.TrimSpace(input.Title) == "" {
        http.Error(w, "title обязателен и не может быть пустым", http.StatusBadRequest)
        return
    }

    task := s.tasks.Add(input.Title)

    w.Header().Set("Content-Type", "application/json")
    w.WriteHeader(http.StatusCreated)
    if err := json.NewEncoder(w).Encode(task); err != nil {
        // На этом этапе уже отправлен статус 201, поэтому изменить его нельзя.
        // Можно только записать в логи.
        log.Printf("ошибка кодирования ответа: %v", err)
    }
}
```

Заметь последний момент: после `WriteHeader(http.StatusCreated)` мы **уже не можем** изменить статус. Если `Encode` упадёт — что мы можем сделать? Только залогировать. Клиент уже получил 201, дальше — что есть, то есть.

В реальной жизни такие ошибки очень редки (обычно `Encode` падает только на странных типах). Но привычка логировать важна — без этого ты никогда не узнаешь, что что-то пошло не так.

## Расширенная валидация (опциональная)

Для более строгих API можно добавить проверки:

- **Длина**: `len(input.Title) > 200` → 400 «title слишком длинный»
- **Запрещённые символы**: например, не пускать управляющие символы
- **Дубликаты**: проверка, что задача с таким title уже не существует → 409 Conflict

Для нашего проекта достаточно проверки на пустоту. Но в продакшен-API валидация обычно гораздо строже, и под неё пишут отдельные библиотеки (`go-playground/validator` — самая популярная).
