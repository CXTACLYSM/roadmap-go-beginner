# Модуль 1. JSON — условие

## Задание

Написать **отдельную небольшую программу**, которая демонстрирует сериализацию и десериализацию структуры `Task` в JSON и обратно. Это **не интеграция в TaskList** — это полигон для отработки навыка JSON. Интеграцию сделаешь в модулях 2–3.

## Что должна делать программа

```
--- Одна задача в JSON ---
{
  "id": 1,
  "title": "Купить молоко",
  "done": false
}
--- Восстановлено: ---
ID=1 Title="Купить молоко" Done=false

--- Список задач в JSON ---
[
  {
    "id": 1,
    "title": "Купить молоко",
    "done": false
  },
  {
    "id": 2,
    "title": "Позвонить маме",
    "done": true
  },
  {
    "id": 3,
    "title": "Сделать домашку",
    "done": false
  }
]
--- Восстановлено: ---
ID=1 Title="Купить молоко" Done=false
ID=2 Title="Позвонить маме" Done=true
ID=3 Title="Сделать домашку" Done=false
```

## Требования

1. Создай **новую папку** `project-3-todo-file/practice-json/` для этого модуля. Не трогай код Проекта 2 — он понадобится в модуле 3 как основа.
2. В новой папке `go mod init json-practice`.
3. Тип `Task` с тремя полями (ID, Title, Done) и **JSON-тегами** на каждом поле.
4. Имена в JSON — **с маленькой буквы** (`id`, `title`, `done`).
5. Программа делает четыре действия:
   - Создаёт **одну** задачу
   - Сериализует её через `json.MarshalIndent` (с двумя пробелами отступа)
   - Десериализует обратно в новую переменную, выводит для проверки
   - То же самое для **слайса** из трёх задач
6. Все ошибки от `Marshal`/`Unmarshal` проверяются явно.

## Шаги

1. Создай папку и инициализируй модуль
2. Объяви `Task` с тегами
3. В `main` создай одну задачу, сериализуй, выведи результат
4. Десериализуй в `var t2 Task`, выведи через `Printf("%+v\n", t2)`
5. Создай слайс из трёх задач
6. Сделай то же самое для слайса
7. Запусти, проверь вывод

## Эксперименты

1. **Убери JSON-теги** со всех полей. Запусти. Что изменилось в выводе? Почему имена теперь с большой буквы?

2. **Сделай поле `Title` приватным** — переименуй в `title`. Что произойдёт со сериализацией? Что пропало из JSON и почему? (После эксперимента верни как было.)

3. **Добавь поле `Notes string` с тегом `json:"notes,omitempty"`**. Создай задачу без notes, посмотри JSON. Создай задачу с непустым notes — посмотри JSON. Что изменилось?

4. **Используй `Marshal` вместо `MarshalIndent`.** Сравни вывод. Как ты думаешь, какой формат лучше передавать по сети, а какой — сохранять в файл, который будет смотреть человек?

5. **Подсунь битый JSON в Unmarshal:**
   ```go
   bad := []byte(`{"id": 1, "title": "test"`)  // нет закрывающей скобки
   var t Task
   err := json.Unmarshal(bad, &t)
   fmt.Println("err:", err)
   ```
   Какое сообщение об ошибке?

6. **Подсунь JSON с неправильным типом:**
   ```go
   bad := []byte(`{"id": "abc", "title": "test", "done": false}`)
   ```
   Почему `"abc"` нельзя положить в `int ID`? Какое сообщение?

7. **Подсунь JSON с лишним полем:**
   ```go
   extra := []byte(`{"id": 1, "title": "test", "done": false, "priority": 5}`)
   ```
   Что произошло с полем `priority`? Появилась ли ошибка?

## Что показать Claude

Покажи Claude свой код и спроси: «зачем нужны теги, если без них тоже работает?». Claude предложит представить, что твой JSON будет принимать какой-нибудь Python или JavaScript бэкенд. Они ожидают `id`, а не `ID`. Это и есть главная причина — **совместимость с внешним миром**.

Запиши себе на будущее: **всегда** ставь JSON-теги. Даже если сейчас кажется, что не нужно — потом нужно будет, и переделывать **трудно**, потому что от форматов JSON могут зависеть другие системы.
