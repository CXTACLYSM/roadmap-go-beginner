# Модуль 4. PostgreSQL для бота — условие

## Задание

Заменить in-memory `Storage` (модуль 3) на `Repository` поверх PostgreSQL. Каждый пользователь видит **только свои** задачи. Данные **переживают перезапуск**.

## Что должно произойти

Сценарий с двумя пользователями (Левом и Аней):

```
Лев:    /add Купить молоко
Бот:    Задача #1 добавлена: Купить молоко

Аня:    /add Сделать домашку
Бот:    Задача #1 добавлена: Сделать домашку

Лев:    /list
Бот:    1. [ ] Купить молоко

Аня:    /list
Бот:    1. [ ] Сделать домашку
```

Заметь: **у обоих первая задача имеет id=1**. Это работает per-user `user_task_id`.

После **перезапуска бота**:

```
Лев:    /list
Бот:    1. [ ] Купить молоко
```

Данные на месте.

## Требования

1. Создана таблица `tasks` через миграцию:
   ```sql
   CREATE TABLE tasks (
       id           BIGSERIAL PRIMARY KEY,
       chat_id      BIGINT NOT NULL,
       user_task_id BIGINT NOT NULL,
       title        TEXT NOT NULL,
       done         BOOLEAN NOT NULL DEFAULT false,
       created_at   TIMESTAMPTZ NOT NULL DEFAULT now()
   );
   CREATE INDEX idx_tasks_chat_id ON tasks (chat_id);
   CREATE UNIQUE INDEX idx_tasks_chat_user_task_id ON tasks (chat_id, user_task_id);
   ```

2. Создан тип `Task` с полями `ID`, `ChatID`, `UserTaskID`, `Title`, `Done`

3. Создан тип `Repository` с полем `db *sql.DB`

4. Реализован конструктор `NewRepository(db *sql.DB) *Repository`

5. Реализованы методы:
   - `Add(chatID int64, title string) (Task, error)`
   - `List(chatID int64) ([]Task, error)`
   - `Done(chatID int64, userTaskID int64) error`
   - `Delete(chatID int64, userTaskID int64) error`

6. **Каждый запрос содержит `WHERE chat_id = $1`** (или `chat_id` в INSERT)

7. `Add` использует `COALESCE + MAX + 1` для вычисления `user_task_id`

8. `Add` использует `INSERT ... RETURNING ...` для возврата созданной задачи

9. `Done` и `Delete` используют `UPDATE/DELETE` + `RowsAffected` для обнаружения «не найдено»

10. `List` корректно работает с `defer rows.Close()` и `rows.Err()`

11. **Никакого `sync.Mutex`** в репозитории — БД сама управляет конкурентностью

12. Тип `Bot` обновлён — поле `repo *Repository` вместо `storage *Storage`

13. Все handle-функции обрабатывают ошибки от репозитория:
    - **«Не найдено»** → отправить сообщение пользователю
    - **Техническая ошибка** → лог + сообщение «попробуй позже»

14. Сообщения для пользователя содержат `task.UserTaskID`, не `task.ID`

15. В `main`:
    - Читает `BOT_TOKEN` и `DATABASE_URL` из env
    - Открывает БД через `sql.Open`
    - Настраивает пул
    - Делает `db.Ping`
    - Создаёт `Repository`, `Bot`, запускает `bot.Run()`

16. Все эндпоинты бота продолжают работать как в модуле 3

17. **Двух-пользовательский тест работает корректно**: разные пользователи видят разные данные

18. **После перезапуска бота данные на месте**

## Эксперименты

1. **Проверь, что у двух пользователей независимые id:** добавь по задаче с двух аккаунтов, сравни. У обоих должно быть `#1`. Это работа per-user `user_task_id`.

2. **Сделай SELECT через `psql`** напрямую:
   ```sql
   SELECT id, chat_id, user_task_id, title, done FROM tasks ORDER BY chat_id, user_task_id;
   ```
   Увидишь, как структурированы данные. Заметь: `id` глобальный (1, 2, 3...), `user_task_id` — per-chat (1, 1, 2, 2...).

3. **Симулируй потерю фильтра по chat_id:** временно убери `WHERE chat_id = $1` из метода `Done`:
   ```go
   r.db.Exec("UPDATE tasks SET done = true WHERE user_task_id = $1", userTaskID)
   ```
   Запусти бота. Если у двух пользователей есть задача `user_task_id = 1`, и Лев пишет `/done 1` — **у Ани тоже** задача станет выполненной. Проверь через `/list` Аней. Это **IDOR** в действии. Верни фильтр.

4. **Удали все задачи одного пользователя через psql:**
   ```sql
   DELETE FROM tasks WHERE chat_id = <твой chat_id>;
   ```
   `chat_id` найдёшь через `SELECT DISTINCT chat_id FROM tasks;`. После удаления `/list` в боте должен показать пустой список.

5. **Создай задачу через psql напрямую:**
   ```sql
   INSERT INTO tasks (chat_id, user_task_id, title)
   VALUES (<твой_chat_id>, 999, 'тестовая задача')
   RETURNING *;
   ```
   В боте сделай `/list` — должна появиться. Это показывает, что бот **читает** свежие данные из БД, а не из кеша.

6. **Воспроизведи race condition (если очень хочется):** одновременно с двух терминалов вызови `r.db.Exec("...")` для одного `chat_id`. С `COALESCE + MAX` теоретически возможно столкновение через UNIQUE constraint. Но для одного пользователя в Telegram это не воспроизводится — Telegram гарантирует последовательность.

7. **Замерь, сколько задач может быть у одного пользователя:** добавь скриптом 1000 задач (через INSERTs в psql или цикл в Go). Сделай `/list`. Что произошло? Telegram не пропустит сообщение длиннее 4096 символов. **Это не баг, это фича** — нужна пагинация в реальных ботах. Для модуля 4 — заметь и пройди дальше.

8. **Останови БД во время работы бота:**
   ```bash
   docker stop todo-pg
   ```
   Напиши боту команду. Что происходит? Бот логирует ошибку и отправляет «попробуй позже». **Не падает.** Запусти БД (`docker start todo-pg`), повтори команду — работает. Это resilience.

9. **Вернись к модулю 3 (in-memory) и сравни код.** Сколько строк в `Storage` (in-memory) vs `Repository` (БД)? У многих строк примерно одинаково, но **сложность ниже** — нет мьютексов, нет копирований, и многопроцессная безопасность из коробки.

## Что показать Claude

Покажи Claude свой `Add` метод и спроси: «правильно ли я обрабатываю случай, когда два запроса от **разных** пользователей одновременно вычисляют `MAX` для своих `chat_id`?». Это хороший вопрос:

- **Между разными `chat_id`** — никакого конфликта. Каждый `chat_id` — отдельное «пространство», `MAX` независимый.
- **Внутри одного `chat_id`** — теоретически возможен конфликт (если бы у нас был API, не Telegram). Telegram гарантирует последовательность, поэтому реальной проблемы нет.

И ещё спроси: «что если я добавлю **второго бота** на ту же таблицу — будут ли проблемы?». Ответ — нет, потому что бот идентифицируется не в этой таблице, а через токен. Один токен = один аккаунт = много пользователей. Если у тебя два разных бота с разными токенами и одной БД — они **тоже** будут работать корректно, потому что `chat_id` всё равно уникальны по всему Telegram.

Запиши себе на будущее: **multi-tenant дизайн** — это одна из главных тем backend архитектуры. Все большие SaaS-сервисы (Slack, Notion, Asana) построены вокруг этого принципа: одна БД, много изолированных пространств, фильтр по `tenant_id` или `org_id` в каждом запросе.
