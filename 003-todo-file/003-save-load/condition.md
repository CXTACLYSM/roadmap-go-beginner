# Модуль 3. Save/Load и интеграция в REPL — условие

## Задание

**Это интеграция.** Возьми код Проекта 2 (TaskList с REPL) и добавь к нему сохранение в файл `tasks.json`. Поведение всех существующих команд должно остаться **точно таким же**, плюс данные теперь переживают перезапуск программы.

## Что должна делать программа

Первый запуск (файла нет):

```
$ go run .
To-Do приложение. Команды: add, list, done, remove, exit
> add Купить молоко
Задача добавлена: #1 Купить молоко
> add Позвонить маме
Задача добавлена: #2 Позвонить маме
> done 1
Задача #1 отмечена как выполненная
> exit
До встречи!

$ ls
go.mod  main.go  tasks.json

$ cat tasks.json
{
  "tasks": [
    {
      "id": 1,
      "title": "Купить молоко",
      "done": true
    },
    {
      "id": 2,
      "title": "Позвонить маме",
      "done": false
    }
  ],
  "nextID": 3
}
```

Второй запуск:

```
$ go run .
To-Do приложение. Команды: add, list, done, remove, exit
> list
1. [x] Купить молоко
2. [ ] Позвонить маме
> add Сделать домашку
Задача добавлена: #3 Сделать домашку
> exit
До встречи!
```

Заметь: задачи **не пропали**, и `nextID` продолжается с **3**, а не с **1**.

## Требования

1. Создай **новую папку** `project-3-todo-file/app/` для интеграции. Скопируй туда код Проекта 2 (можешь упаковать архив или просто скопировать `main.go`).
2. **Сделай поля `TaskList` публичными:**
   ```go
   type TaskList struct {
       Tasks  []Task `json:"tasks"`
       NextID int    `json:"nextID"`
   }
   ```
   (И обнови все обращения внутри методов: `tl.tasks` → `tl.Tasks` и т.д.)
3. Тип `Task` тоже должен иметь JSON-теги (`id`, `title`, `done`).
4. Реализован метод `(tl *TaskList) Save(filename string) error`.
5. Реализован метод `(tl *TaskList) Load(filename string) error`:
   - Если файла нет — возвращает `nil`, оставляя список как есть
   - Если другая ошибка — возвращает обёрнутую ошибку
6. В `main`:
   - **Перед циклом** вызывается `list.Load("tasks.json")`
   - Если `Load` вернул ошибку — вывод в stderr и `os.Exit(1)`
   - **После каждой изменяющей команды** (`add`, `done`, `remove`) вызывается `Save`
   - После `list` сохранение **не вызывается** (она ничего не меняет)
   - Для устранения дублирования сохранения — используй helper-функцию `saveOrWarn`
7. Имя файла — константа в начале файла: `const dataFile = "tasks.json"`.
8. Все поведения из Проекта 2 продолжают работать.

## Шаги

1. Создай папку `app/`, инициализируй модуль (`go mod init todo`)
2. Перенеси код Проекта 2 в `main.go`
3. Сделай поля `TaskList` и `Task` публичными, добавь JSON-теги
4. Запусти программу — она должна работать как раньше (без сохранения)
5. Реализуй метод `Save`
6. В `main`, после `case "exit":`, добавь временный вызов `list.Save("tasks.json")` для теста
7. Запусти, добавь задачу, выйди, проверь, что файл создался
8. Реализуй метод `Load`
9. В начале `main` добавь вызов `Load`
10. Перезапусти программу, проверь, что задачи загрузились
11. Замени временный `Save` на `exit` на полную интеграцию: helper `saveOrWarn` плюс вызовы после каждой изменяющей команды
12. Полный прогон сценария: создание задач → exit → перезапуск → проверка

## Эксперименты

1. **Удали `tasks.json`** перед запуском. Программа должна стартовать без ошибок с пустым списком — это проверка обработки `os.ErrNotExist`.

2. **Сломай JSON в файле руками.** Открой `tasks.json`, удали закрывающую скобку. Запусти программу. Что произошло? Программа упала с понятной ошибкой? Это правильно — лучше явно ошибиться, чем затереть данные пустым списком.

3. **Сохраняй только при exit** (временно). Удали все вызовы `saveOrWarn` после команд, оставь только в `case "exit"`. Запусти, добавь 3 задачи. Прерви программу через **Ctrl+C** (вместо `exit`). Запусти снова — где задачи? Это и есть аргумент в пользу «сохранять после каждой команды».

4. **Прокачай команды с конкуренцией:** запусти **две копии** программы одновременно в одной папке. В первой добавь задачу, во второй добавь другую, в обеих сделай exit. Что в файле? Это — **race condition**: одна копия перезаписала изменения другой. Это серьёзная проблема, но решать её мы будем в Проекте 5 (БД), сейчас просто заметь, что она существует.

5. **Замени `saveOrWarn` на функции-значения** (если хочешь, опционально):
   ```go
   func runWithSave(list *TaskList, action func()) {
       action()
       saveOrWarn(list)
   }
   // в main:
   case "add":
       runWithSave(list, func() { handleAdd(list, arg) })
   ```
   Запусти. Работает? Нравится больше или меньше предыдущего варианта? Запиши свои ощущения — это вопрос вкуса, и оба варианта валидны.

6. **Добавь команду `clear`**, которая удаляет все задачи (`list.Tasks = nil` или через метод `Clear`). Не забудь вызвать `saveOrWarn` после неё. Перезапусти — пусто? Хорошо.

## Что показать Claude

Покажи Claude код метода `Load`. Спроси: «правильно ли я обрабатываю случай, когда файл существует, но JSON в нём битый? Что произойдёт с `tl.Tasks` после неудачного `Unmarshal`?». Это тонкий момент: `Unmarshal` может **частично** заполнить структуру, прежде чем встретить ошибку. В нашем случае это безопасно — мы возвращаем ошибку и `main` сразу выходит. Но в более сложных случаях (например, если ты решил «продолжить с тем, что распарсилось») — это превращается в сложный вопрос дизайна.

Запиши себе на будущее: **частичные сбои** — одна из главных тем в надёжных системах. В Проекте 7 ты увидишь это в полный рост на примере транзакций БД.
