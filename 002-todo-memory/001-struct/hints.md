# Модуль 1. Структура Task — подсказки

> ⚠️ Открывай только после 30 минут самостоятельной борьбы.

## Уровень 1: наводящие вопросы

1. С какого ключевого слова начинается объявление структуры?
2. Где в синтаксисе метода указывается, к какому типу он привязан?
3. Какая разница между `func (t Task) ...` и `func (t *Task) ...`?
4. Если метод **изменяет** структуру, какой из двух вариантов нужен?

## Уровень 2: про синтаксис struct

Если ты не помнишь форму:

```go
type ИмяТипа struct {
    Поле1 тип1
    Поле2 тип2
}
```

Например:

```go
type Task struct {
    ID    int
    Title string
    Done  bool
}
```

Заметь: внутри `struct { ... }` поля идут **на отдельных строках**, без запятых в конце. Это идиома Go. Запятые между полями работают, но так не пишут.

## Уровень 3: про синтаксис метода

Метод — это функция со специальным «получателем» перед именем:

```go
func (t Task) String() string {
    //   ^^^^^^ это получатель
    return "..."
}
```

`(t Task)` означает: «эта функция — метод типа `Task`, и внутри `t` будет ссылаться на конкретную задачу, для которой вызвали метод».

Аналогия из других языков: `t` — это аналог `this` или `self`. Только в Go ты явно даёшь ему имя сам.

## Уровень 4: про String() и форматирование

Самый удобный способ собрать строку с подстановками — `fmt.Sprintf`:

```go
return fmt.Sprintf("%d. %s %s", t.ID, status, t.Title)
```

`Sprintf` работает как `Printf`, но не печатает на экран, а **возвращает строку**. Глаголы:

- `%d` — целое число
- `%s` — строка
- `%v` — любое значение (универсальный)
- `%t` — bool
- `\n` — перевод строки

## Уровень 5: про условный статус

Тебе нужно вывести либо `[ ]`, либо `[x]` в зависимости от поля `Done`. Простой `if`:

```go
status := "[ ]"
if t.Done {
    status = "[x]"
}
```

Можешь сразу использовать `status` в `Sprintf`.

## Уровень 6: про указатель в receiver

Если ты написал так:

```go
func (t Task) MarkDone() {
    t.Done = true
}
```

— и ничего не работает, это **не баг**, это правильное поведение Go. Receiver-значение — это **копия**. Ты изменил копию, а оригинал в `main` не тронут.

Решение — звёздочка перед типом:

```go
func (t *Task) MarkDone() {
    t.Done = true
}
```

Теперь `t` — не копия, а **указатель на оригинал**. Изменения через `t.Done = true` затрагивают саму задачу.

Самое странное: вызов **выглядит так же**:

```go
task.MarkDone()  // Go сам подставит &task
```

Не нужно писать `(&task).MarkDone()`. Go умеет автоматически брать адрес от переменной, если метод требует указатель.

## Уровень 7: про эксперимент с позиционной инициализацией

Когда ты добавишь поле `Priority`, твой позиционный конструктор сломается:

```go
t := Task{2, "Тест", true}  // упс, теперь четыре поля, а здесь три
```

Компилятор скажет: `too few values in struct literal of type Task`. А именованный конструктор — продолжит работать, потому что он явно указывает, какое значение в какое поле:

```go
t := Task{ID: 2, Title: "Тест", Done: true}  // Priority получит 0 — нулевое значение
```

Это и есть **главная причина**, почему именованную инициализацию любят: код не ломается при добавлении новых полей. Запомни это правило: **именованная инициализация по умолчанию, позиционная — только для очень маленьких структур из 1–2 полей**.
