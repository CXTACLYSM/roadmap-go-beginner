# Модуль 2. Работа с файлами — теория

## Контекст

В модуле 1 ты научился превращать структуры в JSON-байты. Но эти байты пока живут только в памяти программы — как только программа завершается, они исчезают. Чтобы данные **пережили перезапуск**, их нужно записать на диск, в файл.

Работа с файлами — это, на первый взгляд, простая задача: «прочитать содержимое» и «записать содержимое». В Go для большинства случаев это действительно две функции. Но за ними стоит важная концепция: **файл — это внешний ресурс, и работа с ним всегда может пойти не так**.

Файла может **не быть**. Может **не быть прав** на чтение. Диск может быть **переполнен**. Кто-то может **удалить** файл прямо во время работы программы. Поэтому каждая файловая операция в Go возвращает `error`, и проверять его — не формальность.

## Ядро 1: чтение файла

Самая простая функция чтения:

```go
import "os"

data, err := os.ReadFile("tasks.json")
if err != nil {
    // что-то пошло не так
    return err
}
// data — это []byte с содержимым файла
```

`os.ReadFile` делает три вещи в одной функции:

1. Открывает файл для чтения
2. Читает **всё** содержимое в память
3. Закрывает файл

Это идеально для небольших файлов. Для очень больших (гигабайты) нужно потоковое чтение через `os.Open` + `bufio.Scanner` — но об этом в более продвинутых проектах. Для нашего to-do списка `ReadFile` — самый удобный выбор.

Возвращает `[]byte` — те самые байты, которые ты уже видел в модуле 1 как результат `Marshal`. То есть после `os.ReadFile` ты сразу можешь вызвать `json.Unmarshal` на этих данных.

## Ядро 2: запись файла

Симметрично:

```go
data := []byte(`{"id":1,"title":"test"}`)
err := os.WriteFile("tasks.json", data, 0644)
if err != nil {
    return err
}
```

`os.WriteFile` тоже делает три вещи:

1. Создаёт файл (или **полностью перезаписывает** существующий!)
2. Пишет в него все байты
3. Закрывает файл

**Важно: WriteFile перезаписывает файл целиком.** Если до этого в файле было 100 задач, а ты записал 1 — старые 99 пропали. Это и хорошо, и опасно: хорошо потому, что не нужно думать про «как обновить часть», опасно — потому что **нельзя случайно записать пустой файл**.

Третий аргумент — `0644` — это **права доступа** на файл. Что это и зачем — ниже.

## Что такое 0644 — права доступа

В Unix-системах (Linux, macOS) у каждого файла есть **права доступа** для трёх категорий:

- **Владелец** (тот, кто создал файл)
- **Группа** (группа пользователей, к которой принадлежит файл)
- **Все остальные**

И для каждой категории — три типа разрешений:

- **r** — чтение (read)
- **w** — запись (write)
- **x** — выполнение (execute, для программ)

`0644` — это компактная запись таких прав:

| Цифра | Категория | Биты | Что значит |
|-------|-----------|------|-------------|
| `6` | Владелец | `rw-` | Чтение + запись |
| `4` | Группа | `r--` | Только чтение |
| `4` | Все остальные | `r--` | Только чтение |

Цифры — это сумма: `r=4`, `w=2`, `x=1`. `6 = 4+2 = rw`, `5 = 4+1 = rx`, `7 = 4+2+1 = rwx`.

Ноль в начале — это просто обозначение восьмеричного числа в Go (как `0x` для шестнадцатеричных).

**Для обычных файлов с данными `0644` — стандартный выбор.** Владелец может читать и менять, остальные могут только читать. Это разумная безопасность по умолчанию.

Если файл должен быть приватным (например, содержит пароли) — `0600`: владелец может всё, остальные ничего.

На Windows права доступа работают по-другому, но Go-программа примет это значение и сделает что-то осмысленное.

## Ядро 3: что делать, если файла нет

**Это самый важный концептуальный момент модуля.**

При первом запуске твоей программы файл `tasks.json` ещё не существует. Если ты просто вызовешь `os.ReadFile("tasks.json")` — получишь ошибку:

```
open tasks.json: no such file or directory
```

Что это? **Это ошибка или нет?**

Ответ: **зависит от контекста**. Для нашей программы — это **нормальная ситуация**. «Файла нет» означает «список задач пустой», и программа должна спокойно стартовать с пустым слайсом, а не падать.

Но `ReadFile` об этом не знает. Он возвращает ошибку для **любой** причины, по которой не смог прочитать файл. Это может быть «файла нет», а может быть «нет прав» или «диск сломан». Эти случаи **разные**, и обрабатывать их надо по-разному.

## errors.Is — проверка типа ошибки

Чтобы отличить «файла нет» от других ошибок, в Go есть функция `errors.Is`:

```go
import (
    "errors"
    "os"
)

data, err := os.ReadFile("tasks.json")
if err != nil {
    if errors.Is(err, os.ErrNotExist) {
        // файла нет — нормально, начинаем с пустого списка
        return []Task{}, nil
    }
    // другая ошибка — это серьёзно, возвращаем её
    return nil, err
}
```

**`errors.Is(err, target)`** — спрашивает: «эта ошибка `err` — это (или содержит в себе) ошибку типа `target`?». Возвращает `bool`.

**`os.ErrNotExist`** — стандартная переменная, представляющая ошибку «файл не существует». Аналогично есть `os.ErrPermission` (нет прав), `io.EOF` (конец файла) и другие.

Этот паттерн — один из самых частых в Go-коде. Привыкай: «получил ошибку — посмотри, какая именно, и реши, как обработать».

## Атомарность записи (упоминание)

Если твоя программа упадёт **посередине** записи в файл — что произойдёт? Файл может оказаться в **частично записанном** состоянии: половина старых данных, половина новых, между ними мусор.

Это называется **проблема атомарности**, и для нашего проекта она не критична — пользователь просто перезапустит программу. Но в продакшен-коде её решают через паттерн «записать в **временный** файл, потом **переименовать** в целевой». Переименование — атомарная операция: либо старый файл, либо новый, никаких промежуточных состояний.

Запоминай это название (atomic file write) и идею. В Проекте 7 ты можешь захотеть применить это для важных данных.

## Полный пример

```go
package main

import (
    "encoding/json"
    "errors"
    "fmt"
    "os"
)

type Task struct {
    ID    int    `json:"id"`
    Title string `json:"title"`
    Done  bool   `json:"done"`
}

func saveTasks(filename string, tasks []Task) error {
    data, err := json.MarshalIndent(tasks, "", "  ")
    if err != nil {
        return fmt.Errorf("ошибка сериализации: %w", err)
    }

    err = os.WriteFile(filename, data, 0644)
    if err != nil {
        return fmt.Errorf("ошибка записи файла: %w", err)
    }

    return nil
}

func loadTasks(filename string) ([]Task, error) {
    data, err := os.ReadFile(filename)
    if err != nil {
        if errors.Is(err, os.ErrNotExist) {
            // файла нет — это нормально, возвращаем пустой список
            return []Task{}, nil
        }
        return nil, fmt.Errorf("ошибка чтения файла: %w", err)
    }

    var tasks []Task
    err = json.Unmarshal(data, &tasks)
    if err != nil {
        return nil, fmt.Errorf("ошибка парсинга JSON: %w", err)
    }

    return tasks, nil
}

func main() {
    // Сохраняем
    tasks := []Task{
        {ID: 1, Title: "первая", Done: false},
        {ID: 2, Title: "вторая", Done: true},
    }
    if err := saveTasks("test.json", tasks); err != nil {
        fmt.Println("Ошибка сохранения:", err)
        return
    }
    fmt.Println("Сохранено")

    // Загружаем
    loaded, err := loadTasks("test.json")
    if err != nil {
        fmt.Println("Ошибка загрузки:", err)
        return
    }
    fmt.Printf("Загружено %d задач\n", len(loaded))
    for _, t := range loaded {
        fmt.Printf("  %+v\n", t)
    }

    // Загружаем несуществующий файл
    empty, err := loadTasks("nonexistent.json")
    if err != nil {
        fmt.Println("Ошибка:", err)
    } else {
        fmt.Printf("Загружено %d задач из несуществующего файла\n", len(empty))
    }
}
```

Несколько новых идей в этом коде:

- **`fmt.Errorf("...: %w", err)`** — это обёртывание ошибки. `%w` (от **w**rap) сохраняет оригинальную ошибку **внутри** новой, при этом добавляя контекст. После такого можно использовать `errors.Is` на новой ошибке, и она найдёт оригинал. Очень полезный паттерн для слоистых программ.
- **Функции `saveTasks` и `loadTasks`** возвращают понятные ошибки с контекстом, а не голые ошибки от `os` или `json`. Это правильно: вызывающий код должен видеть «ошибка сохранения задач», а не «open: no such file».
- **`loadTasks` для несуществующего файла возвращает пустой слайс и `nil` ошибку.** Это сознательный дизайн: «нет файла» — это не ошибка, это «пустые данные». Если завтра ты захочешь сделать это **ошибкой** — поменяешь только эту одну функцию, остальной код не тронешь.
