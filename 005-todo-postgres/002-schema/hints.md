## Уровень 2: про команду psql -f

```bash
psql -h localhost -U postgres -d postgres -f migrations/001_create_tasks_table.sql
```

`-f` (от **f**ile) говорит psql «выполни SQL из этого файла». В файле может быть одна команда или много — psql выполнит их по порядку.

Если пароль каждый раз запрашивается — установи переменную окружения:

```bash
export PGPASSWORD=secret
```

После этого psql не будет спрашивать пароль до конца сессии терминала.

## Уровень 3: про BIGSERIAL vs SERIAL

```sql
id BIGSERIAL PRIMARY KEY    -- 64 бита, 9 квинтиллионов значений
id SERIAL PRIMARY KEY        -- 32 бита, ~2 миллиарда значений
```

Используй **`BIGSERIAL`** по умолчанию. Это правило современных проектов. `SERIAL` теоретически может закончиться — у некоторых компаний это случалось в продакшене.

## Уровень 4: про DEFAULT now()

```sql
created_at TIMESTAMPTZ NOT NULL DEFAULT now()
```

`now()` — встроенная функция PostgreSQL, возвращает текущее время с часовым поясом сервера.

`DEFAULT now()` означает: «если при INSERT не указано значение для этой колонки — записать результат вызова `now()`». Каждая новая строка получает **текущее на момент INSERT** время, а не время создания таблицы.

Альтернатива — `CURRENT_TIMESTAMP`, означает то же самое.

## Уровень 5: про порядок применения

Всегда применяй миграции **по порядку номеров**:

```bash
psql ... -f migrations/001_*.sql
psql ... -f migrations/002_*.sql
psql ... -f migrations/003_*.sql
```

Можно через bash:

```bash
for file in migrations/*.sql; do
    psql -h localhost -U postgres -d postgres -f "$file"
done
```

Это применит **все** файлы в алфавитном порядке. Поэтому важна **трёхзначная нумерация**: `001`, `002`, `003`. Если ты напишешь `1`, `2`, `10` — алфавитный порядок будет `1, 10, 2`, и всё сломается.

## Уровень 6: про CREATE TABLE и его части

Полный синтаксис:

```sql
CREATE TABLE имя_таблицы (
    имя_колонки тип [CONSTRAINTS],
    имя_колонки тип [CONSTRAINTS],
    ...
    [ограничения уровня таблицы]
);
```

Ограничения колонки (constraints):
- `PRIMARY KEY` — первичный ключ
- `NOT NULL` — обязательная
- `UNIQUE` — уникальная
- `DEFAULT значение` — значение по умолчанию
- `REFERENCES другая_таблица(колонка)` — foreign key
- `CHECK (выражение)` — проверка

Пример с несколькими constraints:

```sql
title TEXT NOT NULL CHECK (length(title) > 0)
```

— колонка `title`, обязательная, и нельзя пустую строку.

Это **полезно** для надёжности, но в нашем учебном проекте обходимся базовыми constraints.

## Уровень 7: про CREATE INDEX

```sql
CREATE INDEX [имя] ON таблица (колонка);
CREATE INDEX [имя] ON таблица (колонка1, колонка2);  -- composite index
CREATE UNIQUE INDEX ... -- уникальный индекс
```

Несколько правил для имени индекса:

- По соглашению: `idx_<table>_<columns>`
- Например: `idx_tasks_done`, `idx_users_email`, `idx_orders_user_id_created_at`
- PostgreSQL может сгенерировать имя сам (если опустить), но явное лучше для отладки

Для нашего проекта:

```sql
CREATE INDEX idx_tasks_done ON tasks (done);
```

## Уровень 8: про IF NOT EXISTS

Если ты хочешь, чтобы миграция была **безопасна для повторного запуска**:

```sql
CREATE TABLE IF NOT EXISTS tasks (...);
CREATE INDEX IF NOT EXISTS idx_tasks_done ON tasks (done);
```

— миграция не упадёт, если объект уже есть.

**Это уловка**, и я не рекомендую её в учебных целях. Лучше учиться **не запускать миграции дважды** через нормальную систему отслеживания. В продакшене инструменты (`golang-migrate`) делают это автоматически.

Если используешь — будь готов, что в будущем тебе придётся делать `ALTER`, и `IF NOT EXISTS` там не сработает («ALTER не идемпотентен»). Так что это решает только часть проблемы.

## Уровень 9: про эволюцию схемы

Когда схема меняется в существующей БД (где уже есть данные) — это не «удалить и пересоздать». Это **последовательность ALTER**:

```sql
-- migrations/003_add_priority.sql
ALTER TABLE tasks ADD COLUMN priority INTEGER NOT NULL DEFAULT 0;
```

Что произойдёт:
- Колонка добавляется
- Все существующие строки получают `priority = 0` (благодаря DEFAULT)
- Новые INSERT тоже могут не указывать `priority` — будет 0

Если бы у нас не было `DEFAULT`:

```sql
ALTER TABLE tasks ADD COLUMN priority INTEGER NOT NULL;
```

— упадёт с ошибкой, потому что существующие строки не имеют значения. Это пример «миграции, которая ломается на существующих данных».

## Если совсем тупик

Скажи Claude конкретно: «модуль 2 проекта 5, [конкретная проблема]». Самые частые ситуации:

- **«ERROR: relation "tasks" already exists»** → миграция уже была применена. Удали и пересоздай volume.
- **«psql: connection failed»** → контейнер не запущен или порт занят. См. подсказки модуля 1.
- **«syntax error at or near ..."** → опечатка в SQL. Проверь скобки и `;`.
- **«column does not exist»** → ты ссылаешься на колонку, которой нет в таблице. Проверь `\d tasks`.
