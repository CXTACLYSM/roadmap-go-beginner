# Модуль 6. Graceful shutdown — чек-лист готовности

## Поведенческие маркеры

- [ ] В `main` создан `ctx, cancel := context.WithCancel(context.Background())`
- [ ] Сразу после — `defer cancel()`
- [ ] Создан канал `sigChan := make(chan os.Signal, 1)` с буфером 1
- [ ] Зарегистрированы оба сигнала: `signal.Notify(sigChan, os.Interrupt, syscall.SIGTERM)`
- [ ] Запущена горутина, которая ждёт `<-sigChan` и вызывает `cancel()`
- [ ] `Bot.Run(ctx context.Context)` принимает контекст
- [ ] В цикле `Run` есть проверка `ctx.Done()` через `select` в начале каждой итерации
- [ ] Реализованы `*Context` версии методов клиента: `GetMeContext`, `GetUpdatesContext`, `SendMessageContext`
- [ ] HTTP-запросы используют `http.NewRequestWithContext` + `client.Do(req)`
- [ ] Реализованы `*Context` версии методов репозитория
- [ ] В репозитории используются `QueryRowContext`, `QueryContext`, `ExecContext`
- [ ] Контекст пробрасывается через `handleMessage`, `handleCommand`, `handleStateful`, все `handle*` функции
- [ ] Retry в `Run` использует `select + time.After`, не `time.Sleep`
- [ ] После выхода из `Run` в `main` логируется «Бот остановлен корректно»
- [ ] `defer db.Close()` присутствует
- [ ] **Ctrl+C завершает бота быстро** (за миллисекунды, не за 30 секунд)
- [ ] **`echo $?` после Ctrl+C показывает 0**
- [ ] Все команды модулей 3-5 продолжают работать
- [ ] Двух-пользовательский тест проходит
- [ ] Код закоммичен

Это был **переломный по парадигме** проект. Ты построил долгоживущий процесс, который:

- Подключается к **внешнему API** через HTTP (Telegram Bot API)

- Корректно завершается через **graceful shutdown**

Что у тебя теперь есть:

- Понимание ботов «изнутри», без магии библиотек
- Опыт работы с **HTTP как клиентом**, а не сервером
- Знание паттерна **long polling**
- Свой минимальный «telegram-bot-api» — тонкая обёртка над raw HTTP
- Опыт **multi-tenant дизайна** с фильтрацией по владельцу
- Понимание **IDOR** и как от него защищаться
- Практика с **finite state machines** для диалогов
- Уверенное владение **`context.Context`** для отмены и таймаутов
- Реализация **graceful shutdown** через сигналы и контексты

Самые важные мысли проекта:

1. **Бот — это HTTP-клиент.** Никакой магии. Когда ты это понимаешь, любой внешний API становится простым: HTTP + JSON.

2. **Долгоживущий процесс = другая дисциплина.** Он должен переживать сетевые ошибки, не утекать памятью, корректно завершаться. Это не «сложнее», это **другой mindset**.

3. **State per user** — главная мысль multi-tenant систем. Одна таблица, фильтр в каждом запросе. Это масштабируется на любые SaaS-сервисы.

4. **FSM — простой, но мощный паттерн.** В чистом Go он реализуется в 30 строк, и решает задачи, для которых в других системах подключают тяжёлые библиотеки.

5. **Контексты — это не «сложная фишка»**, это **стандартная практика** Go. Каждая функция, делающая I/O, должна принимать контекст. Это — часть «бэкендера, который пишет на Go».

6. **Graceful shutdown — обязательная часть** production-сервиса. Без него ты теряешь данные при каждом перезапуске.
