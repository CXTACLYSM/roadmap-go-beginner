# Проект 3. To-Do лист с сохранением в файл

**Длительность:** 2 недели
**Цель:** научиться сохранять состояние программы между запусками. Освоить работу с файлами, JSON-сериализацию, и впервые разложить код по нескольким файлам.

## Что в итоге будет уметь программа

То же самое, что в Проекте 2 (`add`, `list`, `done`, `remove`, `exit`), **плюс**:

- При запуске **загружает** список задач из файла `tasks.json`
- После любого изменения **сохраняет** обновлённый список в тот же файл
- При первом запуске (когда файла ещё нет) — стартует с пустым списком, не падает
- Код разложен по **нескольким файлам** в одном пакете

```
$ go run .
> add Купить молоко
Задача добавлена: #1 Купить молоко
> add Позвонить маме
Задача добавлена: #2 Позвонить маме
> exit
До встречи!

$ ls
go.mod  main.go  task.go  storage.go  tasks.json

$ cat tasks.json
{
  "tasks": [
    {"id": 1, "title": "Купить молоко", "done": false},
    {"id": 2, "title": "Позвонить маме", "done": false}
  ],
  "nextID": 3
}

$ go run .
> list
1. [ ] Купить молоко
2. [ ] Позвонить маме
> exit
```

Заметь главное: между двумя запусками задачи **не пропали**. Это первый момент, когда твоя программа становится по-настоящему **полезной**.

## Что нового по сравнению с Проектами 1 и 2

- **JSON-сериализация** — превращение Go-структур в текст и обратно. Это тот же навык, что используется во **всех** REST-API. По сути, ты сейчас тренируешь то, что в Проекте 4 будет фундаментом веб-сервера.
- **Файловый ввод-вывод** — `os.ReadFile`, `os.WriteFile`. Простые функции, но за ними стоит важная концепция: **внешние ресурсы ненадёжны**, и работа с ними всегда требует обработки ошибок.
- **Персистентность** — данные переживают перезапуск программы. Пока это файл, в проекте 5 будет PostgreSQL, но идея та же: «состояние живёт где-то снаружи программы».
- **Multi-file проект** — твой код больше не помещается в один файл, и это нормально. В Go это очень просто: всё в одном пакете, файлы видят друг друга автоматически. Никаких «include» и «import» для своих файлов.

## Что нового **не** будет

- Пакеты (`package storage`, `package task`) — это придёт позже. Сейчас всё в `package main`.
- База данных — это Проект 5.
- HTTP — это Проект 4.
- Шифрование, права доступа в духе Linux — нет.

Сознательно держим объём управляемым.

## Главная новая идея проекта

В Проектах 1 и 2 твоя программа была **самодостаточна**. Все данные жили внутри неё. После этого проекта ты впервые столкнёшься с понятием **внешнего ресурса**: файл — это что-то **снаружи** программы, чем программа не управляет полностью.

Файл может **отсутствовать** (первый запуск). Может **существовать**, но быть **повреждённым** (кто-то его руками отредактировал и сломал). Может быть **недоступен** для чтения (нет прав). Может **исчезнуть** между двумя вызовами (кто-то его удалил параллельно).

Это требует другого мышления. Каждое обращение к файлу — это **попытка**, которая может не получиться. Поэтому почти каждая операция возвращает `error`, и проверять его — не формальность, а необходимость.

В Проекте 4 (HTTP) и в Проекте 5 (база данных) это будет нормой жизни. Привыкай сейчас.
