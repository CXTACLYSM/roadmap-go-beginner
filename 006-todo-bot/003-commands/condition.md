# Модуль 3. Команды — условие

## Задание

Превратить эхо-бота в **функционально полный to-do бот**: команды `/start`, `/help`, `/list`, `/add`, `/done`, `/delete`. Данные хранятся в памяти, **per-user** через `chat_id`. Без БД пока (это модуль 4).

## Что должно происходить

```
Ты:    /start
Бот:   Привет! Я твой to-do бот. Команды:
       /list — показать задачи
       /add <текст> — добавить задачу
       /done <id> — отметить выполненной
       /delete <id> — удалить
       /help — справка

Ты:    /list
Бот:   Список пуст. Добавь задачу через /add

Ты:    /add Купить молоко
Бот:   Задача #1 добавлена: Купить молоко

Ты:    /add Позвонить маме
Бот:   Задача #2 добавлена: Позвонить маме

Ты:    /list
Бот:   1. [ ] Купить молоко
       2. [ ] Позвонить маме

Ты:    /done 1
Бот:   Задача #1 отмечена выполненной

Ты:    /list
Бот:   1. [x] Купить молоко
       2. [ ] Позвонить маме

Ты:    /done abc
Бот:   Использование: /done <id>

Ты:    /done 999
Бот:   задача не найдена: 999

Ты:    /delete 2
Бот:   Задача #2 удалена

Ты:    /add
Бот:   Использование: /add <текст задачи>

Ты:    привет
Бот:   Не понимаю команду. Используй /help для справки.
```

## Требования

1. Создан тип `Task` с полями `ID int64`, `Title string`, `Done bool`
2. Создан тип `Storage` с `tasksByChat map[int64][]Task`, `nextIDByChat map[int64]int64`, `mu sync.Mutex`
3. Реализован конструктор `NewStorage()`
4. Реализованы методы `Storage`:
   - `Add(chatID int64, title string) Task`
   - `List(chatID int64) []Task` — возвращает **копию** слайса
   - `Done(chatID int64, id int64) error`
   - `Delete(chatID int64, id int64) error`
5. **Все методы** `Storage` используют `Lock + defer Unlock`
6. Создан тип `Bot` со ссылками на `*TelegramClient` и `*Storage`
7. Реализован метод `(b *Bot) Run()` — главный цикл из модуля 2, плюс новая `handleMessage`
8. Реализован метод `(b *Bot) handleMessage(msg *Message)`:
   - Парсит команду через `strings.SplitN(text, " ", 2)`
   - Обрезает `@bot_name` если есть
   - Делает `switch` по команде
9. Реализованы все handle-функции:
   - `handleStart`, `handleHelp` — приветствие со списком команд
   - `handleList` — список задач или «Список пуст»
   - `handleAdd` — валидация пустого аргумента, добавление, ответ
   - `handleDone` — парсинг id, обработка ошибок, отметка
   - `handleDelete` — парсинг id, обработка ошибок, удаление
   - `handleUnknown` — для всего остального
10. Helper `parseTaskID(arg string) (int64, error)` для парсинга
11. Helper `(b *Bot) send(chatID, text)` — обёртка с логированием ошибки
12. Каждый пользователь видит **только свои** задачи
13. **Каждый пользователь имеет свои id** задач (1, 2, 3..., у каждого своё)
14. **Бот реально работает** — попробовать в Telegram с двух разных аккаунтов (если есть второй) или со своего

## Шаги

## Эксперименты

1. **Без префикса `/`:** напиши `add Купить молоко` (без слэша). Что произошло? Бот ответит «Не понимаю команду». Это потому, что мы делаем switch только по командам с `/`.

2. **С `@bot_name`:** напиши `/list@lev_todo_bot`. Должно сработать (благодаря обрезке `@`). Это важно для групповых чатов, где разные боты могут иметь одинаковые команды.

3. **Регистр:** напиши `/LIST` (большими буквами). Сработает или нет? Зависит от того, добавил ли ты `strings.ToLower`. Если нет — добавь. Стандартное соглашение: команды нечувствительны к регистру.

4. **Длинный список:** добавь 50 задач циклом (можешь руками или скриптом). Сделай `/list`. Что произошло?
   - Если меньше 4096 символов — отображается одним сообщением
   - Если больше — Telegram не примет (вернёт ошибку «message is too long»)
   - **В реальном боте** надо разбивать на несколько сообщений или делать пагинацию (`/list 1`, `/list 2`)
   - Для модуля 3 — просто заметь проблему, решать её не надо

5. **Эмодзи в задаче:** `/add 🛒 Купить молоко`. Должно работать — UTF-8.

6. **Многострочный текст:** через мобильный Telegram отправь `/add` с переводом строки внутри (Shift+Enter в десктопе). Что произошло? Парсер `SplitN(..., 2)` не разбирает переводы строк, так что текст с переводами должен сохраниться целиком.

7. **Команда без аргумента:** `/done` (без id). Бот ответит «Использование: /done <id>». Это валидация «обязательный аргумент отсутствует».

8. **Проверка изоляции пользователей:** напиши с одного аккаунта `/add Задача 1`, с другого — `/add Задача A`. Сделай `/list` с обоих — каждый должен видеть только свою.

9. **Ошибка отправки:** временно сломай токен в `bot.send` (или отправь `chatID = 0`). Бот **не должен падать** — должен залогировать и продолжить. Это проверка resilience.

## Что показать Claude

Покажи Claude свой `handleMessage` и спроси: «правильно ли я разделяю парсинг и обработку? Что бы я улучшил, если бы команд было 50, а не 6?». Это вопрос **масштабирования архитектуры**:

- **Простой подход (модуль 3)** — `switch` по строке. Работает для малого числа команд.
- **Лучше для большего числа** — `map[string]func(*Bot, int64, string)` — таблица команд. Можно регистрировать команды по одной, можно динамически добавлять.
- **Production** — отдельный объект `Router` с middleware (логирование, метрики, авторизация). По сути — то же самое, что `http.ServeMux`, только для Telegram-команд.

Для модуля 3 `switch` достаточно. Понимать варианты — обязательно.

Запиши себе на будущее: **архитектура должна соответствовать масштабу**. Применять «map с регистрацией» к 6 командам — оверинжиниринг. Но к 60 — уже необходимо.
