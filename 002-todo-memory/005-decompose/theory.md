# Модуль 5. Декомпозиция и обработка ошибок — теория

## Контекст

После модуля 4 у тебя есть рабочее приложение. Оно умеет всё, что должно. Но `main` превратился в монстра на 150+ строк, в котором:

- Бесконечный цикл REPL
- Парсинг команды
- Шесть веток switch
- В каждой ветке — проверки, обработка ошибок, работа со слайсом
- Дублирование между `done` и `remove`

Этот код **работает**, но он **плохо устроен**. Если тебе завтра скажут «добавь команду `priority <ID> <число>`», ты будешь долго искать, куда её вставить, и ещё дольше — копировать общие проверки.

Этот модуль — **рефакторинг**. Ты не добавляешь нового функционала. Ты **переписываешь** существующий код так, чтобы он стал чище, понятнее и расширяемее.

Это критически важный навык. В реальной работе ты будешь рефакторить **больше**, чем писать с нуля. Уметь видеть, что код «созрел для рефакторинга», и аккуратно переделывать его без потери функциональности — это сеньорский навык, и тренировать его надо с самого начала.

## Ядро 1: что такое хорошая функция

Хорошая функция:

1. **Делает одну вещь.** Если ты говоришь «эта функция парсит ID **и** ищет задачу **и** меняет статус **и** выводит результат» — это четыре функции в одной.
2. **Имеет понятное имя.** По имени должно быть ясно, что функция делает. `processCmd` — плохо. `markTaskDone` — хорошо.
3. **Имеет минимум зависимостей.** Чем меньше параметров и чем меньше она знает о внешнем мире — тем проще её переиспользовать и тестировать.
4. **Возвращает осмысленный результат.** Либо данные, либо ошибку, либо `(данные, ошибка)`.
5. **Короткая.** Жёсткого предела нет, но «больше 30 строк — повод задуматься» — рабочее правило.

## Ядро 2: тип TaskList — структура с поведением

Сейчас у тебя слайс `tasks` и счётчик `nextID` — две **отдельные** переменные. Они логически связаны (это «список задач со счётчиком»), но в коде живут отдельно.

Идиоматичный Go-подход — **сгруппировать их в структуру**:

```go
type TaskList struct {
    tasks  []Task
    nextID int
}
```

Теперь это **одна** сущность с понятным именем и состоянием.

И сразу можно сделать на ней методы:

```go
func (tl *TaskList) Add(title string) Task {
    task := Task{ID: tl.nextID, Title: title}
    tl.tasks = append(tl.tasks, task)
    tl.nextID++
    return task
}

func (tl *TaskList) MarkDone(id int) error {
    i := tl.findIndex(id)
    if i == -1 {
        return fmt.Errorf("задача не найдена: %d", id)
    }
    tl.tasks[i].Done = true
    return nil
}

func (tl *TaskList) Remove(id int) error {
    i := tl.findIndex(id)
    if i == -1 {
        return fmt.Errorf("задача не найдена: %d", id)
    }
    tl.tasks = append(tl.tasks[:i], tl.tasks[i+1:]...)
    return nil
}

func (tl *TaskList) findIndex(id int) int {
    for i, t := range tl.tasks {
        if t.ID == id {
            return i
        }
    }
    return -1
}

func (tl *TaskList) All() []Task {
    return tl.tasks
}
```

Заметь несколько вещей:

1. **Все receiver-ы — указатели** (`*TaskList`). Потому что почти все методы изменяют состояние списка. Даже `findIndex`, который только читает, тоже использует указатель — для **единообразия**. Считается хорошим тоном: либо все методы типа с указателем, либо все с значением. Смешение смущает читателя.

2. **`findIndex` с маленькой буквы** — это **приватный** метод. Снаружи пакета его не вижут (с буквой пока всё в одном пакете, но привычка важна — это вспомогательная функция, она не должна светиться наружу).

3. **`MarkDone` и `Remove` возвращают `error`.** Если задачи нет — это ошибка, и пусть вызывающий код сам решает, что с ней делать. Сама функция `MarkDone` не должна знать про `fmt.Println` — это не её ответственность.

4. **`Add` возвращает созданную задачу.** Чтобы вызывающий мог сразу её показать (например, вывести подтверждение).

5. **`All()` возвращает слайс задач.** Это нужно для `list`. Можно было бы просто экспортировать поле `Tasks`, но через метод гибче — потом можно добавить фильтрацию, сортировку и так далее.

## Ядро 3: `main` как тонкий координатор

После выноса методов `main` становится **существенно проще**:

```go
func main() {
    list := &TaskList{nextID: 1}
    scanner := bufio.NewScanner(os.Stdin)

    fmt.Println("To-Do приложение. Команды: add, list, done, remove, exit")

    for {
        fmt.Print("> ")
        if !scanner.Scan() {
            break
        }
        line := strings.TrimSpace(scanner.Text())
        if line == "" {
            continue
        }

        command, arg := parseCommand(line)

        switch command {
        case "add":
            handleAdd(list, arg)
        case "list":
            handleList(list)
        case "done":
            handleDone(list, arg)
        case "remove":
            handleRemove(list, arg)
        case "exit":
            fmt.Println("До встречи!")
            return
        default:
            fmt.Println("Неизвестная команда:", command)
        }
    }
}
```

Каждый `case` — это **один вызов**. Вся логика спрятана в обработчиках:

```go
func parseCommand(line string) (string, string) {
    parts := strings.SplitN(line, " ", 2)
    if len(parts) == 1 {
        return parts[0], ""
    }
    return parts[0], parts[1]
}

func handleAdd(list *TaskList, arg string) {
    if arg == "" {
        fmt.Println("Использование: add <текст задачи>")
        return
    }
    task := list.Add(arg)
    fmt.Printf("Задача добавлена: #%d %s\n", task.ID, task.Title)
}

func handleList(list *TaskList) {
    tasks := list.All()
    if len(tasks) == 0 {
        fmt.Println("Список пуст")
        return
    }
    for _, t := range tasks {
        fmt.Println(t.String())
    }
}

func handleDone(list *TaskList, arg string) {
    id, err := parseID(arg, "done")
    if err != nil {
        fmt.Println(err)
        return
    }
    if err := list.MarkDone(id); err != nil {
        fmt.Println(err)
        return
    }
    fmt.Printf("Задача #%d отмечена как выполненная\n", id)
}

func handleRemove(list *TaskList, arg string) {
    id, err := parseID(arg, "remove")
    if err != nil {
        fmt.Println(err)
        return
    }
    if err := list.Remove(id); err != nil {
        fmt.Println(err)
        return
    }
    fmt.Printf("Задача #%d удалена\n", id)
}

func parseID(arg, command string) (int, error) {
    if arg == "" {
        return 0, fmt.Errorf("использование: %s <ID>", command)
    }
    id, err := strconv.Atoi(arg)
    if err != nil {
        return 0, fmt.Errorf("ID должен быть числом: %s", arg)
    }
    return id, nil
}
```

Что мы получили:

- `parseID` — общая функция для двух команд (`done` и `remove`), которая раньше дублировалась
- Каждая команда живёт в своём `handleXxx`
- `main` стал тонким — он только координирует
- Логика работы со списком отделена от логики ввода-вывода

## Ядро 4: разделение слоёв

Заметь невидимое разделение, которое мы только что сделали:

| Слой | Кто | Что делает |
|------|-----|-------------|
| **Domain** | `Task`, `TaskList` и их методы | Бизнес-логика. Знает, что такое задача и как с ней работать. **Не знает** про терминал. |
| **Handlers** | `handleAdd`, `handleList` и т.д. | Обработка команд. Знает, как вызвать domain и как показать результат пользователю. |
| **Main loop** | `main` с REPL | Чтение ввода, парсинг, диспетчеризация. Знает про `bufio`, `os.Stdin` и команды. |

Каждый слой знает **только о слое снизу**. `Task` ничего не знает про `main`. `handleAdd` ничего не знает про `bufio.Scanner`. Это и есть **архитектура слоями** — фундаментальный принцип, на котором строится **любой** серьёзный сервер.

В Проекте 7 (блог-платформа) ты увидишь те же слои, только с другими названиями: domain → service → handler → main. Сейчас ты их встречаешь в миниатюре.

## Зачем это всё, если оно работало и так

Справедливый вопрос. Старый код **работал** — пользователь не заметит разницы. Зачем тратить день на рефакторинг?

Ответ: **код пишется один раз, а читается и меняется десятки раз**. Хорошая структура окупается на втором, третьем, десятом изменении. Например:

- Завтра ты захочешь сохранять задачи в файл → поправишь только `TaskList` (метод `Save`/`Load`). REPL не тронешь.
- Послезавтра захочешь добавить веб-интерфейс вместо REPL → поправишь только цикл. `TaskList` останется как есть.
- Через неделю поймёшь, что нужны тесты → ты сможешь тестировать `TaskList.Add` отдельно, без всякого ввода-вывода.

Каждое из этих изменений в **старом** коде стоило бы дня перетряхивания и кучи багов. В **новом** — пары часов.
