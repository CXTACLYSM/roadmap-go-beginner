# Модуль 6. Graceful shutdown — условие

## Задание

Добавить в бота **корректное завершение**: при Ctrl+C или SIGTERM бот доделывает текущие операции, отменяет висящие HTTP-запросы к Telegram, закрывает БД и завершается с кодом 0. Контекст пробрасывается через все слои.

## Что должно происходить

```bash
$ BOT_TOKEN="..." DATABASE_URL="..." go run .
2026/04/13 12:00:00 Подключение к PostgreSQL установлено
2026/04/13 12:00:00 Бот запускается...
2026/04/13 12:00:00 Бот авторизован: @lev_todo_bot
2026/04/13 12:00:15 Сообщение от @lev: /list
^C
2026/04/13 12:00:18 Получен сигнал interrupt, начинаем graceful shutdown
2026/04/13 12:00:18 Контекст отменён, выходим из цикла
2026/04/13 12:00:18 Бот остановлен корректно
```

Заметь: между Ctrl+C и финальным сообщением — **меньше секунды**. Это потому, что висящий `getUpdates(timeout=30)` отменяется через контекст мгновенно.

## Требования

1. В `main` создаётся корневой контекст: `ctx, cancel := context.WithCancel(context.Background())`
2. `defer cancel()` сразу после
3. Создан буферизированный канал сигналов: `sigChan := make(chan os.Signal, 1)`
4. Зарегистрированы оба сигнала: `signal.Notify(sigChan, os.Interrupt, syscall.SIGTERM)`
5. Запущена горутина, которая слушает `sigChan` и вызывает `cancel()` при получении сигнала
6. `Bot.Run` принимает первым аргументом `ctx context.Context`
7. В цикле `Run` есть **проверка отмены** через `select { case <-ctx.Done(): return; default: }` в начале каждой итерации
8. **Все методы клиента** имеют версии с контекстом:
   - `GetMeContext(ctx)`
   - `GetUpdatesContext(ctx, offset)`
   - `SendMessageContext(ctx, chatID, text)`
9. HTTP-запросы используют `http.NewRequestWithContext` + `client.Do(req)` вместо `client.Get`/`client.Post`
10. **Все методы репозитория** имеют версии с контекстом:
    - `AddContext`, `ListContext`, `DoneContext`, `DeleteContext`
11. Используются `QueryRowContext`, `QueryContext`, `ExecContext` вместо обычных
12. Контекст пробрасывается из `Run` через `handleMessage` → `handleCommand` / `handleStateful` → `handleAdd` / `handleList` / etc. → методы репозитория
13. **Retry с задержкой** в Run использует `select + time.After` вместо `time.Sleep`, чтобы можно было прервать ожидание контекстом
14. После выхода из `Run` в `main` логируется «Бот остановлен корректно»
15. `defer db.Close()` присутствует
16. Программа завершается с кодом **0** (не 130 от Ctrl+C, не 1 от ошибки)
17. Все сценарии модулей 3-5 продолжают работать
18. **Граф ный shutdown реально работает**: Ctrl+C → быстрое завершение

## Эксперименты

1. **Замерь, насколько быстро завершается бот:**
   ```bash
   time (echo "ждём 5 сек, потом Ctrl+C" && sleep 5 && kill -INT $$)
   ```
   Должно быть ~5 секунд (5 секунд ожидания + почти мгновенное завершение).

2. **Без контекста vs с контекстом:**
   - Сначала запусти **старую** версию (модуль 5), нажми Ctrl+C во время `getUpdates`. Замерь время до завершения.
   - Запусти новую с graceful shutdown, повтори. Сравни.
   - Старая может висеть до 30 секунд (long polling timeout). Новая — мгновенно.

3. **Пошли SIGTERM:**
   ```bash
   # В одном терминале запусти бота:
   BOT_TOKEN=... DATABASE_URL=... go run . &
   echo $!  # запомни PID

   # Во втором:
   kill -TERM <PID>
   ```
   Бот должен корректно завершиться. Это симулирует, что делает kubernetes при остановке пода.

4. **Симулируй долгую БД-операцию:** временно добавь `time.Sleep(10 * time.Second)` в `AddContext`. Запусти бота, начни добавление задачи, в момент задержки нажми Ctrl+C. Что происходит? Зависит от того, успеет ли `pgsql` отменить запрос. С контекстом — БД получает сигнал отмены и `INSERT` не доходит до выполнения.

5. **Что произойдёт, если нажать Ctrl+C дважды быстро?** В нашей реализации — второй сигнал тоже попадёт в `sigChan`, но горутина уже занята обработкой первого. Ничего страшного. В production-версии (с timeout) второй Ctrl+C принудительно выходит.

6. **Добавь shutdown timeout** (опционально, продвинутое):
   ```go
   go func() {
       sig := <-sigChan
       log.Printf("Сигнал %v, shutdown", sig)
       cancel()

       select {
       case sig := <-sigChan:
           log.Printf("Второй сигнал %v, принудительный выход", sig)
           os.Exit(1)
       case <-time.After(30 * time.Second):
           log.Println("Timeout shutdown, принудительный выход")
           os.Exit(1)
       }
   }()
   ```

7. **Проверь exit code:** после `Ctrl+C` сделай `echo $?`. Должно быть **0** (успех), не 130. Если 130 — значит, Go получил SIGINT, но не успел корректно завершиться.

8. **Проверь, что соединения с БД действительно закрыты:**
   ```bash
   # До остановки бота:
   psql ... -c "SELECT count(*) FROM pg_stat_activity WHERE datname='todo_bot';"
   # Должно быть несколько активных

   # После остановки:
   psql ... -c "SELECT count(*) FROM pg_stat_activity WHERE datname='todo_bot';"
   # Должно быть 0 (или сильно меньше)
   ```

9. **Сравни с веб-сервером:** в Проекте 4 ты использовал `http.ListenAndServe`, который не имеет встроенной поддержки graceful shutdown. Современный паттерн — `http.Server` с методом `Shutdown(ctx)`. Посмотри на `http.Server.Shutdown` в документации Go — это аналог нашего graceful shutdown для HTTP-сервера.

## Что показать Claude

Покажи Claude свой `main` с graceful shutdown и спроси: «правильно ли я обрабатываю случай, когда `Run` уже вышел сам (не из-за сигнала), а сигнал ещё не приходил?». Это важный edge case:

- Если `Run` вышел из-за фатальной ошибки → горутина с `<-sigChan` остаётся висеть навсегда
- Решение — `defer cancel()` плюс `<-ctx.Done()` в горутине вместо `<-sigChan`:
  ```go
  go func() {
      select {
      case sig := <-sigChan:
          log.Printf("сигнал %v", sig)
          cancel()
      case <-ctx.Done():
          // контекст уже отменён, выходим
      }
  }()
  ```

Это **более чистая** версия, в которой горутина гарантированно завершается в любом случае.

Запиши себе на будущее: **graceful shutdown — это про дисциплину**, не про сложные паттерны. Главное — пробросить контекст везде, где есть длительные операции, и закрыть всё, что открыл. Всё остальное — детали.

Если ты сейчас вернёшься к Проекту 4 (HTTP-сервер) и добавишь туда graceful shutdown через `http.Server.Shutdown(ctx)` — ты узнаешь, как это делается для серверов. Это полезное упражнение по итогам Проекта 6.
