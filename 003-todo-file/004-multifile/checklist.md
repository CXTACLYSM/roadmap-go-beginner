## Самопроверка через структуру

Открой свой проект. Запусти `tree app-multifile/` или просто посмотри файлы. Должно быть примерно так:

```
app-multifile/
├── go.mod
├── go.sum (если есть зависимости)
├── handlers.go
├── main.go
├── task.go
├── tasklist.go
└── tasks.json (после первого запуска)
```

Открой каждый файл и спроси себя: **«если я завтра захочу X, в какой файл я пойду?»**

- «Поменять формат вывода задачи» → `task.go` ✓
- «Добавить новую команду» → `main.go` (switch) + `handlers.go` (новый handler) ✓
- «Сменить место хранения файла» → `main.go` (константа) ✓
- «Добавить метод на список** → `tasklist.go` ✓
- «Поправить парсинг команды» → `handlers.go` ✓

Если на каждый вопрос у тебя один **очевидный** файл — рефакторинг удался.

## Самопроверка: финальный сценарий

Прогон с нуля:

```bash
cd app-multifile/
rm -f tasks.json
go run .
> add Тест 1
> add Тест 2
> done 1
> exit
go run .
> list
# должны быть две задачи
> exit
```

Если работает — модуль закрыт.

4. **«Файла нет» = нормальное состояние, а не ошибка** — пример **дизайна API**, в котором техническая ситуация превращается в осмысленный возврат
