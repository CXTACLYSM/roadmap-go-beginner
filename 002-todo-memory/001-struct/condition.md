# Модуль 1. Структура Task — условие

## Задание

Напиши программу, которая работает с **одной задачей** через структуру `Task`. Никакого списка задач, никаких команд от пользователя — только тип, методы и демонстрация их работы.

## Что должна делать программа

```
1. [ ] Купить молоко
1. [x] Купить молоко
```

Программа создаёт одну задачу, выводит её, отмечает как выполненную и выводит снова.

## Требования

1. Объявлен тип `Task` со **тремя полями**: `ID int`, `Title string`, `Done bool`.
2. Имена полей — с большой буквы.
3. Есть метод `String() string` с **receiver-значением**, который форматирует задачу для вывода. Пустая задача → `[ ]`, выполненная → `[x]`.
4. Есть метод `MarkDone()` с **receiver-указателем**, который отмечает задачу как выполненную.
5. В `main` создаётся одна задача, вызываются оба метода, результаты выводятся через `fmt.Println`.
6. Все поля заполняются **по именам**, а не позиционно.

## Шаги

1. Создай новую папку для проекта 2, инициализируй go-модуль (`go mod init todo`)
2. Создай `main.go` со скелетом
3. Объяви тип `Task`
4. Реализуй метод `String() string` с receiver `(t Task)`
5. Реализуй метод `MarkDone()` с receiver `(t *Task)`
6. В `main` создай задачу с понятным `Title`
7. Выведи её
8. Вызови `MarkDone()`
9. Выведи снова
10. Запусти `go run main.go`

## Эксперименты

1. **Сломай receiver MarkDone.** Поменяй `(t *Task)` на `(t Task)` (без звёздочки). Запусти. Что произойдёт со вторым выводом? Почему?
2. **Создай задачу через `var t Task`** без инициализации. Выведи её через `String()`. Что увидишь? Откуда взялись эти значения?
3. **Создай задачу позиционно:** `t := Task{2, "Тест", true}`. Работает? Теперь поменяй порядок полей в самом объявлении типа (например, `Done` сделай первым). Что произошло с твоим позиционным конструктором? Это и есть причина, почему так не пишут.
4. **Добавь поле `Priority int`** в структуру. Что сломалось? Где? Почему именованная инициализация ничего не сломала, а позиционная — сломала?
5. **Добавь второй метод** — `MarkUndone() *Task)`. Используй его в main. Заметь симметрию с `MarkDone`.

## Что показать Claude

Покажи Claude свой код **до и после эксперимента 1** (с поломанным receiver). Спроси: «почему именно копия — а не оригинал — изменяется?». Claude нарисует тебе словами, что происходит в памяти. Это самое важное понимание модуля 1 — и оно понадобится тебе во всех последующих модулях.
