# Проект 5. To-Do с PostgreSQL

**Длительность:** 3–4 недели
**Цель:** заменить файловое хранилище на настоящую базу данных. Освоить SQL, познакомиться с `database/sql`, понять, что такое транзакции и зачем они нужны.

> 🔥 **Это второй переломный проект.** Проект 4 сделал тебя «бэкендером, который пишет HTTP». Проект 5 сделает тебя «бэкендером, который умеет хранить данные правильно». Без этого навыка ни один реальный проект не существует.

## Что в итоге будет уметь программа

Тот же REST API, что в Проекте 4 (`GET/POST/PUT/DELETE /tasks`), **но**:

- Данные лежат в **PostgreSQL**, а не в JSON-файле
- Поиск задачи по ID — через **индекс**, мгновенный даже на миллионе записей
- Параллельные запросы корректно обрабатываются благодаря **транзакциям** БД (а не нашему ручному `sync.Mutex`)
- Можно запустить **два экземпляра** сервера на одной БД — оба будут работать корректно
- Данные сохраняются автоматически — никакого `Save` после каждого изменения

## Что нового по сравнению с Проектом 4

- **PostgreSQL** — самая распространённая реляционная БД в мире Go-разработки. Бесплатная, стабильная, мощная. Знание PostgreSQL — это необходимый минимум.
- **SQL** — отдельный декларативный язык, который ты пишешь **внутри** Go-кода. SQL — это «язык работы с данными», и его обязан знать любой бэкендер.
- **`database/sql`** — стандартный пакет Go для работы с реляционными БД. Это **абстракция над любым SQL-драйвером**: твой код почти не изменится, если ты потом захочешь переехать на MySQL или SQLite.
- **Драйвер pgx** — самый популярный современный драйвер для PostgreSQL в Go. Мы используем его через интерфейс `database/sql`, но он же предоставляет нативный API для продвинутых случаев.
- **Транзакции** — механизм, который превращает несколько операций в **одну атомарную единицу**. Либо все прошли, либо ни одна. Это решает проблему частичных сбоев, которая в Проекте 4 была «костылём» через мьютекс.
- **Миграции** — структурированный способ изменять схему БД. В Проекте 5 — простые `.sql` файлы, в продакшене бывают сложнее (golang-migrate, goose, atlas).
- **Connection pooling** — `database/sql` сам управляет пулом соединений. Тебе не нужно открывать и закрывать коннект для каждого запроса — это происходит автоматически.

## Что нового **не** будет (но появится потом)

- **ORM** — Gorm, ent, sqlboiler. Эти инструменты генерируют код на основе схемы и упрощают работу с БД, но прячут SQL. Сначала ты должен понимать, **что** делает ORM, прежде чем доверять ему. Поэтому тут — голый SQL.
- **sqlc** — кодогенератор, который превращает SQL-запросы в типобезопасные Go-функции. Очень популярен в современном Go. Ты познакомишься с ним в Проекте 6 или 7.
- **Миграции через библиотеку** — `golang-migrate`, `goose`. Полезные инструменты, но в учебном проекте они добавляют когнитивную нагрузку. Сейчас — `psql -f migrations/*.sql`.
- **Связи между таблицами** — JOIN, foreign keys, многие-ко-многим. Это будет в Проекте 7 (блог-платформа).
- **Индексы кроме primary key** — упомянем, но не углубимся.

## Главные мысли проекта

**Первое.** SQL — это не «язык БД», это **язык работы с данными**. Раз освоив SQL, ты будешь использовать его везде: PostgreSQL, MySQL, SQLite, ClickHouse, BigQuery, Snowflake — везде один и тот же стандарт с минимальными отличиями. Это инвестиция на десятилетия.

**Второе.** `database/sql` — это **низкий уровень**. Тебе придётся писать SQL руками, маппить колонки на поля Go-структур, обрабатывать ошибки. Это **не неудобно** — это **прозрачно**. Ты точно знаешь, что и когда происходит. ORM можно подключать **потом**, когда понимание уже есть.

**Третье.** Транзакции — это **самая важная идея**, которую ты увезёшь из Проекта 5. До них ты решал проблему «параллельных изменений» через мьютекс. После них ты увидишь, что БД делает это **на порядок мощнее**: не только защищает от гонок, но и даёт **атомарность** (всё или ничего), **изоляцию** (одна транзакция не видит «сырые» данные другой) и **долговечность** (после COMMIT данные точно на диске).

**Четвёртое.** Connection pooling — это **дар database/sql**. Ты не будешь думать «когда открыть коннект, когда закрыть». Просто пишешь `db.Query(...)`, и пакет сам берёт коннект из пула, использует, возвращает. Это спасает от большого класса багов в многопоточных серверах.

## Перед началом

Установи **Docker**. На Linux/macOS — через официальную инструкцию. На Windows — через WSL 2.

Также установи `psql` — клиент командной строки для PostgreSQL. Это базовый инструмент, без которого работать с БД неудобно:

- **Linux**: `sudo apt install postgresql-client` (Ubuntu/Debian)
- **macOS**: `brew install libpq && brew link --force libpq`
- **Windows**: ставится вместе с PostgreSQL для Windows

Проверь:

```bash
docker --version
psql --version
```

Если обе команды работают — ты готов начинать модуль 1.
