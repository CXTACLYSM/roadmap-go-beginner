# Модуль 6. Интеграция с файлом — условие

## Задание

Подключить сохранение в файл к HTTP-серверу. Это **финальный модуль** Проекта 4. После него у тебя будет полноценный REST API с переживающим перезапуск состоянием.

## Требования

1. Перенесены/добавлены методы `Save` и `Load` на `TaskList`
2. Оба метода **захватывают мьютекс** через `Lock + defer Unlock`
3. `Load` корректно обрабатывает «файла нет» — возвращает `nil`, оставляя список пустым
4. В `main` **до запуска сервера** вызывается `tasks.Load(dataFile)`
5. При ошибке `Load` программа завершается через `log.Fatalf`
6. Имя файла — константа `const dataFile = "tasks.json"`
7. Реализован helper `(s *Server) saveOrWarn()`, который вызывает `Save` и логирует ошибку
8. `saveOrWarn` вызывается в **трёх** обработчиках:
   - `createTask` (POST)
   - `updateTask` (PUT)
   - `deleteTask` (DELETE)
9. `saveOrWarn` **не** вызывается в `listTasks` и `getTask` (только чтение)
10. Используется пакет `log` для серверных сообщений (а не `fmt`)
11. Все эндпоинты из модулей 2-4 продолжают работать
12. Защита из модуля 5 сохранена — `go run -race .` не ругается под нагрузкой
13. После перезапуска сервера задачи **не пропадают**

## Эксперименты

1. **Удали `tasks.json`** перед запуском. Сервер должен стартовать с пустым списком (не упасть).

2. **Сломай JSON в файле руками.** Открой `tasks.json`, удали закрывающую скобку. Запусти сервер. Что произошло? Программа упала с понятной ошибкой через `log.Fatalf`. Это правильно — лучше отказаться запускаться, чем затереть данные.

3. **Запусти нагрузочный тест с -race снова.** Создай 100 задач параллельно. Дождись окончания. Останови сервер. Запусти снова. Проверь, что в файле **ровно 100 задач** и все ID уникальны. Это **тест на корректность с persistence**.

4. **Замерь, сколько раз пишется файл за нагрузочный тест.** Отслеживай размер файла или модификации:
   ```bash
   while true; do stat -c "%s %y" tasks.json 2>/dev/null; sleep 0.5; done
   ```
   Заметь — файл переписывается **полностью** при каждом изменении. На 100 задач это 100 записей. **Это очень неэффективно** — настоящие БД пишут только инкременты. Но для нашего проекта приемлемо.

5. **Попробуй вариант B** (сохранение внутри методов TaskList) — сделай ветку или копию проекта. Сравни читаемость. Какой вариант тебе больше нравится?

6. **Запусти ДВА экземпляра сервера на разных портах** с одним и тем же файлом:
   ```bash
   PORT=8080 go run .  # один терминал
   PORT=8081 go run .  # другой терминал
   ```
   (Тебе придётся поправить код, чтобы порт читался из env.) Создай задачу в первом, потом во втором, потом проверь файл. Что произошло? Это та самая «проблема параллельной записи в файл», которую решает БД через блокировки.

7. **Сделай graceful shutdown** (опционально, продвинутое):
   ```go
   import "os/signal"

   func main() {
       // ...
       go func() {
           if err := http.ListenAndServe(":8080", nil); err != nil { ... }
       }()

       // Ждём сигнал
       sigChan := make(chan os.Signal, 1)
       signal.Notify(sigChan, os.Interrupt)
       <-sigChan

       log.Println("Получен сигнал, сохраняем и выходим...")
       tasks.Save(dataFile)
   }
   ```
   Это **дополнительное** упражнение — graceful shutdown будет в Проектах 5-7. Сейчас просто посмотри, как примерно это устроено.

## Что показать Claude

Покажи Claude свой `createTask` после интеграции и спроси: «Что произойдёт, если `Save` упадёт? Какие у меня варианты обработки этой ошибки, и почему я выбрал именно логирование, а не возврат 500 клиенту?». Это вопрос **дизайна** — у обоих ответов есть смысл, и выбор зависит от того, какое поведение важнее: «всегда ответить успехом, если изменение в памяти прошло» или «всегда быть честным с клиентом про итоговое состояние».

В **продакшен-системах** на этот вопрос обычно отвечают БД через **транзакции**: «изменение прошло, только если оно сохранено на диск». Если что-то пошло не так — **откатывается** и в памяти, и на диске. Это атомарность. Сейчас у нас её нет — изменение в памяти и запись на диск — две **разные** операции. В Проекте 5 ты увидишь, как это устроено правильно.
