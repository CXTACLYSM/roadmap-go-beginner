# Модуль 3. POST — создание задачи — условие

## Задание

Добавить к серверу из модуля 2 эндпоинт `POST /tasks`, который принимает JSON, валидирует его и создаёт новую задачу. Должны корректно обрабатываться все ошибочные случаи.

## Что должна делать программа

```bash
# Создание
$ curl -i -X POST http://localhost:8080/tasks \
    -H "Content-Type: application/json" \
    -d '{"title": "Купить молоко"}'
HTTP/1.1 201 Created
Content-Type: application/json

{"id":3,"title":"Купить молоко","done":false}

# Невалидный JSON
$ curl -i -X POST http://localhost:8080/tasks \
    -H "Content-Type: application/json" \
    -d 'не json'
HTTP/1.1 400 Bad Request
...

# Пустой title
$ curl -i -X POST http://localhost:8080/tasks \
    -H "Content-Type: application/json" \
    -d '{"title": ""}'
HTTP/1.1 400 Bad Request

# Title из пробелов
$ curl -i -X POST http://localhost:8080/tasks \
    -H "Content-Type: application/json" \
    -d '{"title": "   "}'
HTTP/1.1 400 Bad Request

# Пустое тело
$ curl -i -X POST http://localhost:8080/tasks
HTTP/1.1 400 Bad Request

# GET всё ещё работает
$ curl http://localhost:8080/tasks
[..., {"id":3,"title":"Купить молоко","done":false}]
```

## Требования

1. Реализован метод `(s *Server) createTask(w, r)`.
2. Зарегистрирован эндпоинт `POST /tasks` (тот же путь, что у GET, но другой метод).
3. Используется `json.NewDecoder(r.Body).Decode(...)`.
4. Используется **локальная анонимная структура** для входа (DTO):
   ```go
   var input struct {
       Title string `json:"title"`
   }
   ```
5. Невалидный JSON → 400 с понятным сообщением.
6. Пустое тело → 400.
7. Пустой `title` (включая пробелы) → 400.
8. Успешное создание → **201 Created** + JSON созданной задачи в теле.
9. После каждой ошибки — `return`.
10. Метод `Add` на `TaskList` возвращает созданную задачу (если ещё не так — поправь).
11. **GET-эндпоинты из модуля 2 продолжают работать** без изменений.

## Шаги

1. Открой свой `app/` из модуля 2
2. Открой `server.go`, добавь метод `createTask`
3. Проверь, что `Add` на `TaskList` возвращает `Task`
4. В `main.go` зарегистрируй `http.HandleFunc("POST /tasks", server.createTask)`
5. Запусти `go run .`
6. Прогоняй каждый сценарий из примеров через `curl`
7. Особенно — все ошибочные случаи. Каждый должен возвращать 400.

## Эксперименты

1. **Создай задачу через POST, потом запроси GET /tasks** — увидишь, что она появилась в списке. Это первая «настоящая» интерактивность твоего API.

2. **Попробуй POST без заголовка `Content-Type`:**
   ```bash
   curl -i -X POST http://localhost:8080/tasks -d '{"title": "test"}'
   ```
   Работает? **Да, работает.** Заголовок `Content-Type` для нашего обработчика не обязателен — мы парсим JSON независимо от того, что заявил клиент. Но **в строгих API** часто проверяют этот заголовок и возвращают 415 Unsupported Media Type, если он не `application/json`. Подумай, нужна ли тебе такая строгость.

3. **Пришли валидный JSON, но с лишним полем:**
   ```bash
   curl -X POST http://localhost:8080/tasks \
       -H "Content-Type: application/json" \
       -d '{"title": "test", "evil": true, "id": 999}'
   ```
   Что произошло? Лишние поля **игнорируются**, ID **не использован** (наш сервер всегда присваивает свой). Это **хорошо**: клиент не может установить ID в обход. Это и есть одна из причин, почему мы используем DTO, а не `Task` напрямую.

4. **Пришли вложенный JSON или массив:**
   ```bash
   curl -X POST http://localhost:8080/tasks \
       -H "Content-Type: application/json" \
       -d '[{"title": "test"}]'
   ```
   Что произошло? Массив нельзя декодировать в структуру — `Decode` вернёт ошибку, ты вернёшь 400. Хорошо.

5. **Используй `Task` напрямую вместо локальной структуры:**
   ```go
   var task Task
   json.NewDecoder(r.Body).Decode(&task)
   ```
   Запусти POST с `{"id": 999, "done": true, "title": "evil"}`. Что произошло? Если у тебя нет защиты — клиент **установит** ID 999 и `done: true`, в обход твоего `Add`. Это **дыра в API**. Поэтому DTO — не каприз, а защита.

6. **Создай 5 задач подряд через POST.** Заметь, что ID идут последовательно (3, 4, 5, 6, 7). Это работа `nextID`. Если бы у нас было параллельное обращение (что мы исправим в модуле 5), могли бы возникнуть race conditions.

7. **Используй MarshalIndent для красивого вывода JSON в ответе** (опционально). Заметь, что для API это обычно **плохая практика** — лишние пробелы тратят трафик. Используется только для отладочных эндпоинтов или для запросов с явным флагом `?pretty=true`.

## Что показать Claude

Покажи Claude свой `createTask` и спроси: «правильно ли я отделяю DTO от внутренней модели? Что было бы плохо, если бы я использовал `Task` напрямую?». Это важный момент **безопасности**: если клиент может установить любые поля задачи через POST, он может, например, установить чужой `userID`, или флаг `isAdmin`, или что-то ещё, что должно контролироваться сервером. Использование DTO — первая линия защиты от таких атак.

В Проекте 7 (блог-платформа), где будет аутентификация, эта мысль вернётся в полный рост: DTO для авторизации, DTO для регистрации, отдельный DTO для PATCH (только обновляемые поля). Сейчас ты учишься думать в этих категориях на простом примере.
