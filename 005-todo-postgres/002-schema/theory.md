# Модуль 2. Схема и миграции — теория

## Контекст

В модуле 1 ты создал таблицу `tasks` руками через `psql`. Это работает для одного человека на одной машине. Но в реальном проекте:

- Таблиц много (десятки или сотни)
- Над проектом работает **команда** — у каждого своя локальная БД, которую надо синхронизировать
- Схема **эволюционирует** — добавляются колонки, индексы, новые таблицы
- В **продакшене** БД отдельная, и её схема должна обновляться вместе с релизами

Если пытаться делать это «руками» — ничего не получится. Нужен **систематический** способ описывать изменения схемы и применять их в одном и том же порядке везде.

Этот способ называется **миграции**.

## Ядро 1: что такое миграция

**Миграция** — это **изменение схемы БД, сохранённое в виде файла** (обычно `.sql`), который можно применить к любой совместимой БД и получить тот же результат.

Простейшая форма — папка `migrations/` с пронумерованными файлами:

```
migrations/
├── 001_create_tasks_table.sql
├── 002_add_created_at.sql
├── 003_add_index_on_done.sql
└── 004_add_priority_column.sql
```

Каждый файл содержит SQL для **одного** изменения. Чтобы развернуть схему с нуля, ты применяешь файлы **по порядку**:

```bash
psql -f migrations/001_create_tasks_table.sql
psql -f migrations/002_add_created_at.sql
psql -f migrations/003_add_index_on_done.sql
psql -f migrations/004_add_priority_column.sql
```

Или, если у тебя уже есть таблица `001`, и ты добавил `005`:

```bash
psql -f migrations/005_new_change.sql
```

Главные принципы:

1. **Файл миграции — это immutable.** Один раз создал, **никогда не меняй**. Если нужно что-то поправить — создавай **новый** файл.
2. **Порядковый номер — обязателен.** Без него непонятно, в каком порядке применять.
3. **Каждая миграция — атомарна.** Внутри одного файла — одно логическое изменение.
4. **Имена должны быть осмысленными.** `001_create_tasks_table.sql` лучше, чем `001_init.sql` или `001.sql`.

## Ядро 2: forward-only vs reversible миграции

Есть два подхода к миграциям:

### Forward-only

Миграции **только добавляют** изменения. Если что-то пошло не так — пишешь новую миграцию, которая исправляет.

```
001_create_tasks.sql
002_add_priority.sql
003_remove_priority.sql   ← если решили, что priority не нужен
```

**Плюсы:** просто, понятно, минимум кода.
**Минусы:** история становится длинной, могут оставаться «мёртвые» колонки.

### Reversible (up/down)

Каждая миграция имеет **две части**: `up` (применить) и `down` (откатить).

```
001_create_tasks.up.sql
001_create_tasks.down.sql
```

`up.sql`:
```sql
CREATE TABLE tasks (...);
```

`down.sql`:
```sql
DROP TABLE tasks;
```

Если что-то сломалось — откатываешь через `down`.

**Плюсы:** можно вернуться назад.
**Минусы:** в два раза больше кода, и не все миграции **реально** обратимы (например, как «откатить» удаление колонки с данными?).

**Для нашего проекта используем forward-only.** Это проще, и для большинства случаев достаточно. Reversible миграции имеют смысл в крупных командах с CI/CD.

## Ядро 3: где хранить миграции

В **самом проекте**, в папке `migrations/`. Это **часть исходного кода**, а не отдельный артефакт. Когда ты добавляешь миграцию, ты делаешь это **в том же коммите**, что и Go-код, который от неё зависит.

```
005-todo-postgres/
├── go.mod
├── main.go
├── server.go
├── tasklist.go
├── task.go
└── migrations/
    ├── 001_create_tasks_table.sql
    └── 002_add_created_at.sql
```

Это правило: **миграции — это часть проекта**, а не «то, что ДBA делает руками».

## Ядро 4: типы данных PostgreSQL для нашего проекта

Помимо `INTEGER`, `TEXT`, `BOOLEAN`, в PostgreSQL есть много полезных типов. Самые важные для бэкендера:

| Тип | Назначение |
|-----|------------|
| `INTEGER` (`INT4`) | Целое 32 бита |
| `BIGINT` (`INT8`) | Целое 64 бита |
| `SERIAL` | Автоинкрементное `INTEGER` |
| `BIGSERIAL` | Автоинкрементное `BIGINT` |
| `TEXT` | Строка любой длины |
| `VARCHAR(N)` | Строка с ограничением длины |
| `BOOLEAN` | `true`/`false` |
| `TIMESTAMP` | Дата + время **без** часового пояса |
| `TIMESTAMPTZ` | Дата + время **с** часовым поясом |
| `DATE` | Только дата |
| `NUMERIC(precision, scale)` | Точное десятичное (для денег) |
| `JSONB` | Структурированные JSON-данные |
| `UUID` | Универсальный идентификатор |

Несколько правил для бэкендера:

- **Используй `BIGSERIAL` вместо `SERIAL`** для ID. `INTEGER` (32 бита) теоретически может закончиться. `BIGINT` — нет.
- **Используй `TIMESTAMPTZ`, не `TIMESTAMP`.** Часовые пояса — источник багов, и `TIMESTAMPTZ` решает их корректно. Это **правило** в современных проектах.
- **Никогда не используй `FLOAT` или `DOUBLE PRECISION` для денег.** Используй `NUMERIC(10, 2)` (или подходящие precision/scale). `FLOAT` накапливает ошибки округления — это катастрофа в финансах.
- **Не злоупотребляй `JSONB`.** Это удобно, но превращает реляционную БД в полу-документную. Хорошее правило: если у поля **есть структура и оно используется для фильтрации** — это должны быть **колонки**. Если это «контейнер для произвольных данных» — `JSONB`.

Для нашего проекта — простая схема:

```sql
CREATE TABLE tasks (
    id         BIGSERIAL PRIMARY KEY,
    title      TEXT NOT NULL,
    done       BOOLEAN NOT NULL DEFAULT false,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

`now()` — встроенная функция PostgreSQL, возвращает текущее время. `DEFAULT now()` означает «если не указано — записать текущее время».

## Ядро 5: ALTER TABLE для эволюции

Когда ты хочешь **добавить** что-то к существующей таблице, не пересоздавая её — используется `ALTER TABLE`:

```sql
-- Добавить колонку
ALTER TABLE tasks ADD COLUMN priority INTEGER DEFAULT 0;

-- Удалить колонку (опасно!)
ALTER TABLE tasks DROP COLUMN priority;

-- Переименовать колонку
ALTER TABLE tasks RENAME COLUMN done TO is_done;

-- Изменить тип колонки (тоже опасно)
ALTER TABLE tasks ALTER COLUMN title TYPE VARCHAR(255);

-- Добавить ограничение NOT NULL (потребует, чтобы все существующие строки имели значение)
ALTER TABLE tasks ALTER COLUMN priority SET NOT NULL;
```

`ALTER TABLE` — мощный инструмент, но **опасный**:

- Удаление колонки — потеря данных
- Изменение типа — может сломать существующие данные
- Установка NOT NULL — упадёт, если есть строки с NULL
- На больших таблицах ALTER может **заблокировать** таблицу на минуты, что приведёт к простою сервиса

Для учебного проекта это всё не критично. Но запоминай: в продакшен-системах любой `ALTER` нужно делать **осторожно** и с учётом нагрузки.

## Ядро 6: индексы

**Индекс** — это вспомогательная структура, которая ускоряет поиск по определённой колонке (или группе колонок). Без индекса PostgreSQL вынужден просматривать **все строки** таблицы, чтобы найти нужные. С индексом — лезет напрямую.

Простой пример:

```sql
CREATE INDEX idx_tasks_done ON tasks (done);
```

Теперь запрос `SELECT * FROM tasks WHERE done = false` будет работать **быстрее** на больших таблицах.

**Когда нужен индекс?** Когда:
- Колонка часто используется в `WHERE`, `JOIN`, `ORDER BY`
- Таблица большая (хотя бы тысячи строк)
- Селективность хорошая (значение делит таблицу на маленькие группы)

**Индексы автоматические:** `PRIMARY KEY` всегда индексируется. `UNIQUE` тоже.

**Минусы индексов:** замедляют `INSERT`/`UPDATE`/`DELETE`, занимают место. Поэтому **не добавляй индекс «на всякий случай»** — добавляй, когда видишь медленный запрос на больших данных.

В нашем проекте мы добавим **один** индекс — на `done` — больше для упражнения, чем по необходимости.

## Полный пример первой миграции

`migrations/001_create_tasks_table.sql`:

```sql
CREATE TABLE tasks (
    id         BIGSERIAL PRIMARY KEY,
    title      TEXT NOT NULL,
    done       BOOLEAN NOT NULL DEFAULT false,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

И вторая миграция, которая добавляет индекс:

`migrations/002_index_done.sql`:

```sql
CREATE INDEX idx_tasks_done ON tasks (done);
```

Применение:

```bash
psql -h localhost -U postgres -d postgres -f migrations/001_create_tasks_table.sql
psql -h localhost -U postgres -d postgres -f migrations/002_index_done.sql
```

Или, для удобства, **через переменную окружения**:

```bash
export PGPASSWORD=secret
psql -h localhost -U postgres -d postgres -f migrations/001_create_tasks_table.sql
psql -h localhost -U postgres -d postgres -f migrations/002_index_done.sql
```

С `PGPASSWORD` ты не вводишь пароль каждый раз. Но **никогда не делай этого в продакшене** — для серверов используют файлы с правами или vault. В разработке — нормально.

## Что мы НЕ делаем (и должны делать в продакшене)

В этом модуле мы делаем **миграции уровня джуна**:

- Применяем руками
- Не отслеживаем, какие применены
- Не имеем `down` миграций
- Не имеем транзакционности (если миграция упала посередине — частичное состояние)

В продакшен-проектах используют **специальные библиотеки**:

- **golang-migrate** (`github.com/golang-migrate/migrate`) — самая популярная для Go
- **goose** (`github.com/pressly/goose`) — альтернатива, чуть проще
- **atlas** — современная, поддерживает декларативные миграции

Эти инструменты:

- Создают таблицу `schema_migrations` для отслеживания применённых
- Применяют только то, что не было применено
- Откатывают через `down`
- Оборачивают каждую миграцию в транзакцию

**Когда переходить?** Когда твоему проекту больше пары недель и ты работаешь с командой. Для учебного проекта — простые `.sql` файлы достаточны.
