# Модуль 1. Первый HTTP-сервер — теория

## Контекст

До этого момента все твои программы работали по одной схеме: запустился, что-то сделал, завершился. Даже REPL из Проектов 2 и 3 — это просто **долго работающая** программа, но всё ещё с одним пользователем (тобой), который вводит команды через клавиатуру.

В этом модуле ты впервые напишешь программу совершенно другого типа — **сервер**. Сервер не завершается «когда сделал работу». Он **слушает** — ждёт, когда к нему обратятся, отвечает, и ждёт следующего запроса. И так — до тех пор, пока его кто-нибудь не остановит.

Это фундаментально другое мышление. И HTTP — самый распространённый протокол, по которому разговаривают серверы и клиенты в современном вебе.

## Ядро 1: что такое HTTP

**HTTP** (HyperText Transfer Protocol) — это **текстовый протокол** поверх TCP, по которому общаются веб-клиенты (браузер, мобильное приложение, другая программа) и веб-серверы.

Когда ты заходишь в браузере на `https://example.com/about`, происходит примерно такая последовательность:

1. Браузер устанавливает TCP-соединение с сервером `example.com` на порту `443` (для HTTPS) или `80` (для HTTP)
2. Браузер отправляет в это соединение **HTTP-запрос** в текстовом виде
3. Сервер читает запрос, что-то делает, отправляет обратно **HTTP-ответ** в текстовом виде
4. Браузер читает ответ и показывает его пользователю

Самое важное: **запрос и ответ — это просто текст**. Никакой магии. Если бы ты подключился к серверу `telnet`-ом и набил руками правильный текст — сервер ответил бы как настоящему браузеру.

Пример HTTP-запроса (как он выглядит в сети):

```
GET /about HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Accept: text/html

```

Пример ответа:

```
HTTP/1.1 200 OK
Content-Type: text/html
Content-Length: 137

<html>
<body>
<h1>About</h1>
</body>
</html>
```

Структура одинаковая для запроса и ответа:

1. **Стартовая строка** — метод и путь (для запроса) или статус (для ответа)
2. **Заголовки** — пары «ключ: значение», по одному на строку
3. **Пустая строка**
4. **Тело** (опционально) — данные, которые передаются

## Ядро 2: HTTP-методы

**Метод** — это глагол, который описывает, **что** клиент хочет сделать. Самые важные:

| Метод | Назначение | Идемпотентный? |
|-------|------------|-----------------|
| **GET** | Получить ресурс (читать) | Да |
| **POST** | Создать новый ресурс | Нет |
| **PUT** | Полностью заменить ресурс | Да |
| **PATCH** | Частично обновить ресурс | Нет (обычно) |
| **DELETE** | Удалить ресурс | Да |

**Идемпотентный** означает: «вызов один раз = вызов десять раз с теми же аргументами». GET идемпотентный — если ты запросишь `/tasks/1` десять раз, ничего не изменится. POST не идемпотентный — если ты создашь задачу десять раз, у тебя будет десять задач.

Это важное свойство. Идемпотентные запросы можно безопасно повторять, если ответа не пришло (например, оборвалось соединение). Неидемпотентные — нельзя.

## Ядро 3: HTTP-статусы

**Статус** — это трёхзначный код, который сервер возвращает в ответе. Он говорит клиенту, как обработался запрос. Группы:

| Группа | Значение | Примеры |
|--------|----------|---------|
| **2xx** | Успех | `200 OK`, `201 Created`, `204 No Content` |
| **3xx** | Перенаправление | `301 Moved Permanently`, `302 Found` |
| **4xx** | Ошибка клиента | `400 Bad Request`, `404 Not Found`, `409 Conflict` |
| **5xx** | Ошибка сервера | `500 Internal Server Error`, `503 Service Unavailable` |

Ключевое правило, которое часто путают:
- **4xx** — виноват **клиент**. Прислал неправильные данные, попросил несуществующий ресурс, не имеет прав
- **5xx** — виноват **сервер**. Что-то сломалось внутри сервера: упала база, вылетела горутина, ошибка в коде

Например, если клиент попросил `/tasks/abc` (а ID должен быть числом) — это **400 Bad Request**, не 500. Сервер работает нормально, проблема в запросе клиента.

## Ядро 4: первый сервер на Go

В Go стандартная библиотека уже содержит всё для HTTP-сервера. Никаких внешних зависимостей. Минимальный сервер выглядит так:

```go
package main

import (
    "fmt"
    "net/http"
)

func helloHandler(w http.ResponseWriter, r *http.Request) {
    fmt.Fprintln(w, "Hello, world!")
}

func main() {
    http.HandleFunc("/hello", helloHandler)

    fmt.Println("Сервер запущен на http://localhost:8080")
    err := http.ListenAndServe(":8080", nil)
    if err != nil {
        fmt.Println("Ошибка сервера:", err)
    }
}
```

Запускаешь:

```bash
$ go run .
Сервер запущен на http://localhost:8080
```

В другом терминале:

```bash
$ curl http://localhost:8080/hello
Hello, world!
```

Или открываешь в браузере `http://localhost:8080/hello` — увидишь то же.

Разберём по частям.

## Что такое handler

**Handler** (обработчик) — это функция, которая обрабатывает HTTP-запрос. Сигнатура у неё всегда одинаковая:

```go
func имя(w http.ResponseWriter, r *http.Request)
```

Два аргумента:

1. **`w http.ResponseWriter`** — это твой канал для **записи ответа**. Через него ты отправляешь клиенту статус, заголовки и тело.
2. **`r *http.Request`** — это **запрос**, который пришёл от клиента. Здесь лежат: метод, путь, заголовки, тело, query-параметры — всё, что прислал клиент.

Имя `w` (writer) и `r` (request) — это идиоматичные сокращения в Go-сообществе. Можно называть `respWriter` и `req`, но `w` и `r` короче и узнаваемы.

## ResponseWriter и его интерфейсы

`ResponseWriter` — это **интерфейс**, который умеет три вещи:

```go
type ResponseWriter interface {
    Header() Header                    // получить заголовки для записи
    Write([]byte) (int, error)         // записать байты в тело ответа
    WriteHeader(statusCode int)        // установить статус
}
```

- **`Write([]byte)`** — записывает байты в тело ответа. Можно вызывать несколько раз.
- **`WriteHeader(int)`** — устанавливает статус ответа. Должен быть вызван **до** `Write`.
- **`Header()`** — возвращает мапу заголовков. Можно изменить **до** первого вызова `Write` или `WriteHeader`.

Если ты не вызываешь `WriteHeader` явно, Go установит **200 OK** автоматически при первом `Write`.

`fmt.Fprintln(w, ...)` — это удобная обёртка над `Write`, которая работает с `Writer`-ами вообще (включая `ResponseWriter`).

## HandleFunc и ListenAndServe

```go
http.HandleFunc("/hello", helloHandler)
```

**`HandleFunc(path, handler)`** регистрирует функцию-обработчик для определённого пути. Когда придёт запрос на `/hello` — будет вызван `helloHandler`. На любой другой путь — вернётся `404 Not Found` (это поведение по умолчанию).

```go
http.ListenAndServe(":8080", nil)
```

**`ListenAndServe(addr, handler)`** — запускает сервер. Это **блокирующий** вызов: программа будет работать здесь до тех пор, пока сервер не упадёт или его не остановят. Поэтому `ListenAndServe` обычно идёт в самом конце `main`.

- **`":8080"`** — адрес и порт. Двоеточие в начале означает «слушать на всех сетевых интерфейсах». Если бы было `"localhost:8080"`, сервер слушал бы только локальные подключения.
- **`nil`** во втором аргументе означает «использовать **глобальный мультиплексор** (`DefaultServeMux`)», в который ты регистрировал обработчики через `http.HandleFunc`. Это самый простой подход для начала.

## Что значит «порт»

**Порт** — это число от 1 до 65535, которое определяет, к **какой** программе на компьютере подключаться. Один компьютер может одновременно иметь много программ, которые слушают разные порты.

Стандартные порты:
- **80** — HTTP (без шифрования)
- **443** — HTTPS (с шифрованием)
- **22** — SSH
- **5432** — PostgreSQL
- **6379** — Redis

**Почему 8080?** Потому что порты ниже 1024 на Linux/macOS требуют **прав суперпользователя**. Для разработки удобнее брать порт повыше — `8080` (как `80` + 80) или `3000` или `8000`. Это просто соглашение, ничего особенного.

## Тестирование сервера

Когда сервер запущен, его нужно как-то проверить. Три основных способа:

**1. Браузер.** Простейший. Открой `http://localhost:8080/hello` — увидишь ответ. Работает только для GET.

**2. curl.** Командная утилита для HTTP-запросов. Для GET:
```bash
curl http://localhost:8080/hello
```

С подробностями:
```bash
curl -v http://localhost:8080/hello
```

`-v` (verbose) показывает заголовки запроса и ответа. Очень полезно для отладки.

**3. Postman.** Графический клиент. Удобнее `curl`-а для сложных запросов (с заголовками, телом, файлами). Бесплатный.

В этом проекте мы будем использовать в основном `curl`.

## Полный пример с двумя обработчиками

```go
package main

import (
    "fmt"
    "net/http"
)

func helloHandler(w http.ResponseWriter, r *http.Request) {
    fmt.Fprintln(w, "Hello, world!")
}

func timeHandler(w http.ResponseWriter, r *http.Request) {
    fmt.Fprintln(w, "Метод:", r.Method)
    fmt.Fprintln(w, "Путь:", r.URL.Path)
    fmt.Fprintln(w, "User-Agent:", r.Header.Get("User-Agent"))
}

func main() {
    http.HandleFunc("/hello", helloHandler)
    http.HandleFunc("/info", timeHandler)

    fmt.Println("Сервер запущен на http://localhost:8080")
    err := http.ListenAndServe(":8080", nil)
    if err != nil {
        fmt.Println("Ошибка сервера:", err)
    }
}
```

После запуска:

```bash
$ curl http://localhost:8080/hello
Hello, world!

$ curl http://localhost:8080/info
Метод: GET
Путь: /info
User-Agent: curl/7.68.0
```

Заметь, что мы достали из запроса:
- `r.Method` — HTTP-метод
- `r.URL.Path` — путь
- `r.Header.Get("User-Agent")` — конкретный заголовок

Это лишь верхушка того, что можно получить из запроса. В следующих модулях ты увидишь, как читать тело, query-параметры и path-параметры.

## Как остановить сервер

В терминале, где запущен сервер, нажми **Ctrl+C**. Это пошлёт сигнал прерывания, и Go-программа завершится.

«Корректное» завершение (graceful shutdown), которое дожидается обработки текущих запросов перед выходом — это более продвинутая тема. Сейчас Ctrl+C достаточно.
