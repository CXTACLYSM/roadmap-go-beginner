# Проект 7. Блог-платформа с авторизацией

**Длительность:** 4 недели
**Цель:** собрать всё, что освоено в roadmap-е, в **полноценный многопользовательский веб-сервис** с настоящей аутентификацией, многотабличной схемой, тестами и production-ready структурой.

> 🏁 **Это финальный проект roadmap-а.** До этого ты строил по одному навыку за раз. Здесь — собираешь всё в одно решение, которое **не стыдно показать на собеседовании** как пет-проект.

## Что в итоге будет уметь сервис

REST API блог-платформы, с которым может работать любой клиент (браузер, мобильное приложение, другой бэкенд):

```
POST   /register            → создать аккаунт
POST   /login               → получить JWT-токен
GET    /me                  → информация о себе (требует токен)

GET    /posts               → список всех постов (публично)
GET    /posts/{id}          → один пост с автором (публично)
POST   /posts               → создать пост (требует токен)
PUT    /posts/{id}          → редактировать (только автор)
DELETE /posts/{id}          → удалить (только автор)

GET    /posts/{id}/comments → комментарии к посту (публично)
POST   /posts/{id}/comments → добавить комментарий (требует токен)
```

Пример сценария:

```bash
# Регистрация
$ curl -X POST http://localhost:8080/register \
    -H "Content-Type: application/json" \
    -d '{"email":"lev@example.com","password":"strong-password-123"}'
{"id":1,"email":"lev@example.com","created_at":"2026-04-13T12:00:00Z"}

# Логин
$ curl -X POST http://localhost:8080/login \
    -H "Content-Type: application/json" \
    -d '{"email":"lev@example.com","password":"strong-password-123"}'
{"token":"eyJhbGciOiJIUzI1NiIs..."}

# Создание поста (с токеном)
$ TOKEN="eyJhbGciOiJIUzI1NiIs..."
$ curl -X POST http://localhost:8080/posts \
    -H "Authorization: Bearer $TOKEN" \
    -H "Content-Type: application/json" \
    -d '{"title":"Привет, мир","body":"Это мой первый пост"}'
{"id":1,"author_id":1,"title":"Привет, мир","body":"...","created_at":"..."}

# Чтение постов (публично, без токена)
$ curl http://localhost:8080/posts
[{"id":1,"author":{"id":1,"email":"lev@example.com"},"title":"...","body":"..."}]

# Попытка отредактировать чужой пост → 403 Forbidden
$ curl -X PUT http://localhost:8080/posts/1 -H "Authorization: Bearer $WRONG_USER_TOKEN" -d '...'
HTTP/1.1 403 Forbidden
{"error":"forbidden: not the author"}
```

Это **полный** REST-сервис с аутентификацией, авторизацией, многотабличной схемой и тестами. Архитектурно похожий на то, что ты увидишь в реальных продакшен-проектах на Go.

## Что нового по сравнению с предыдущими проектами

- **Много пакетов** — `internal/api`, `internal/storage`, `internal/auth`, `cmd/server`. Это первый раз, когда твой код **не помещается** в `package main`. Реальная архитектура.
- **Аутентификация** — bcrypt для хеширования паролей, JWT для сессий, middleware для проверки токенов. Это базовый стек, который встретишь в 90% Go-бэкендов.
- **Authorization (kohirpicht)** — не путать с аутентификацией. Аутентификация = «кто ты», авторизация = «что тебе можно». Проверка `author_id == current_user_id` — простейшая форма авторизации.
- **Multi-table схема с FK** — `users`, `posts`, `comments`. Foreign keys, JOIN-ы, ON DELETE CASCADE. Это первая «настоящая» реляционная схема в roadmap-е.
- **Тесты** — впервые серьёзно. `testing` пакет, table-driven tests, `httptest.NewRecorder`, testcontainers для интеграционных тестов с реальной БД. Без тестов современный Go-проект не сдаёшь.
- **Middleware-паттерн** — оборачивание HTTP-обработчиков для общей логики (логирование, аутентификация, CORS). Это фундамент любого веб-фреймворка, и `net/http` его поддерживает «из коробки».
- **Request context для request-scoped данных** — в обработчик через `r.Context()` приходит `user_id` из middleware. Стандартный паттерн.

## Что нового **не** будет

- **Frontend** — мы пишем только API. Если хочется UI — можно подключить любой клиент (React, мобильное приложение, Postman).
- **OAuth** (Google, GitHub login) — это отдельная тема, в основной поток не входит. JWT с email/password достаточно для понимания принципов.
- **WebSockets** — для блога не нужны
- **Загрузка файлов** (картинки) — отдельная тема
- **Email-нотификации** — отдельная тема
- **Pagination** — упомянем, не реализуем
- **Search** — не нужен в учебном проекте
- **Rate limiting** — упомянем как production concern
- **Caching** (Redis) — упомянем как next step
- **gRPC** — это другой проект для другого курса
- **Kubernetes / Docker Compose** — упомянем, не делаем
- **CI/CD** — не входит в roadmap

Все эти темы — естественные **следующие шаги** после roadmap-а. Когда дойдёшь до них, у тебя уже будет фундамент.

## Главные мысли проекта

**Первое.** Архитектура «handler → service → repository» — это **не оверинжиниринг**, это **способ переживать рост**. В маленьком проекте кажется лишним. В проекте на 50 эндпоинтов — спасает жизнь. Ты увидишь это на собственном опыте, когда будешь добавлять `posts` после `users`: 80% кода уже структурно готово, нужно только повторить паттерн.

**Второе.** Аутентификация — это **простая** тема, если ты понимаешь два инструмента: **bcrypt** (хеширование паролей) и **JWT** (выдача токенов). Эти два инструмента — отраслевой стандарт. Раз освоив, ты сможешь добавить auth в любой Go-проект за несколько часов.

**Третье.** Foreign keys — это **гарантии целостности**, которые БД даёт тебе бесплатно. Без них ты должен **сам** проверять, что `author_id` в посте указывает на существующего пользователя. С ними — БД делает это за тебя, и любая попытка вставить «висячий» пост падает на уровне БД.

**Четвёртое.** Тесты — это не «бонус для отличников», это **часть профессионального кода**. После того как ты увидишь, как `httptest.NewRecorder` позволяет тестировать HTTP-обработчики **без** запуска сервера — обратной дороги нет. Это даёт уверенность при рефакторинге, который иначе страшен.

**Пятое.** **Пакеты — это границы.** В одном пакете все файлы видят друг друга, между пакетами — только публичный API (с большой буквы). Это **дисциплина дизайна**: если функция нужна снаружи — она публичная, иначе — приватная. Эта дисциплина переносится на проекты любого размера.
